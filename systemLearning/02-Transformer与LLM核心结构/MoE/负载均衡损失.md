# 负载均衡损失

> **所属章节**: [[MoE]]
> **所属模块**: [[02-Transformer与LLM核心结构]]
> **难度**: 中级（从零讲清"专家坍缩为什么发生、如何用损失和偏置治它"）

## 1. 一句话定义

**负载均衡损失（load balancing loss，辅助损失，auxiliary loss，均衡损失）** 是 [[MoE]] 训练时**额外加进总损失**的一个项，用来惩罚"token 分配到各专家不均匀"——防止路由器总选少数专家（**专家坍缩**）而让其余专家饿死；其经典形式 $\mathcal{L}_{\text{aux}} = \alpha E \sum_i f_i P_i$ 同时惩罚"被选多"和"给高概率"，是 MoE 能训起来的关键兜底机制。DeepSeek-V3 进一步提出**无辅助损失均衡**（用可学习偏置替代损失），避免干扰主目标。


> [!note] 解答：负载均衡是只针对训练吗？推理时还用顾虑吗？
> 
> **辅助损失本身是训练专属**（推理时不加 loss、不反传），但**"均衡"这件事推理时仍要关心**，只是手段变了：
> 
> | 阶段 | 均衡的手段 | 不均衡的后果 |
> |---|---|---|
> | **训练** | 辅助损失 / DeepSeek 偏置 / 容量因子 | 专家坍缩、参数浪费、训练发散 |
> | **推理** | 容量因子 / 路由调度 / 专家 placement / 专家并行负载均衡 | 热点专家过载、吞吐不均、延迟升高 |
> 
> **推理时为什么仍要顾虑均衡**：
> 1. **专家并行下热点专家过载**：[[专家并行]] 把不同专家放不同 GPU，若某专家被选过多，那块 GPU 过载、其余闲置，All-to-All 通信也因不均衡低效（见 [[#8.3 均衡与系统效率的双重意义]]）。
> 2. **batch 内 token 路由不均**：一个 batch 里若大部分 token 都路由到同一专家，该专家成为瓶颈，拖慢整个 batch 吞吐。
> 3. **延迟不稳**：热门专家排队，请求延迟波动大。
> 
> **推理时的均衡手段（无 loss）**：
> - **容量因子**：超量 token 走残差或丢弃（推理一般走残差保质量）；
> - **路由调度**：请求级路由，把会命中热门专家的请求分散到不同 GPU / 时刻；
> - **专家 placement**：把热门专家复制或多放几块 GPU（expert replication）；
> - **专家并行负载均衡**：All-to-All 前做 token 重排，让各 GPU 负载均匀。
> 
> **一句话**：训练时靠**损失**逼路由器学均衡；推理时路由器已固定，靠**系统调度**治不均。辅助损失是训练工具，但"均衡"是训练和推理都要关心的事，只是层次不同。
## 2. 为什么需要它（动机与背景）

### 2.1 专家坍缩：MoE 训练的头号杀手

不加任何干预时，[[MoE路由]] 的路由器会**专家坍缩**（expert collapse）：总选少数几个专家，其余专家几乎不被选。

恶性循环（正反馈）：

```
某专家偶然被多选
    → 该专家训练得更多 → 表达力更强
    → 路由器给它更高分数 → 更容易被选
    → 该专家被选更多 → ...
其他专家
    → 几乎不被选 → 得不到梯度(见 MoE路由 §4.2)
    → 永远不更新 → 表达力差 → 路由器更不选它
    → 饿死,参数浪费
```

后果：$E$ 个专家里只有 2-3 个在工作，MoE 退化成小模型，**总参数浪费、容量没兑现**。这是 MoE 训练的头号失败模式。

### 2.2 为什么路由器自己学不会均衡

直觉上"均衡用所有专家"应该更好，但路由器**只从被选中专家得梯度**（[[MoE路由]] §4.2）：未被选的专家分数梯度为 0，路由器感受不到"还有别的专家可用"。一旦初始偶然偏了，正反馈把它锁死，路由器自己逃不出来。必须**外部加干预**——这就是负载均衡损失的作用。

## 3. 核心概念详解

### 3.1 辅助损失的经典形式（Shazeer / GShard）

$$
\mathcal{L}_{\text{aux}} = \alpha \cdot E \cdot \sum_{i=1}^{E} f_i \cdot P_i
$$

各项含义：

| 符号 | 含义 | 范围 |
|---|---|---|
| $\alpha$ | 辅助损失权重（超参，常 0.01） | 标量 |
| $E$ | 专家数 | 常数 |
| $f_i$ | 分配给专家 $i$ 的 **token 比例**（被选次数 / 总 token 数） | $\sum_i f_i = k$（每 token 选 k 个） |
| $P_i$ | 路由器给专家 $i$ 的**平均概率**（softmax 前的 logits 平均，或 softmax 后的平均） | $\sum_i P_i = 1$ |

**直觉**：

- $f_i$ 大 = 专家 $i$ 被频繁选中；
- $P_i$ 大 = 路由器倾向给专家 $i$ 高分；
- 两者都大 = "既被多选、路由器又偏心" → 该被惩罚；
- 乘积 $f_i P_i$ 在两者同时大时最大，惩罚最重；
- 理想均衡：$f_i = k/E$，$P_i = 1/E$，此时 $\mathcal{L}_{\text{aux}} = \alpha E \cdot E \cdot \frac{k}{E} \cdot \frac{1}{E} = \alpha k$（最小）。

> [!note] 为什么乘积形式
> 用 $f_i \cdot P_i$ 而非 $f_i + P_i$，是因为乘积能捕捉"同时偏"：如果只看 $f_i$（被选次数），路由器可能把概率压低来骗过损失（实际还是选它）。乘积让"被选多"和"概率高"必须同时被压，才能让损失降。

### 3.2 为什么 $f_i$ 不可导但能工作

$f_i$（被选 token 比例）是 top-k 的结果，**对路由器权重不可导**。工程上的处理：

- $f_i$ 用"硬"统计（不可导，但作为常数参与损失）；
- $P_i$（平均路由概率）是 **softmax 前的 logits 平均**或 **softmax 后概率平均**，**对路由器权重可导**；
- 梯度只通过 $P_i$ 流回路由器，$f_i$ 当作 stop-gradient 的常数。

效果：损失惩罚高 $P_i$ 的专家（让路由器降低它的概率），间接推动均衡。这是"用可导部分近似不可导目标"的典型工程技巧。

### 3.3 容量因子与 token dropping

除了辅助损失，另一类均衡手段是**容量因子（capacity factor）**：

$$
\text{cap}_i = \frac{T \cdot k}{E} \cdot c \quad \text{（每专家最多处理 token 数）}
$$

- $T$：batch 内 token 总数；
- $c$：容量因子（常 1.0~1.5），给余量；
- 超过 $\text{cap}_i$ 的 token 被**丢弃**（dropped）或走**残差**（直接传到下一层，不经过专家）。

作用：硬性限制热门专家过载，逼路由器把多余 token 分给别的专家。代价：被丢的 token 没经过专家加工，质量受损。

### 3.4 噪声路由

给路由 logits 加高斯噪声再 top-k：

$$
\text{logits}' = \text{logits} + \epsilon, \quad \epsilon \sim \mathcal{N}(0, \sigma^2)
$$

- 噪声扰动让"边缘选中"的专家偶尔换别的，打破正反馈锁死；
- 训练初期噪声大（鼓励探索），后期衰减（精修）；
- Shazeer 2017 原版用，现代实现可选。

### 3.5 DeepSeek 无辅助损失均衡（auxiliary-loss-free）

辅助损失的问题：它**直接进总损失**，会**干扰主目标**（语言模型 loss），调 $\alpha$ 太大伤质量、太小不均衡。DeepSeek-V3 提出**不加辅助损失**的均衡方案：

核心思路：给路由 logits 加一个**可学习的偏置项** $b_i$：

$$
\text{logits}'_i = \text{logits}_i + b_i
$$

- $b_i$ **不参与路由器梯度**（stop-gradient），仅由一个简单的均衡控制器更新；
- 若专家 $i$ 被选过多 → 增大其他专家的 $b$、减小 $i$ 的 $b$ → 把路由引开；
- $b_i$ 不进损失，只是动态调整路由分数，**完全不干扰主目标**。

这是 MoE 均衡的工程新标杆，避免了辅助损失的质量损失。

## 4. 数学原理 / 公式

### 4.1 辅助损失的最小值推导

求 $\mathcal{L}_{\text{aux}} = \alpha E \sum_i f_i P_i$ 在约束 $\sum_i f_i = k$、$\sum_i P_i = 1$ 下的最小值。

用拉格朗日乘子法（$f_i, P_i$ 独立约束）：

$$
\nabla_{P_i} \left[\alpha E \sum_j f_j P_j + \lambda(\sum_j P_j - 1)\right] = \alpha E f_i + \lambda = 0 \Rightarrow f_i \text{ 对所有 } i \text{ 相同}
$$

$$
\nabla_{f_i} \left[\alpha E \sum_j f_j P_j + \mu(\sum_j f_j - k)\right] = \alpha E P_i + \mu = 0 \Rightarrow P_i \text{ 对所有 } i \text{ 相同}
$$

故最小值在 $f_i = k/E$、$P_i = 1/E$（完全均衡）取到：

$$
\mathcal{L}_{\text{aux}}^{\min} = \alpha E \cdot E \cdot \frac{k}{E} \cdot \frac{1}{E} = \alpha k
$$

任何不均衡都使 $\mathcal{L}_{\text{aux}} > \alpha k$，梯度推路由器走向均衡。

### 4.2 梯度流向（为什么只通过 $P_i$）

记 $s_i = x W_g^{(i)}$（专家 $i$ 的 logit），$P_i = \text{softmax}_E(s_i)$ 的平均（对所有 token 平均）。$f_i$ 是 top-k 统计（常数）。

$$
\frac{\partial \mathcal{L}_{\text{aux}}}{\partial s_i} = \alpha E \cdot f_i \cdot \frac{\partial P_i}{\partial s_i}
$$

- $\frac{\partial P_i}{\partial s_i}$ 是 softmax 雅可比，**可导**；
- $f_i$ 当作常数（stop-gradient）；
- 梯度让 $s_i$ 往"降低 $P_i$"方向走（若 $f_i$ 大），从而压低过热专家的路由概率。

### 4.3 容量因子的丢弃率

设实际分配给专家 $i$ 的 token 数 $n_i$，容量 $\text{cap} = Tk/E \cdot c$。丢弃率：

$$
\text{drop\_rate}_i = \max\left(0, 1 - \frac{\text{cap}}{n_i}\right) \quad (n_i > \text{cap 时为正})
$$

- 均衡时 $n_i \approx Tk/E$，$\text{drop\_rate} \approx \max(0, 1 - c) = 0$（$c \geq 1$）；
- 不均衡时热门专家 $n_i \gg \text{cap}$，大量 token 被丢。

## 5. 代码示例

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

def load_balancing_loss(router_logits, num_experts, top_k, alpha=0.01):
    """
    经典辅助损失 L_aux = alpha * E * sum_i f_i * P_i
    router_logits: (T, E) 路由器对 T 个 token 的 logits
    """
    E = num_experts
    T = router_logits.shape[0]
    # 1. f_i: 每个专家被选的 token 比例(不可导,用 top-k 统计)
    with torch.no_grad():
        _, topk_idx = router_logits.topk(top_k, dim=-1)     # (T, k)
        counts = torch.zeros(E, device=router_logits.device)
        for e in range(E):
            counts[e] = (topk_idx == e).sum()
        f = counts / T                                       # (E,) 每专家被选比例, sum(f)=k
    # 2. P_i: 路由器给每专家的平均概率(可导,softmax 后对 token 平均)
    probs = F.softmax(router_logits, dim=-1)                # (T, E)
    P = probs.mean(dim=0)                                    # (E,) 平均概率, sum(P)=1
    # 3. 辅助损失
    loss = alpha * E * (f * P).sum()
    return loss

# 验证
logits = torch.randn(100, 8)  # 100 token, 8 专家
loss = load_balancing_loss(logits, num_experts=8, top_k=2, alpha=0.01)
print("辅助损失:", loss.item())  # 理想均衡下 ≈ 0.01 * 2 = 0.02; 越偏越大

# DeepSeek 风格无辅助损失:用可学习偏置调整路由(不进损失)
class DeepSeekBiasBalancer:
    """每步根据负载更新偏置 b, 不进损失, 只改路由 logits。"""
    def __init__(self, num_experts, update_rate=0.1):
        self.b = torch.zeros(num_experts)
        self.ur = update_rate
    def update(self, topk_idx, num_experts):
        with torch.no_grad():
            counts = torch.zeros(num_experts)
            for e in range(num_experts):
                counts[e] = (topk_idx == e).sum()
            # 被选多的专家,b 减小; 被选少的,b 增大
            mean_count = counts.mean()
            self.b += self.ur * (mean_count - counts)  # 把流量从过热专家引开
    def apply(self, logits):
        return logits + self.b.to(logits.device)  # 加偏置后再 top-k
```

> [!tip] 工程要点
> 实际框架（如 Megatron-LM 的 MoE）会把辅助损失和主损失一起反传，$\alpha$ 是关键超参。DeepSeek 方案则把均衡逻辑从损失里剥离，更干净。两者可混用：偏置为主、小 $\alpha$ 兜底。

## 6. 与其他知识点的关系

- **上游（依赖）**: [[MoE路由]]（均衡损失治路由的坍缩）、[[MoE]]（是 MoE 训练的必要组件）、[[损失函数分类]]（属于辅助/正则损失）。
- **下游（应用）**: MoE 训练稳定性（均衡是稳定的前提）、[[专家并行]]（均衡也是系统效率问题，热点专家拖慢 All-to-All）。
- **对比 / 易混**:
  - **辅助损失 vs L2 正则**：L2 压参数范数防过拟合，辅助损失压专家不均防坍缩。都是"附加损失项"，但目标不同。
  - **辅助损失 vs [[entropy regularization]]**：entropy 正则是鼓励策略熵大（RL 场景），辅助损失是鼓励专家负载均（MoE 场景）。都是"防退化"，但对象不同。
  - **辅助损失 vs 容量因子**：前者软（损失梯度引导），后者硬（超量直接丢）。常配合用。

## 7. 常见误区与易错点

> [!warning] 误区清单
> 1. **以为 $\alpha$ 越大越好**：错。$\alpha$ 太大→辅助损失主导→主目标（语言建模）被干扰→质量掉。$\alpha$ 常很小（0.01 量级），DeepSeek 干脆不用损失改用偏置。
> 2. **以为 $f_i$ 可导**：错。$f_i$ 是 top-k 统计，对路由器权重不可导。梯度只通过 $P_i$ 流，$f_i$ 当常数。这是实现里最容易踩的坑（忘加 `no_grad` 会报错或算错梯度）。
> 3. **以为不加均衡也能训**：小规模/短训练可能侥幸，规模化必坍缩。MoE 训练均衡损失/偏置是标配，不是可选。
> 4. **混淆 $f_i$ 和 $P_i$**：$f_i$ 是"实际被选次数比例"（硬统计），$P_i$ 是"路由器给的概率"（软概率）。两者都可大可小，损失惩罚"同时大"。
> 5. **容量因子设太小**：$c < 1$ 会大量丢 token，质量崩。常 1.0~1.5。
> 6. **以为 DeepSeek 方案就一定更好**：无辅助损失避免干扰主目标，但偏置调参也有工程门槛，小团队未必调得动。经典辅助损失更成熟、好调。
> 7. **均衡 = 完全均匀**：理想是"按需求分配"，不是死板均匀。某些专家天然该多处理（如高频 token）。损失是"防极端不均"，不是"强制绝对均匀"。

## 8. 延伸细节

### 8.1 辅助损失的演化

- **Shazeer 2017**：原版噪声 top-k + 辅助损失；
- **GShard**：固定形式 $\alpha E \sum f_i P_i$，成为标准；
- **Switch Transformer**：简化为 top-1，辅助损失形式微调；
- **DeepSeek-V3**：无辅助损失 + 偏置均衡，2024 年新方案。

### 8.2 偏置均衡为什么有效

DeepSeek 偏置 $b_i$ 的更新规则（简化版）：被选多的专家 $b_i$ 减小、被选少的 $b_i$ 增大。因为 top-k 选的是 $\text{logits} + b$ 最大的，调 $b$ 等于"人为抬高低负载专家的分数"，把流量引过去。这相当于一个**简单的反馈控制器**，不进损失、不干扰梯度，只改路由分数。比辅助损失更"外科手术"。

### 8.3 均衡与系统效率的双重意义

负载均衡既是**质量问题**（防专家饿死）也是**系统问题**（防热点专家过载）。[[专家并行]] 下，不同专家在不同 GPU，若某专家被选过多，那块 GPU 过载、其余闲置，All-to-All 通信也因不均衡而低效。所以均衡损失的"系统价值"不亚于"质量价值"。

### 8.4 监控指标

训练 MoE 要监控：

- **每专家被选比例** $f_i$：应接近 $k/E$，方差小；
- **每专家平均概率** $P_i$：应接近 $1/E$；
- **token 丢弃率**：应接近 0；
- **辅助损失值**：应接近 $\alpha k$（最小值）。

任一指标偏离都说明均衡出问题，要调 $\alpha$ / 偏置 / 容量因子。
