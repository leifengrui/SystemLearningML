## 1. 一句话定义

**新模型接入** 是把一个 HuggingFace 仓库的新模型（如刚发布的 Llama-4 / DeepSeek-V3 / 某 VLM）接入 vLLM / SGLang 推理引擎、让它能被 serve 的**完整工程流程**——读 HF `config.json` 抽维度（`hidden_size`/`num_heads`/`num_kv_heads`/`layers`/`vocab`/`rope_scaling`/`sliding_window`）→ 写**权重名映射**（HF ckpt 命名 → 引擎内部模块名，由每个 `weight_loader` 函数处理 name 映射 + TP/EP 切分）→ 实现**模型类**（继承引擎的 model 模板，实现 `__init__` 建层 + `forward` 调层）→ 选**算子**（attention backend 选 flashinfer/pagedattention/flashattn，norm/rope 用 fused kernel）→ 写**多模态 processor**（图像/音频 token 化 + 时序对齐，纯文本模型跳过）→ **注册**到 registry（vLLM `ModelRegistry._REGISTRY` 按 `config.architectures` 匹配 / SGLang model loader）→ **测试**（权重加载 + forward 输出对齐 HF transformers）。难点不在"能跑"，而在**权重点缀与 TP/EP 切分逻辑**（合并 QKV、gate+up 融合、MoE expert 切分、MTP 多 token 预测头、rope_scaling 变体、sliding window）这些细节——错一点就数值不对或显存炸。

> [!note] 三句话定位
> - **是什么**：HF 新模型 → 引擎能 serve 的适配流程：config → 权重映射 → 模型类 → 算子 → 多模态 processor → 注册 → 测试。
> - **核心难点**：`weight_loader` 里的 name 映射 + TP/EP 切分（QKV 融合切片、gate+up 融合、MoE expert 分配、rope_scaling 变体、MTP 头、sliding window）。
> - **与 [[model runner]] 关系**：runner 调模型类的 `forward`；模型类用引擎的算子层（`ColumnParallelLinear`/`Attention`/`RMSNorm`），runner 只管建张量与采样，模型类只管建图与 forward。两层解耦。


## 2. 为什么需要它（动机与背景）

### 2.1 新模型层出不穷，引擎必须可扩展

LLM 迭代极快（Llama/Qwen/DeepSeek/Mistral 每季出新版，MoE/VLM/MTP 新结构不断）。引擎不可能每出新模型就重写，必须有一套**通用接入流程**：开发者照模板填几个文件，引擎就能 load 权重 + forward + serve。这是 vLLM/SGLang "model zoo" 达成百模型支持的根基。

### 2.2 HF 命名不统一，必须做映射

HuggingFace 各模型的 ckpt 权重命名**五花八门**（`q_proj`/`qkv_proj`/`query_key_value`、`gate_proj`/`gate_up_proj`/`mlp.gate`、`experts.0.w1`/`experts.gate_up`...）。引擎内部是统一的模块结构（`QKVParallelLinear`/`MergedColumnParallelLinear`/`MoE`），必须有一层**名字映射 + 张量重排**把 HF 的命名与布局翻译成引擎内部期望的。这层就是 `weight_loader`。

### 2.3 TP/EP 切分逻辑必须与权重加载耦合

引擎开 TP 时，每个权重要按 `tp_rank` 切片加载（列并行按 out 切、行并行按 in 切）。开 EP 时，MoE expert 要按 `ep_rank` 分组加载。这个切分**不是 forward 时做**（那时权重已在显存），而是**加载时做**——`weight_loader` 据 shard id 决定把哪片写入哪个 param 的哪个切片。所以权重映射与切分逻辑必须耦合在加载阶段。

### 2.4 算子选择影响性能与数值

不同 attention backend（flashattn / flashinfer / pagedattention(Triton)）在不同硬件/序列长/是否带 prefix cache 下性能与精度不同。norm（RMSNorm/LayerNorm）、rope（各变体 rope_scaling）也有 fused kernel。新模型要声明它用哪些算子（或用引擎默认），才能高效且数值正确。

### 2.5 多模态要对齐时序

VLM（视觉语言模型）要把图像变 token 并在文本序列里**对齐时序**（图像 token 插在 `<image>` 占位处，position 连续）。不同 VLM 的 processor（LLaVA/Qwen-VL/InternVL）对齐方式不同，引擎要适配各模型的 processor 把图像特征与文本 token 拼接对齐。纯文本模型跳过此步。


## 3. 核心概念详解

### 3.1 接入七步总览

```
① 读 HF config     -> 抽维度 (hidden/heads/kv_heads/layers/vocab/rope/sliding_window)
② 权重名映射       -> weight_loader: HF 名 -> 内部名 + TP/EP 切分
③ 实现模型类       -> __init__ 建层 (用引擎算子层), forward 调层
④ 算子选择         -> attention backend (flashinfer/paged/flashattn), norm/rope fused
⑤ 多模态 processor -> 图像/音频 token 化 + 时序对齐 (纯文本跳过)
⑥ 注册 registry   -> vLLM ModelRegistry._REGISTRY (按 architectures 匹配) / SGLang loader
⑦ 测试             -> 权重加载无误 + forward 输出对齐 HF transformers
```

| 步 | 产物 | 关键文件（vLLM） | 关键文件（SGLang） |
|---|---|---|---|
| ① config | 模型维度 | `vllm/transformers_utils/config.py` | `sglang/srt/configs/model_config.py` |
| ② 权重映射 | `weight_loader` 函数 | 各 model 文件 + `vllm/model_executor/model_loader/` | `sglang/srt/model_loader/` |
| ③ 模型类 | `*ForCausalLM` 类 | `vllm/model_executor/models/*.py` | `sglang/srt/models/*.py` |
| ④ 算子 | backend 选择 | `vllm/attention/backends/`、`vllm/model_executor/layers/` | `sglang/srt/layers/attention/` |
| ⑤ 多模态 | processor | `vllm/transformers_utils/processing.py`、模型 `data_parser` | `sglang/srt/managers/multimodal_*` |
| ⑥ 注册 | registry 条目 | `vllm/model_executor/models/registry.py` | `sglang/srt/model_loader/loader.py` |
| ⑦ 测试 | 对齐脚本 | `tests/models/` | `test/` |

### 3.2 ① 读 HF config

引擎从 HF 仓库的 `config.json` 读 `PretrainedConfig`，抽出建层所需的维度：

| 字段 | 含义 | 用途 |
|---|---|---|
| `hidden_size` `H` | 隐藏维 | 所有 linear 的 in/out |
| `num_attention_heads` `nh` | query head 数 | attention 切 head |
| `num_key_value_heads` `nkv` | KV head 数（GQA） | TP 上限受此约束 |
| `num_hidden_layers` `L` | 层数 | 建 L 个 transformer block |
| `vocab_size` `V` | 词表 | embedding + lm_head |
| `intermediate_size` | MLP 中间维 | gate/up/down |
| `rope_theta` / `rope_scaling` | RoPE 基频 / 变体 | 建 RoPE（dynamic/yarn/longrope 等） |
| `max_position_embeddings` | 最大上下文 | 决定 max_model_len 默认 |
| `sliding_window` | 滑窗 attn | 滑窗 attention 开关（见 3.9） |
| `num_experts` / `num_experts_per_tok` | MoE expert 数 / top-k | 建 MoE 层 + EP 切分 |
| `n_routed_experts` / `n_shared_experts` | DeepSeek 式路由/共享 expert | MoE 变体 |
| `mtp_layer_num` / `num_nextn_predict_layers` | MTP 头数 | 多 token 预测头（见 3.10） |

vLLM 的 `vllm/transformers_utils/config.py` 做解析与补全（缺省默认、`--hf-overrides` 覆盖、`--rope-scaling` CLI 覆盖 rope_scaling）。SGLang 的 `ModelConfig` 同理。

### 3.3 ② 权重名映射与 weight_loader（核心难点）

引擎不是"按名直接 load"，而是**每个 param 挂一个 `weight_loader` 函数**，它接收（`param, loaded_weight, loaded_shard_id, loaded_shard_id_packed`）并决定：
- **name 映射**：这个 HF 权重名该写到我的哪个 param？（如 HF `q_proj.weight` → 引擎 `self.qkv_proj` 的 q 段）
- **TP/EP 切分**：这个 loaded_weight 是完整还是已切？我该取它的哪个切片写入本 tp_rank 的 param？

`DefaultModelLoader` 遍历 HF state_dict，对每个 (name, tensor) 调对应 param 的 `weight_loader`。

**典型映射示例**：

```
HF ckpt (llama):                    引擎内部:
  model.layers.0.self_attn.q_proj  ->  layers.0.self_attn.qkv_proj (Q 段, 列并行切)
  model.layers.0.self_attn.k_proj  ->  layers.0.self_attn.qkv_proj (K 段)
  model.layers.0.self_attn.v_proj  ->  layers.0.self_attn.qkv_proj (V 段)
  model.layers.0.self_attn.o_proj  ->  layers.0.self_attn.o_proj  (行并行切)
  model.layers.0.mlp.gate_proj      ->  layers.0.mlp.gate_up_proj  (gate 段, 融合)
  model.layers.0.mlp.up_proj         ->  layers.0.mlp.gate_up_proj  (up 段, 融合)
  model.layers.0.mlp.down_proj       ->  layers.0.mlp.down_proj    (行并行切)
  model.embed_tokens                ->  model.embed_tokens        (vocab 列并行切)
  lm_head                           ->  lm_head                   (列并行切)
```

`QKVParallelLinear` 把 `q_proj`/`k_proj`/`v_proj` **融合成一个**权重（便于一次 kernel 算 QKV），`weight_loader` 据 `loaded_shard_id`（是 q/k/v）和 `tp_rank` 决定写入融合权重中 q/k/v 段的哪个 tp 切片。GQA 时 q 的 head 数与 kv 不同，切分逻辑要单独处理（q 按 `nh/tp` 切，kv 按 `max(1, nkv/tp)` 切，甚至复制）。

`MergedColumnParallelLinear` 同理把 `gate_proj`/`up_proj` 融合。

**MoE 的 weight_loader** 更复杂：expert 权重要按 `ep_rank`（或 tp_rank）分配到本卡持有的 expert 子集。如 DeepSeek 的 `experts.gate_up_proj`，weight_loader 据 expert id 决定该 expert 是否属本卡 + 在本卡的哪个槽。

> [!tip] 实践：weight_loader 是新模型接入最容易出 bug 的地方
> 数值对不齐、TP 下形状错、MoE expert 分错卡、GQA kv head 切错，十之八九出在 weight_loader 的 shard 逻辑。务必用 `tests/` 对齐脚本逐层核。

### 3.4 ③ 实现模型类

模型类（如 `LlamaForCausalLM`）继承引擎基类，实现：

```python
# vllm/model_executor/models/llama.py (示意)
@support_inner_tp            # 声明支持 inner TP (权重在层内切)
@support_moe                 # 若是 MoE (声明 MoE 支持)
class LlamaForCausalLM(nn.Module):
    def __init__(self, *, config, ...):
        # 用引擎算子层建图 (这些层自带 TP 切分)
        self.embed_tokens = VocabParallelEmbedding(V, H)
        self.layers = nn.ModuleList([
            LlamaDecoderLayer(config) for _ in range(L)
        ])
        self.norm = RMSNorm(H, eps)
        self.lm_head = ParallelLMHead(V, H)  # 列并行

    def forward(self, input_ids, positions, kv_caches, attn_metadata):
        h = self.embed_tokens(input_ids)
        for i, layer in enumerate(self.layers):
            h = layer(h, positions, kv_caches[i], attn_metadata)
        h = self.norm(h)
        return self.lm_head(h)   # logits
```

`LlamaDecoderLayer` 内部用 `QKVParallelLinear` + `Attention` + `RowParallelLinear`(o_proj)、`MergedColumnParallelLinear`(gate_up) + `RowParallelLinear`(down)。这些**算子层自带 TP 切分**，模型类只管组装与 forward 流水，不写切分逻辑（切分在算子层与 weight_loader）。`forward` 不写采样——采样归 [[model runner]] 的 `sample`，模型类只返 logits。

### 3.5 ④ 算子选择

| 算子类别 | 可选 backend | 选择方式 | 说明 |
|---|---|---|---|
| **Attention** | `FlashAttentionBackend`、`FlashInferBackend`、`PagedAttentionBackend`(Triton)、`XFormersBackend` | `VLLM_ATTENTION_BACKEND` 环境变量 / 自动按硬件与特性选 | flashattn 通用、flashinfer 对 prefix cache / ffn spec 优、paged(Triton) 无依赖。均接 [[PagedAttention]] 的 paged KV |
| **Norm** | `RMSNorm`(fused kernel) / `LayerNorm` | 内置 | fused kernel 融合 norm + scale |
| **RoPE** | `RotaryEmbedding`(各 rope_scaling 变体) | 据 `rope_scaling` 自动 | dynamic/yarn/longrope/su 等，fused |
| **量化** | awq/gptq/fp8/marlin 等 | `--quantization` | 见 [[AWQ]]/[[GPTQ]]/[[FP8量化方案]] |

模型类通常不显式选 backend——`Attention` 层会按 env / 配置自动选。新模型若有特殊 attention（如 Mamba 的 scan、sliding window、multimodal 的 cross-attn），需在模型类里用对应层或声明。

### 3.6 ⑤ 多模态 processor

VLM 接入要写 processor：把图像（/音频/视频）经 vision encoder 抽特征 → token 化（如 1 图 = 576 token）→ 与文本 token **按时序对齐**（图像 token 插在占位处，position 连续）。流程：

```
① AutoProcessor.from_pretrained(model) -> processor (HF 自带)
② processor(text+image) -> input_ids (含图像占位 token id) + pixel_values (图像张量)
③ 引擎: 文本走 embed_tokens, 图像走 vision_tower -> 得图像特征 token
④ 按 placeholder 位置把图像 token 嵌入文本序列 (时序对齐, position 连续)
⑤ forward 走 LLM backbone
```

vLLM 模型类实现 `_parse_and_validate_image_input`、`data_parser`（多模态数据解析）、`get_multimodal_embeddings`（抽多模态 token）。`--mm-processor-kwargs` 覆盖 processor 参数（如 `{"num_crops": 4}`）。SGLang 有 `multimodal_data` 字段经 `TokenizerManager` 解析。详见各 VLM 模型文件（`llava.py`/`qwen2_vl.py`/`internvl.py`）。

> [!note] 补充：纯文本模型跳过 ⑤
> 纯文本 LLM（Llama/Qwen 文本版/DeepSeek 文本）无多模态，第 ⑤ 步跳过，只做 ①②③④⑥⑦。

### 3.7 ⑥ 注册到 registry

vLLM 的 `vllm/model_executor/models/registry.py` 维护 `ModelRegistry._REGISTRY`（架构名 → 模型类的映射）。加载时引擎读 `config.architectures`（如 `["LlamaForCausalLM"]`），在 registry 查对应类，找不到则报"未支持"。新模型接入要在 registry 注册：

```python
# registry.py (示意)
_REGISTRY: Dict[str, Type[nn.Module]] = {
    "LlamaForCausalLM": LlamaForCausalLM,
    "LlavaForConditionalGeneration": LlavaForConditionalGeneration,
    "DeepseekV2ForCausalLM": DeepseekV2ForCausalLM,   # MoE
    "Qwen2VLForConditionalGeneration": Qwen2VLForConditionalGeneration,  # VLM
    # ... 新模型加这
}
```

`@support_inner_tp` / `@support_moe` 装饰器标记模型是否支持 inner TP / 是否 MoE（影响 EP 切分）。SGLang 的 `sglang/srt/model_loader/loader.py` 同样按 `architectures` 匹配 `sglang/srt/models/` 下的类。

`--served-model-name` 可给模型起别名（API 里的 `model` 字段用别名，不必用 ckpt 路径）。

### 3.8 ⑦ 测试

vLLM 的 `tests/models/` 有对齐脚本：用 HF transformers 跑同一 prompt 得输出，与 vLLM 输出比（logits/采样 token），容差内才算通过。覆盖：权重加载（各 TP size 都要 load 成功且 shape 对）、forward 数值对齐（各 dtype、各 attention backend）、多模态输入对齐。SGLang 的 `test/` 同理。新模型接入必须过这关，否则数值静默错（用户感知不到但结果漂）。

### 3.9 rope_scaling 变体

新模型常用各类 RoPE 变体扩长上下文，引擎要支持：
- **dynamic**（NTK-aware dynamic）：按当前 seq_len 动态调频率。
- **yarn** / **longyarn**：Yet another RoPE extensioN，外推+内插。
- **longrope** / **su**（Llama-3.1/3.3 类）：短段保留高频、长段插值。
- **llama3**：Llama-3 的 rope_scaling 变体。

引擎据 `config.rope_scaling.type` 建 `RotaryEmbedding` 对应实现。CLI `--rope-scaling '{"rope_type":"dynamic","factor":2.0}'` 可覆盖。接入新模型若用新 rope 变体，要在 `vllm/model_executor/layers/rotary_embedding.py` 加实现，否则数值错。

### 3.10 sliding window / MTP / MoE gate（特例要点）

- **sliding window**：模型有 `sliding_window`（如 Mistral 4096）时 attention 只看最近 W token。引擎据 config 自动开滑窗 attention（部分 backend 支持）。`--disable-sliding-window` 可关。接入要确认 attention backend 支持滑窗 + positions 正确。
- **MTP（multi-token prediction）**：DeepSeek-V3 的 MTP 头（额外预测未来 N 个 token，用于投机/训练）。接入要实现 MTP 头的 forward（与主 lm_head 共享或独立）+ 权重加载。MTP 头的 token 也要采样/对齐。这是新结构，引擎支持度待核实。
- **MoE gate**：MoE 的 router（gate）权重是**密集**的（不切，每卡全份或按 TP 切 hidden 维），与 expert 权重（按 EP/TP 切）不同。weight_loader 要区分 gate（dense 切）vs expert（EP 切）。见 [[MoE路由]]、[[专家并行]]。

> [!warning] 误区：所有权重都一样切
> gate（dense）按 TP 切 hidden 维；expert 按 EP 切 expert 维或 TP 切 expert 权重列/行。两者切法不同，weight_loader 必须区分，否则 expert 路由错乱。


## 4. 数学原理 / 公式

### 4.1 QKV 融合的切分推导

设 `H`=hidden、`nh`=query head 数、`nkv`=KV head 数、`d_h`=head dim（`H/nh`）、TP=N。融合 QKV 权重期望形状 `[3·H, H]`（q 段 `H`、k 段 `nkv·d_h`、v 段 `nkv·d_h`，GQA 下 k/v 段比 q 小）。

第 i 个 tp_rank 取：
- q 段：head `[i·nh/N, (i+1)·nh/N)` → 切片 `nh/N · d_h` 行
- k 段：head `[i·nkv/N', (i+1)·nkv/N')` → 切片 `nkv/N' · d_h` 行（`N' = min(N, nkv)`，若 `nkv<N` 则部分 rank 复制同 KV head）
- v 段：同 k

本 rank 持有的融合 QKV 切片形状 `[nh/N·d_h + 2·(nkv/N')·d_h, H]`。GQA 的 `nkv < N` 时 KV 被迫复制，是 TP 上限受 `nkv` 约束的来源。

### 4.2 gate+up 融合的切分

`intermediate` = `I`（MLP 中间维）。融合 gate_up 权重 `[2I, H]`（gate 段 `I`、up 段 `I`）。第 i rank 取 gate 段第 `[i·I/N, (i+1)·I/N)` 切 + up 段同位置切 → 本 rank 持 `[2·I/N, H]`。down_proj（行并行）权重 `[H, I]` 按 in 维切 → 第 i rank 取 `[H, i·I/N:(i+1)·I/N]` → 持 `[H, I/N]`，forward 后 all-reduce 聚合。

### 4.3 MoE expert 的 EP 切分

设 `E` expert、`I` 中间维、EP=N。第 i rank 持 expert 子集 `E_i = [i·E/N, (i+1)·E/N)`。expert 权重 `experts.e·I` 按 expert id 切到对应 rank。token 经 [[MoE路由]] 算 top-k expert id，若目标 expert 在本 rank 则本地算，否则 All-to-All 派发（见 [[TP PP DP EP推理]] §3.5）。

### 4.4 rope_scaling 的频率变换

RoPE 基础频率 $\theta_i = \theta_{base} \cdot 10000^{-2i/d}$。`rope_scaling` 对频率做变换：
- dynamic NTK：$\theta_i' = \theta_i \cdot s^{2i/d}$，$s$ 随 seq_len 动态。
- yarn：分段，高频保留、低频插值。
具体推导见 RoPE 专文。接入新变体要在 `rotary_embedding.py` 实现对应频率表。

### 5.1 vLLM 启动带 rope_scaling / sliding window / 量化

```bash
vllm serve meta-llama/Meta-Llama-3.1-8B-Instruct \
  --tensor-parallel-size 2 \
  --max-model-len 32768 \
  --rope-scaling '{"rope_type":"llama3","factor":8.0,"low_freq_factor":1.0,"high_freq_factor":4.0,"original_max_position_embeddings":8192}' \
```

### 5.2 weight_loader 映射片段（概念，仿 vLLM）

```python
def qkv_proj_weight_loader(param, loaded_weight, loaded_shard_id, tp_rank, tp_size):
    """
    把 HF 的 q_proj/k_proj/v_proj 权重写入融合的 qkv_proj param 的对应切片.
    loaded_shard_id: "q" / "k" / "v" (来自 HF 名)
    """
    nh, nkv, d_h = num_heads, num_kv_heads, head_dim
    # q: 按 query head 切
    if loaded_shard_id == "q":
        n = nh // tp_size
        offset = tp_rank * n * d_h
        # 写入 qkv 中 q 段 [offset : offset+n*d_h]
        param.data[offset:offset + n*d_h].copy_(loaded_weight[...])
    elif loaded_shard_id == "k":
        n = max(1, nkv // tp_size)   # GQA: nkv 可能 < tp, 取 max(1,..)
        offset = nh*d_h + tp_rank * n * d_h   # q 段后是 k 段
        param.data[offset:offset + n*d_h].copy_(loaded_weight[...])
    elif loaded_shard_id == "v":
        n = max(1, nkv // tp_size)
        offset = nh*d_h + nkv*d_h + tp_rank * n * d_h
        param.data[offset:offset + n*d_h].copy_(loaded_weight[...])
```

### 5.3 模型类 forward 骨架

```python
class MyModelForCausalLM(nn.Module):
    def __init__(self, config):
        self.embed_tokens = VocabParallelEmbedding(config.vocab_size, config.hidden_size)
        self.layers = nn.ModuleList([MyDecoderLayer(config) for _ in range(config.num_hidden_layers)])
        self.norm = RMSNorm(config.hidden_size, config.rms_norm_eps)
        self.lm_head = ParallelLMHead(config.vocab_size, config.hidden_size)

    def forward(self, input_ids, positions, kv_caches, attn_metadata):
        h = self.embed_tokens(input_ids)
        for i, layer in enumerate(self.layers):
            h = layer(h, positions, kv_caches[i], attn_metadata)
        return self.lm_head(self.norm(h))   # runner 拿 logits 去采样
```


## 6. 与其他知识点的关系

- **上游（依赖）**: [[tokenization]]（prompt → input_ids）、[[TP PP DP EP推理]]（权重切分维度）、[[Tensor Parallel]]（列/行并行）、[[专家并行]]（MoE expert 切）、[[MoE路由]]（gate）、[[PagedAttention]]（attention backend 的 paged KV）、[[worker与executor]]（load 阶段的 worker）。
- **下游（应用）**: [[model runner]]（runner 调模型类 forward + 采样）、[[OpenAI API server]]（serve 接入的模型）、新模型上线 serving、MoE/VLM 推理、MTP 投机。
- **对比 / 易混**:
  - **新模型接入 vs [[model runner]]**：前者管"建图 + 权重加载 + forward 逻辑"，后者管"建张量 + 调 forward + 采样 + CUDA Graph"。两层解耦。
  - **weight_loader vs forward 切分**：切分在 weight_loader（加载时），forward 时权重已切好各 rank 各算一片。不是 forward 时切。
  - **vLLM vs SGLang 接入**：vLLM 用 registry + `@support_inner_tp` 装饰；SGLang 用 loader 按 architectures 匹配。流程同（config→映射→类→算子→注册→测），文件组织略异。
  - **多模态 processor vs 文本 tokenizer**：前者额外处理图像/音频 token 化与时序对齐，纯文本跳过。


## 7. 常见误区与易错点

> [!warning] 误区 1：权重映射只是改名字
> 不止改名，还要做**张量重排 + TP/EP 切分**（QKV 融合、gate+up 融合、MoE expert 分配、GQA 的 kv head 复制）。weight_loader 是数值正确的关键，错一点就静默漂。

> [!warning] 误区 2：所有 attention backend 数值完全一致
> 多数一致，但 paged(Triton) 与 flashattn 在某些数值边界（如 fp8 KV、大滑窗）可能有 ulp 级差异。新模型对齐测试要固定 backend。

> [!warning] 误区 3：TP 上限只看 GPU 数
> 还受 **GQA 的 `num_kv_heads`** 约束。`num_kv_heads=8` 的模型 TP>8 会复制 KV head，浪费显存不增并行。

> [!warning] 误区 4：MoE gate 和 expert 一样切
> gate（router）是 dense 权重按 TP 切 hidden 维；expert 按 EP 切 expert 维。weight_loader 必须区分，否则路由错乱。

> [!warning] 误区 5：rope_scaling 不实现也能跑
> 不实现会用默认 RoPE，长上下文数值错（外推/内插不对）。新 rope 变体必须在 `rotary_embedding.py` 加实现并测试对齐。

> [!warning] 误区 6：新模型 forward 里自己写采样
> 采样归 [[model runner]] 的 `sample`。模型类只返 logits，不采样。在 forward 里写采样会破坏 runner 的 CUDA Graph / async 流水。

> [!warning] 误区 7：注册了就能 serve
> 注册只是让引擎认得类。还要过权重加载（各 TP size）+ forward 对齐测试。否则能 serve 但数值错。

> [!warning] 误区 8：MTP 头和主 lm_head 一样
> MTP 头是额外预测头（预测未来 token），权重与 forward 逻辑独立，需单独实现 + 权重加载 + 采样对齐。新结构支持度待核实。


## 8. 延伸细节

### 8.1 inner TP 与 model parallel 装饰器

`@support_inner_tp` 标记模型支持"层内 TP"（权重在每层内部切，即标准 Megatron TP）。部分模型（如纯 PP 切的）用不同装饰。这影响 `weight_loader` 的切分维度选择。

### 8.2 量化模型的权重加载

量化模型（AWQ/GPTQ/fp8）的权重有 `qweight`/`scales`/`g_idx` 等额外张量，weight_loader 要处理这些（按 TP 切 qweight 的 group、scales 跟随）。`--load-format sharded_state` 支持预切 ckpt 加速。详见 [[AWQ]]/[[GPTQ]]/[[FP8量化方案]]。

### 8.3 safetensors 与多文件 ckpt

大模型权重分多 safetensors 文件（`model-00001-of-00005.safetensors`）。`DefaultModelLoader` 用 `index.json` 定位每个权重在哪个文件，并行下载/加载（`--max-parallel-loading-workers`）。接入不涉及（引擎通用），但加载大模型时要配。

### 8.4 内容来源

接入流程整理自 vLLM 源码 `vllm/model_executor/models/registry.py`（`ModelRegistry._REGISTRY`）、`vllm/model_executor/model_loader/`（`DefaultModelLoader` + `weight_loader`）、`vllm/model_executor/layers/`（`ColumnParallelLinear`/`RowParallelLinear`/`QKVParallelLinear`/`VocabParallelEmbedding`/`RMSNorm`/`RotaryEmbedding`）、`vllm/attention/backends/`、`vllm/transformers_utils/`、vLLM 官方文档《Adding a New Model》（待核实命名）、SGLang 源码 `sglang/srt/models/` 与 `sglang/srt/model_loader/loader.py`。rope_scaling/sliding_window/MTP/MoE gate 细节见对应模型文件与 [[MoE路由]]/[[专家并行]]。类名/路径若与最新版本有出入以仓库 main 为准。
