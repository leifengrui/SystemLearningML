# 请求生命周期

> **所属章节**: [[请求生命周期与scheduler]]
> **所属模块**: [[16-推理引擎源码]]
> **别名**: 请求生命周期 / request lifecycle / 请求状态机 / preemption / swapping / recompute / 请求调度状态
> **难度**: 中（需懂 [[vLLM scheduler]] + [[KV block manager]] + [[cache eviction与swap offload]] + [[autoregressive decoding]]）


## 1. 一句话定义

**请求生命周期** 是 vLLM 中**一个推理请求从接收到完成的状态流转**——请求经历 **WAITING**（在 waiting 队列，未分配 KV block，等 admission）→ **RUNNING**（被调度器 admission，分配 block，进入 running 队列，正在 prefill/decode）→（可选）**PREEMPTED/RECOMPUTE**（KV block 被驱逐，回 waiting，状态保留已生成 token，下次重算 prefill）或 **SWAPPED**（KV 换出 CPU，回 waiting，下次 swap_in 恢复）→ **FINISHED**（生成完 `<eos>` 或达 max_len，释放 block，出队列）六个状态。核心是 **preemption（抢占）**：running 队列 KV 不够时，scheduler 选低优先级（如刚 prefill 完待 decode 的）请求 **preempt**——**recompute** 策略丢其 KV 回 waiting（下次重算 prefill），或 **swapping** 策略换其 KV 到 CPU 回 waiting（下次 swap_in 恢复），让位给高优先级（decode 中）请求。是 [[vLLM scheduler]] 的状态机——scheduler 据 running/waiting 队列 + KV 预算决定每 iteration 谁进 running、谁被 preempt、谁完成，block manager 执行 block 操作，请求在状态间流转。与 [[cache eviction与swap offload]] 配套（后者是 preemption 的两种策略细节）。

> [!note] 三句话定位
> - **是什么**：请求六状态流转（WAITING→RUNNING→（PREEMPTED/SWAPPED）→FINISHED），核心是 preemption（KV 不够时 recompute 丢 KV 或 swap 换 CPU）。
> - **为什么**：高并发下 KV 不够，需让位高优先级（decode 中），preemption 保服务连续（有序退让而非 OOM 停服）。
> - **与 [[vLLM scheduler]] 关系**：scheduler 是调度器（每 iteration 决定状态流转），请求生命周期是被调度的对象的状态机。scheduler 驱动状态流转。


## 2. 为什么需要它（动机与背景）

### 2.1 请求需管理状态

推理请求不是一步完成（prefill + decode 多步）。需管理其状态（在队列、KV 分配、生成进度）。状态机管理流转。

### 2.2 KV 不够需 preemption

高并发 KV 不够。需让位（preempt 低优先级）。preempt 后请求回 waiting（不是丢弃，下次继续）。状态机保留状态（已生成 token）。

### 2.3 保服务连续

preemption 是有序退让，不是 OOM 停服。请求回 waiting 下次继续（重算或换回）。保服务连续性。

### 2.4 recompute vs swapping 的状态

- recompute：丢 KV，回 waiting，状态标 recompute（下次重算 prefill）
- swapping：换 CPU，回 waiting，状态标 swapped（下次 swap_in）

状态机区分两种 preempt 后的状态，决定下次恢复方式。


## 3. 核心概念详解

### 3.1 六状态

```
WAITING     -> 等 admission (未分配 KV block)
RUNNING     -> 已 admission, 正 prefill/decode
PREEMPTED   -> 被 recompute 策略 preempt (丢 KV, 回 waiting, 标 recompute)
SWAPPED     -> 被 swap 策略 preempt (KV 换 CPU, 回 waiting, 标 swapped)
FINISHED    -> 生成完 (eos/max_len), 释放 block
(ABORTED)   -> 用户取消
```

六状态。核心流转：WAITING→RUNNING→FINISHED，中间可 PREEMPTED/SWAPPED→RUNNING（恢复）。

### 3.2 状态流转

```
WAITING --admission--> RUNNING --eos--> FINISHED
  ^                    |
  |---preempt(recompute)-- PREEMPTED (回 waiting, 标 recompute)
  |---preempt(swap)---- SWAPPED (回 waiting, 标 swapped)
PREEMPTED --重新调度(重算 prefill)--> RUNNING
SWAPPED --重新调度(swap_in 恢复)--> RUNNING
```

正常：WAITING→RUNNING→FINISHED。满载：RUNNING→PREEMPTED/SWAPPED→WAITING→RUNNING（恢复）→FINISHED。

### 3.3 waiting 队列

```
waiting 队列 (FIFO):
  [req_5 (新), req_3 (preempted, recompute), req_7 (preempted, swapped)]
admission 时: 按优先级/顺序取, 分 block
```

waiting 队列存未 admission（新）与 preempted（回 waiting）的请求。admission 时按顺序取。

### 3.4 running 队列

```
running 队列 (decode 中):
  [req_1 (decode step 50), req_2 (decode step 30), req_4 (decode step 80)]
每 iteration: 这些请求一起 decode (continuous batching)
KV block 增长 (每步可能需新 block)
```

running 队列存正在 prefill/decode 的请求。每 iteration 一起 batch forward。KV 增长。

### 3.5 preemption（recompute）

```
KV block 不够新 decode:
  选 victim (LRU, 如 running 中刚 prefill 完待 decode 的)
  free victim 的 block (丢 KV), block 回 free list
  victim 状态 = PREEMPTED (标 recompute)
  victim 回 waiting 队列 (保留已生成 token)
  高优先级拿 block
下次 victim 调度:
  重新 prefill (重算 prompt 的 KV)
  继续 decode (从上次断点, 已生成 token 保留)
```

recompute：丢 KV 回 waiting，下次重算 prefill。详见 [[cache eviction与swap offload]]。

### 3.6 preemption（swapping）

```
KV block 不够:
  选 victim
  swap_out: victim 的 KV 复制到 CPU
  GPU block 回 free list
  victim 状态 = SWAPPED
  victim 回 waiting (保留已生成 token + swapped KV 在 CPU)
下次 victim 调度:
  swap_in: CPU 的 KV 复制回 GPU (新 block)
  继续 decode (KV 已恢复, 不重算)
```

swapping：换 CPU 回 waiting，下次 swap_in 恢复。详见 [[cache eviction与swap offload]]。

### 3.7 FINISHED

```
请求生成 <eos> 或达 max_len:
  状态 = FINISHED
  释放 KV block (block manager free)
  出 running 队列
  返回结果给用户
```

完成释放 block。是 block 回 free list 的正常来源。

### 3.8 状态保留

preempt 时保留：
- 已生成的 token（decode 进度）
- 原始 prompt
- 采样参数（temperature 等）

不保留（recompute 时丢）：
- KV cache（重算恢复）

不保留（swap 时换 CPU）：
- KV 在 CPU（swap_in 恢复）

状态保留让 preempt 后能续生成（不丢进度）。

### 3.9 与 continuous batching 的关系

[[continuous batching]] 的动态进出就是请求生命周期的流转。请求完成（FINISHED）出 running，新请求 admission 进 running。preempt 时出 running 回 waiting。是 continuous batching 的状态基础。详见 [[continuous batching的调度实现]]。


## 4. 数学原理 / 公式

### 4.1 状态转移

状态转移是有限状态机（FSM）。转移条件：
- WAITING→RUNNING：`can_allocate(KV)` 且 admission
- RUNNING→FINISHED：`<eos>` 或 `len >= max_len`
- RUNNING→PREEMPTED：`not can_allocate` 且 strategy=recompute
- RUNNING→SWAPPED：`not can_allocate` 且 strategy=swap 且 `cpu_has_room`
- PREEMPTED→RUNNING：重新调度（重算 prefill）
- SWAPPED→RUNNING：重新调度（swap_in）

### 4.2 生命周期的耗时

$$T_{\text{life}} = T_{\text{wait}} + T_{\text{prefill}} + T_{\text{decode}} \cdot N_{\text{tokens}} + \sum (T_{\text{preempt}} + T_{\text{recover}})$$

$T_{\text{wait}}$=waiting 时间，$T_{\text{prefill}}$=prefill，$T_{\text{decode}}$=每 token，$N_{\text{tokens}}$=生成数。preempt 加 $T_{\text{preempt}}$（丢/换）+ $T_{\text{recover}}$（重算/换回）。preempt 多则延迟升。

### 4.3 recompute 的恢复开销

$$T_{\text{recover}}^{\text{recompute}} = T_{\text{prefill}}(L_{\text{prompt}}) \quad (\text{重算 prefill})$$

### 4.4 swap 的恢复开销

$$T_{\text{recover}}^{\text{swap}} = \frac{M_{\text{KV}}}{\beta_{\text{PCIe}}} \quad (\text{CPU→GPU 拷贝})$$

详见 [[cache eviction与swap offload]]。

## 6. 与其他知识点的关系

- **上游（依赖）**: [[vLLM scheduler]]（驱动状态流转）、[[KV block manager]]（block 分配/回收）、[[cache eviction与swap offload]]（preempt 策略）、[[autoregressive decoding]]（prefill/decode）、[[continuous batching]]（动态进出）、[[KV cache]]。
- **下游（应用）**: vLLM serving 的请求管理、高并发下的 preemption、SLO 保障（preempt 影响延迟）。
- **对比 / 易混**:
  - **请求生命周期 vs [[vLLM scheduler]]**：前者是请求的状态机（被调度的对象），后者是调度器（驱动流转）。前者被动，后者主动。
  - **preempt recompute vs [[gradient checkpointing]]**：前者是推理 KV 丢重算（preempt），后者是训练 activation 丢重算（省显存）。概念同（丢中间态重算），场景不同。
  - **PREEMPTED vs SWAPPED**：前者丢 KV 重算（省显存耗算力），后者换 CPU 保 KV（耗带宽省算力）。据 prefill 长短选。


## 7. 常见误区与易错点

> [!warning] 误区 1：preempt 丢请求
> preempt 不丢请求（回 waiting 保留状态）。丢的是 KV（recompute）或换 CPU（swap）。请求下次继续（从断点）。是有序退让，不是丢弃。

> [!warning] 误区 2：preempt 是错误
> preempt 是有序退让（让位高优先级），保服务连续。不是错误。高并发下 KV 不够是常态，preempt 是设计的容错。

> [!warning] 误区 3：recompute 重算全部
> recompute 只重算 prefill（prompt 的 KV），不重算 decode 已生成 token 的输出（它们是 input）。但 prompt 长，重算 prefill 仍贵。

> [!warning] 误区 4：SWAPPED 立即恢复
> SWAPPED 需重新调度（admission）才 swap_in 恢复。在 waiting 期间 KV 在 CPU（不占 GPU）。swap_in 在重新调度时。

> [!warning] 误区 5：状态机无限状态
> 六状态（WAITING/RUNNING/PREEMPTED/SWAPPED/FINISHED/ABORTED）。preempt 只 recompute 或 swap 两种。是有限状态机。

> [!warning] 误区 6：FINISHED 后 block 不释放
> FINISHED 释放 KV block（block manager free）。是 block 回 free list 的正常来源。不释放会泄漏。


## 8. 延伸细节

### 8.1 preempt 的优先级

preempt 选 victim 据 LRU（最近最少活跃）。优先 preempt waiting 中刚 prefill 完待 decode 的（未开始 decode，损失小）。避免 preempt decode 中的（用户等）。是用户体验保护。

### 8.2 与 SGLang 的对照

SGLang 也有类似请求状态管理。但 RadixAttention 的前缀复用可减少 recompute（preempt 后重算时前缀从树恢复，不从头算）。详见 [[Radix Tree prefix cache]]。
