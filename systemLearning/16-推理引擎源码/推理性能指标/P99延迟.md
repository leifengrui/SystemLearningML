# P99延迟

> **所属章节**: [[推理性能指标]]
> **所属模块**: [[16-推理引擎源码]]
> **别名**: P99 延迟 / 尾延迟 / tail latency / 99 分位数延迟 / P99 / P99 latency / 高分位延迟
> **难度**: 中（需懂 [[TTFT与TPOT]]、[[ITL]]、[[goodput]]、[[continuous batching的调度实现]]、[[KV block manager]]）


## 1. 一句话定义

**P99 延迟（尾延迟，99 分位数）** = 把所有请求（或所有 token 间隔）的延迟排序，**99% 的值低于此数**，剩下 1% 最慢的比这还高——它是刻画**长尾**的指标，因为 LLM 推理的"最慢那部分请求"主导用户体验（用户只记得卡的那次），所以 SLO 几乎都用 P99 而非均值。长尾来源主要是调度层与资源争用——[[continuous batching的调度实现]] 的 preempt 重算、batch 头部请求拖尾、[[KV block manager]] 的 KV swap、RL 的 weight sync、HBM/PCIe 争用——这些偶发事件把少数请求/iteration 拖到极慢，均值看不出，P99 才暴露。P99 可作用于 TTFT（P99 TTFT）、ITL（P99 ITL，见 [[ITL]]）等不同量，是 [[goodput]] 判 violation 的核心阈值（超 P99 SLO 的请求扣 goodput）。常用 **HDR histogram** 高效测量。

> [!note] 三句话定位
> - **是什么**：延迟分布的 99 分位数。99% 请求/间隔低于此值。刻画长尾。
> - **为什么**：长尾主导体验（用户只记得卡的那次），均值平均掉长尾看不出→SLO 用 P99。长尾来自 preempt/swap/batch 拖尾/weight sync/资源争用。
> - **与 [[goodput]] 关系**：超 P99 SLO 的请求算 violation，扣 goodput。P99 是 goodput 的判定阈值。


## 2. 为什么需要它（动机与背景）

### 2.1 长尾主导体验

用户对延迟的感知是"最慢那次"主导，不是平均。100 次请求 99 次快、1 次极慢，用户记住的是那 1 次。均值把这 1 次平均掉→看着达标→实际体验差。所以服务级 SLO 不用均值，用**分位数**（P99 等），专门约束长尾。

```
100 次请求延迟:
  99 次: 200ms
  1 次:  5000ms (preempt 重算)
  均值:  ~250ms  (看着达标?)
  P99:   ~5000ms (长尾暴露, 真实体感)
  -> SLO 用 P99, 不用均值
```

### 2.2 LLM 推理长尾来源多

LLM 推理有大量偶发慢源（见第 3 节），把少数请求/iteration 拖到极慢：preempt 重算（数秒）、KV swap（PCIe 慢）、batch 头部请求拖尾、weight sync（RL 场景）、HBM/PCIe 争用。这些是"99% 快、1% 慢"的典型长尾分布。均值无能为力，P99 专门约束。

### 2.3 SLO 用 P99 是行业标准

主流 LLM 服务 SLO 形如"P99 TTFT < 2s"、"P99 ITL < 200ms"（见 [[ITL]]）。云厂商（OpenAI/Anthropic/各大厂）服务协议都用 P99/P999 分位。是 [[goodput]] 判 violation 的阈值：超 P99 SLO 的请求不计 goodput。

### 2.4 资源规划看 P99 不看均值

容量规划要按 P99（最慢那 1%）留余量，不是均值。按均值留→长尾时违反 SLO。P99 驱动 KV 预留、batch 上限、GPU 数等容量决策。


## 3. 核心概念详解

### 3.1 分位数概念

把延迟样本排序，第 $p$ 分位数 = 使 $p\%$ 样本 $\le$ 此值的点：

```
延迟样本排序: [10, 20, 30, ..., 200, 5000]  (100 个)
  P50  (中位数):  中间值, ~200ms   (一半快一半慢)
  P90:            90% 低于此        ~较高
  P95:            95% 低于此        ~更高
  P99:            99% 低于此        ~长尾起点
  P999:           99.9% 低于此      ~更尾
  Pmax (最大):    100%             ~最慢那次
```

### 3.2 各分位数对比表

| 分位 | 含义 | 用途 | 长尾敏感度 |
|---|---|---|---|
| P50（中位） | 50% 低于此 | "典型"延迟 | 低（不看尾） |
| P90 | 90% 低于此 | 多数体验 | 中 |
| P95 | 95% 低于此 | 较严 SLO | 中高 |
| **P99** | **99% 低于此** | **主流 SLO** | **高** |
| P999 | 99.9% 低于此 | 严格实时 | 极高 |
| Pmax | 最大值 | 最坏情况 | 极高（但易受单次噪声） |

> [!note] 为什么是 P99 而非 Pmax
> Pmax（最大值）受单次噪声影响大（一次 GC/网络抖动就爆），不稳定。P99 忽略最极端 1%，既抓长尾又抗单次噪声，是工程平衡点。更严用 P999。

### 3.3 长尾来源（核心）

```
LLM 推理长尾 (最慢 1%) 来源:
┌──────────────────────────────────────────────────────────────┐
│ 1. preempt 重算    │ KV 不够驱逐重算 -> 数秒 (见 [[KV block manager]])│
│ 2. KV swap          │ swap offload 到 CPU, PCIe 慢 ([[cache eviction与swap offload]])│
│ 3. batch 头部拖尾  │ 长请求占 batch, 短请求等长请求的 iteration  │
│ 4. weight sync      │ RL policy 更新后同步, 同步期间停            │
│ 5. HBM/PCIe 争用    │ 其他 kernel/NCCL 抢带宽 -> 偶发慢           │
│ 6. GPU 抢占/调度抖动│ [[vLLM scheduler]] 决策偶发慢, batch 突变   │
│ 7. 长请求本身       │ 长 prompt/长输出, 该请求延迟天然高           │
│ 8. 网络抖动         │ 前端到推理的网络偶发慢                       │
└──────────────────────────────────────────────────────────────┘
```

### 3.4 P99 作用于不同量

P99 可作用于不同延迟量，最常见：

- **P99 TTFT**：首 token 延迟的 99 分位（见 [[TTFT与TPOT]]）。长尾来自 preempt/长 prompt/排队。
- **P99 ITL**：token 间隔的 99 分位（见 [[ITL]]）。长尾来自 preempt/chunked prefill 混入/batch 突变。
- **P99 E2E（端到端）**：整请求完成延迟的 99 分位。长尾来自长请求/preempt。

SLO 常同时定多个：P99 TTFT < 2s **且** P99 ITL < 200ms。

### 3.5 goodput 与 P99 的关系

[[goodput]] = throughput × (1 - violation_rate)。violation 由 P99 SLO 定义：

```
SLO: P99 TTFT < 2s, P99 ITL < 200ms
violation_rate = (违反 SLO 的请求数) / (总请求数)
  注: 严格说 P99 是分布统计量, 但工程上常把"单请求 TTFT > 2s 
       或单请求某 ITL 超阈值"算 violation
goodput = throughput × (1 - violation_rate)
  -> P99 SLO 越严, violation 越多, goodput 越低
  -> goodput 是"P99 SLO 约束下的有效吞吐"
```

P99 是 goodput 的判定阈值。详见 [[goodput]]。

### 3.6 测量：HDR histogram

P99 需对大量样本排序求分位。直接排序内存大、慢。工程用 **HDR histogram**（High Dynamic Range histogram）——一种对数分桶直方图，用固定桶数覆盖大动态范围（ns ~ min），插入 O(1)、查分位 O(1)，内存恒定。是 Prometheus、vLLM metrics、Java HdrHistogram 的标准实现。

```
普通排序求 P99:  存全部样本 -> 内存 O(N), 查分位 O(N log N)
HDR histogram:  对数分桶 (如 2^t 桶) -> 内存 O(log range), 插入/查分位 O(1)
  -> 适合流式实时监控 P99
```


## 4. 数学原理 / 公式

### 4.1 分位数定义

设样本 $X = \{x_1, x_2, \dots, x_N\}$（延迟样本），排序得 $x_{(1)} \le x_{(2)} \le \dots \le x_{(N)}$。第 $p$ 分位（$p \in [0,1]$）：

$$P_p = x_{(\lceil p \cdot N \rceil)}$$

（最近秩法；也有插值法，略。）P99 即 $p=0.99$：

$$\text{P99} = x_{(\lceil 0.99 N \rceil)}$$

即 99% 样本 $\le$ P99，1% 样本 $>$ P99。

### 4.2 均值 vs 分位数的差异

设 99 个样本 $= a$（快），1 个样本 $= b$（慢，$b \gg a$）：

$$\text{均值} = \frac{99a + b}{100} \approx a + \frac{b}{100}$$

$$\text{P99} = b \quad (\text{那 1 个慢样本就是 P99 位置})$$

> [!note] 推导
> 排序后第 99 百分位恰好落在那个慢样本 $b$ 上（99 个 $a$ 在前，$b$ 是第 100 个即 P99 位置附近）。均值把 $b$ 稀释到 $b/100$，P99 直接是 $b$。这就是"P99 抓长尾、均值漏长尾"的数学。$b$ 越大（preempt 数秒），均值与 P99 差距越大。

### 4.3 长尾分布的均值与 P99 失衡

重尾分布（如对数正态、帕累托）下，均值由尾部主导但仍被稀释，P99 直接取尾部值：

$$\frac{\text{P99}}{\text{均值}} \gg 1 \quad (\text{重尾分布})$$

如均值 250ms、P99 5000ms，比 20 倍。LLM 推理 preempt/swap 让分布重尾，均值与 P99 失衡严重。所以 SLO 必用 P99。

### 4.4 P99 与 violation_rate 的关系

若 SLO 阈值 $S$，设延迟分布 CDF 为 $F$：

$$\text{violation\_rate} = P(X > S) = 1 - F(S)$$

若定 $S = \text{P99}$，则 $F(\text{P99}) = 0.99$，violation_rate $= 0.01$。即按 P99 定 SLO→1% 违反。工程上常定 $S$ 为某固定值（如 2s），violation_rate = 实际超 $S$ 的比例（可 > 或 < 1%）。

[[goodput]]：

$$\text{goodput} = \text{throughput} \times (1 - \text{violation\_rate}) = \text{throughput} \times F(S)$$

SLO 阈值 $S$ 越严（低）→$F(S)$ 越小→violation 越多→goodput 越低。是 goodput 与 P99 的数学纽带。

### 4.5 HDR histogram 的分位查询

HDR histogram 把值域 $[1, 2^R)$ 按 $2^k$ 对数分桶（$k=0..R$），桶 $k$ 再线性细分。记录每桶计数 $c_k$。查 P99：

$$\text{找到最小 } m \text{ 使 } \sum_{k \le m} c_k \ge 0.99 N$$

桶 $m$ 的代表值即 P99 近似。插入 $O(1)$（算桶号++计数），查分位 $O(\log R)$（桶数对数级）。内存 $O(\log \text{range})$ 恒定，适合流式实时监控。


## 5. 代码示例（可选）

### 5.1 用 numpy 算 P99（直方图法）

```python
import numpy as np

def p99_via_sort(samples):
    """直接排序求 P99 (内存 O(N), 适合离线)"""
    return np.percentile(samples, 99)

def p99_via_histogram(samples, bins=1000):
    """直方图法求 P99 (内存 O(bins), 适合流式近似)"""
    hist, edges = np.histogram(samples, bins=bins)
    cum = np.cumsum(hist)
    total = cum[-1]
    idx = np.searchsorted(cum, 0.99 * total)
    return edges[idx]

# 模拟: 99% 200ms, 1% 5000ms (长尾)
rng = np.random.default_rng(0)
normal = rng.normal(0.200, 0.020, size=9900)
tail = rng.normal(5.0, 0.5, size=100)       # 1% 长尾
samples = np.concatenate([normal, tail])

print(f"均值            = {samples.mean()*1e3:.1f}ms")
print(f"P50             = {np.percentile(samples, 50)*1e3:.1f}ms")
print(f"P99 (排序)      = {p99_via_sort(samples)*1e3:.1f}ms")
print(f"P99 (直方图)    = {p99_via_histogram(samples)*1e3:.1f}ms")
print(f"P999            = {np.percentile(samples, 99.9)*1e3:.1f}ms")
print(f"Pmax            = {samples.max()*1e3:.1f}ms")
# 均值被长尾稀释, P99 暴露长尾
```

### 5.2 HDR-style 流式分位（固定桶）

```python
class HDRPercentile:
    """简化 HDR: 按值对数分桶, 流式插入, O(1) 查分位"""
    def __init__(self, max_val=100.0, n_buckets=10000):
        self.edges = np.linspace(0, max_val, n_buckets+1)
        self.counts = np.zeros(n_buckets, dtype=np.int64)
    def record(self, val):
        idx = np.searchsorted(self.edges, val) - 1
        if 0 <= idx < len(self.counts):
            self.counts[idx] += 1
    def percentile(self, p):
        cum = np.cumsum(self.counts)
        total = cum[-1]
        if total == 0:
            return 0.0
        idx = np.searchsorted(cum, p/100 * total)
        return self.edges[min(idx, len(self.edges)-1)]

hdr = HDRPercentile()
for s in samples:
    hdr.record(s)
print(f"流式 P99 (HDR-style) = {hdr.percentile(99)*1e3:.1f}ms")
```


## 6. 与其他知识点的关系

- **上游（依赖）**: [[TTFT与TPOT]]（P99 可作用于 TTFT/TPOT）、[[ITL]]（P99 ITL 是流式 SLO 核心）、[[continuous batching的调度实现]]（preempt/batch 突变是长尾源）、[[KV block manager]]（preempt/swap 是长尾源）。
- **下游（应用）**: [[goodput]]（P99 SLO 定 violation 阈值）、推理服务 SLO 协议、容量规划、调度调优（降 preempt 降 P99）。
- **对比 / 易混**:
  - **P99 vs 均值**：均值平均掉长尾，P99 抓长尾。SLO 用 P99。见 4.2/4.3。
  - **P99 vs Pmax**：Pmax 受单次噪声，P99 抗噪声仍抓尾。见 3.2 注。
  - **P99 ITL vs [[ITL]]**：ITL 是逐次序列，P99 ITL 是其 99 分位统计量。
  - **P99 vs [[goodput]]**：P99 是延迟分位（单请求统计），goodput 是有效吞吐（系统级，用 P99 定 violation）。前者定阈值，后者受其约束。


## 7. 常见误区与易错点

> [!warning] 误区 1：均值达标就够了
> 不够。长尾主导体验，均值平均掉尖峰。P99 才反映"最慢那 1%"的真实体感。SLO 用 P99 不用均值。

> [!warning] 误区 2：P99 越低越好，无限压
> 压 P99 要降 preempt/swap/调度抖动，成本高。极低 P99（如 P999）需大量冗余资源，goodput 大降。是 P99 vs goodput 的 tradeoff。

> [!warning] 误区 3：P99 = 最慢的请求
> 不是。P99 是 99 分位，**忽略最极端 1%**。最慢单次是 Pmax。P99 抗单次噪声，Pmax 不抗。

> [!warning] 误区 4：P99 高一定是算力不够
> 多是调度层问题（preempt/swap/batch 拖尾），不是算力。堆 GPU 不一定降 P99，要查 [[vLLM scheduler]] 配置。

> [!tip] 实践：监控 P99 直方图
> 生产用 HDR histogram 流式监控 P50/P90/P99/P999/Pmax 分布。P99 突涨→查 preempt/swap。P999 突涨→查极端噪声。

> [!note] 补充：P99 与 goodput 的张力
> 降 P99（严 SLO）→violation 涨→goodput 降。升 goodput（松 SLO）→P99 放宽→体验降。P99 与 goodput 是体验 vs 吞吐的张力点，需平衡。见 [[goodput]]。


## 8. 延伸细节

### 8.1 P999 与严格实时

实时场景（如语音对话）需 P999 甚 P9999（99.99%）。要求极高冗余。LLM 语音 agent 常定 P999 ITL < 100ms（待核实）。

### 8.2 多卡并行的 P99

TP/PP 并行引入 NCCL 通信，通信抖动是 P99 长尾源。[[PD分离]] 后 decode 池内并行通信抖动影响 P99 ITL。需 [[NCCL]] 优化。

### 8.3 数字待核实

文中 200ms/5000ms/2s 等为示意量级。具体 SLO 阈值随场景（聊天/语音/agent）变。云厂商实际 SLO 数字为公开协议待核实。
