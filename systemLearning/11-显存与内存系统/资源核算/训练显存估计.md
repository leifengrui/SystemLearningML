# 训练显存估计

> **所属章节**: [[资源核算]]
> **所属模块**: [[11-显存与内存系统]]
> **别名**: training memory estimation / 训练显存预算 / 显存估算 / memory footprint
> **难度**: 中（需懂 [[参数量计算]]、混合精度、Adam optimizer state、激活显存四大件）

## 1. 一句话定义

**训练显存估计（training memory estimation）** 是在写任何代码、花任何钱之前，用纸笔算清"训练一个 $N$ 参数模型到底要多少 GPU 显存"的方法。核心公式只有一行——**Adam 混合精度训练态显存 $\approx 16N$ bytes**（权重 $2N$ + 梯度 $2N$ + optimizer state $12N$），其中 $N$ 是参数量。再加激活 $\propto \text{batch}\times\text{seq}$，就得到单卡峰值显存。以 **92B 模型**为例：纯训练态 $\approx 16\times92 = 1472$ GB，**单卡 80GB 根本装不下**，必须上 [[Fully Sharded Data Parallel|FSDP]] / [[ZeRO (DeepSpeed)]] / [[Megatron-LM]] 分片——这就是为什么大模型训练离不开分布式并行。

> [!note] 三句话定位
> - **是什么**：纸笔估出训练 $N$ 参数模型所需 GPU 显存，公式 $M_{\text{train}} \approx 16N + M_{\text{act}}$。
> - **核心数**：Adam+bf16 训练态 = 权重 $2N$ + 梯度 $2N$ + optimizer $12N$ = $16N$ bytes，**92B → 1472 GB**。
> - **有什么用**：决定单卡能不能装、要不要并行、要多少卡多少节点——算不清显存就开训 = 直接 OOM 烧钱。

## 2. 为什么需要它（动机与背景）

### 2.1 先算账再写代码

LLM 训练动辄几百张 H100、几十万美元电费。在敲下第一行 `model = Llama(...)` 之前，必须能回答：

- 这个模型**单卡**装得下吗？（权重 + optimizer + 激活 vs 单卡 80GB）
- 装不下要**分片**到几张卡？（[[FSDP]] / [[ZeRO (DeepSpeed)]] / [[Megatron-LM]]）
- 激活占多少？要不要 [[gradient checkpointing]]？
- 最终要**多少节点**？成本多少？

**显存估计是训练工程的第一性原理**——不算清这个，后面所有工程决策（并行度、batch、checkpointing、offloading）都是盲拍。这正是 [[参数量计算]] 的下游：$N$ 算出来后，第一件事就是喂进显存公式。

### 2.2 训练显存 ≠ 推理显存

一个常见混淆：以为"92B 模型 184GB"（bf16 权重）就是训练要的显存。**错**。推理显存 $\approx$ 权重 + KV cache（无 optimizer、无梯度、无反向激活），训练显存多出三大块：

| 组成 | 推理 | 训练 |
|---|---|---|
| 权重 | $2N$（bf16） | $2N$（bf16） |
| 梯度 | 0 | $2N$（bf16） |
| optimizer state | 0 | $12N$（Adam fp32 $m,v$ + master） |
| 激活 | 0（前向算完即弃） | $M_{\text{act}}$（供反向） |
| KV cache | 有 | 0 |

训练显存 $\gg$ 推理显存，主因是 **optimizer state（$12N$，是权重的 6 倍）+ 激活**。详见 [[activation memory]] §2.2 的三大组成对比。

### 2.3 92B 是什么概念

92B 不是某个具体模型，而是用户做预算推演的假设量级（介于 Llama-2 70B 与 Qwen-72B、Llama-3.1 70B 等之间，接近 Qwen-72B / Llama-3 70B 量级，略大）。用 92B 是为了让数字"难看一点"——逼出"单卡装不下、必须并行"的结论。本文全程用 $N = 92 \times 10^9$ 做算例。

> [!tip] 量级感
> - **7B**：bf16 权重 14GB，训练态 $16N=112$GB，单卡 80GB **装不下训练态**但权重能装（可单卡推理），训练需 FSDP/ZeRO-2。
> - **70B~92B**：bf16 权重 140~184GB，**单卡连权重都装不下**，训练态 1120~1472GB，必须 FSDP/ZeRO-3 + TP 多维并行。
> - **405B+**（Llama-3 405B）：训练态 6.5TB，需数千卡。

## 3. 核心概念详解

### 3.1 训练显存的四大组成

完整训练显存公式：

$$
M_{\text{train}} = \underbrace{M_{\text{weight}}}_{\text{权重}} + \underbrace{M_{\text{grad}}}_{\text{梯度}} + \underbrace{M_{\text{opt}}}_{\text{optimizer state}} + \underbrace{M_{\text{act}}}_{\text{激活}} + \underbrace{M_{\text{tmp}}}_{\text{临时}}
$$

前三个正比于参数量 $N$（静态，与 batch 无关），第四个正比于 batch×seq（动态），第五个是通信缓冲/通信中间等杂项（小，常忽略或预留 10%）。

### 3.2 权重显存 $M_{\text{weight}}$

模型参数本身占的显存。每个参数占多少 byte 取决于精度：

| 精度 | bytes/param | 92B 权重 |
|---|---|---|
| fp32 | 4 | 368 GB |
| fp16 / bf16 | 2 | 184 GB |
| fp8 | 1 | 92 GB |
| int4 | 0.5 | 46 GB |

训练几乎都用 **bf16 混合精度**（权重 bf16 存 $2N$，但配 fp32 master 权重算更新，见 3.4）。纯 fp32 训练已基本淘汰（显存翻倍且无必要）。

### 3.3 梯度显存 $M_{\text{grad}}$

反向传播算出的每个参数的梯度，与权重同精度（bf16 训练 → 梯度 bf16）：

$$
M_{\text{grad}} = 2N \text{ bytes (bf16)}
$$

92B → 184 GB。注意梯度在反向时**逐层产生**（不是一次性全占），但峰值仍按全量 $2N$ 计。详见 [[gradient memory]]。

### 3.4 Optimizer state 显存 $M_{\text{opt}}$（大头）

这是训练显存的**最大单项**，也是"训练态 $\gg$ 推理态"的根因。以最常用的 **Adam（混合精度）** 为例：

| 项 | 精度 | bytes/param | 92B |
|---|---|---|---|
| 权重（bf16） | bf16 | 2 | 184 GB |
| 梯度（bf16） | bf16 | 2 | 184 GB |
| **fp32 master 权重** | fp32 | 4 | 368 GB |
| **Adam 一阶矩 $m$**（fp32） | fp32 | 4 | 368 GB |
| **Adam 二阶矩 $v$**（fp32） | fp32 | 4 | 368 GB |
| **训练态合计** | — | **16** | **1472 GB** |

> [!note] 为什么有 fp32 master 权重？
> bf16 精度只有 7 位尾数，权重更新量 $\eta \cdot g$ 可能小到 bf16 表示不了（被舍入为 0，"stuck update"）。故混合精度训练保留一份 **fp32 master 权重**做更新累加，每步把 master 转 bf16 喂前向。这就是 [[mixed precision training]] 的核心机制。代价是显存多 $4N$（master）。

不同 optimizer 的 state 大小：

| Optimizer | state bytes/param | 92B optimizer state | 训练态合计 |
|---|---|---|---|
| **SGD**（无动量） | 0 | 0 | $4N = 368$ GB |
| **SGD + momentum** | 4（fp32 $v$） | 368 GB | $8N = 736$ GB |
| **Adam**（bf16 训练，fp32 $m,v$ + master） | 12 | 1104 GB | $16N = 1472$ GB |
| **Adam**（纯 fp32 训练） | 8（$m,v$，无 master） | 736 GB | $12N = 1104$ GB |
| **Adafactor** | $\approx 2$（低秩 $m,v$） | 184 GB | $\approx 6N = 552$ GB |

> [!warning] 误区：以为"Adam state = $8N$"
> $8N$ 只算了 $m + v$（各 fp32 $4N$）。**混合精度训练还要加 fp32 master 权重 $4N$**，共 $12N$。这是 PyTorch `AMP` + `Adam` 的真实开销。若纯 fp32 训练则无 master（权重本身就是 fp32），Adam state = $8N$，权重 $4N$，合计 $12N$。两种路线训练态都是 $16N$ 或 $12N$，**别少算 master**。

### 3.5 激活显存 $M_{\text{act}}$

激活是唯一**与 batch/seq 相关**的大项，详见 [[activation memory]]。简表：

| 配置 | 每层激活 | 92B（$L\approx 80, h\approx 8192$） |
|---|---|---|
| 标准 attention，b=8 s=2048 | $11bsh + 3bn_hs^2$ | ~数百 GB（attention 矩阵主导） |
| + FlashAttention | $11bsh$ | ~80-120 GB |
| + gradient checkpointing | $bsh$（只存层边界） | ~10-15 GB |

激活是**长序列训练显存爆炸**的根因（$s^2$ 项），但通过 FA + checkpointing 可压到很小。训练显存预算时，若用了 FA+ckpt，激活常不是瓶颈（optimizer state 才是）；若没用，激活可能反超。

### 3.6 临时缓冲 $M_{\text{tmp}}$

- 梯度分桶（bucketing）通信缓冲：~$N \times \text{bucket\_cap}$
- all-reduce / all-gather 通信中间张量
- 算子内部 workspace（cuBLAS、FlashAttention 的 smem）
- loss 计算临时（softmax over 大词表）

量级不大，常预留 5~10% 余量。精算时用 [[GPU memory snapshot]] 抓实际峰值。

## 4. 数学原理 / 公式

### 4.1 训练态显存主公式（Adam + bf16 混合精度）

$$
\boxed{M_{\text{train\_state}} = \underbrace{2N}_{\text{权重 bf16}} + \underbrace{2N}_{\text{梯度 bf16}} + \underbrace{12N}_{\text{opt state (master+m+v)}} = 16N \text{ bytes}}
$$

- $N$ = 参数量（见 [[参数量计算]]，Transformer $\approx 12Ld^2$）
- 这是**静态部分**（与 batch 无关），optimizer state 占 $12/16 = 75\%$ 是绝对大头。

### 4.2 加激活后的峰值显存

$$
M_{\text{peak}} = 16N + M_{\text{act}}(b, s, L, h) + M_{\text{tmp}}
$$

### 4.3 92B 完整算例

设 92B 模型，$N = 9.2 \times 10^{10}$，bf16 训练，Adam，batch=8, seq=2048，用 FlashAttention + gradient checkpointing。

**第一步：静态训练态**

$$
M_{\text{state}} = 16N = 16 \times 9.2 \times 10^{10} = 1.472 \times 10^{12} \text{ bytes} = 1472 \text{ GB}
$$

分解：

| 项 | 公式 | 92B |
|---|---|---|
| 权重 bf16 | $2N$ | 184 GB |
| 梯度 bf16 | $2N$ | 184 GB |
| fp32 master 权重 | $4N$ | 368 GB |
| Adam $m$（fp32） | $4N$ | 368 GB |
| Adam $v$（fp32） | $4N$ | 368 GB |
| **训练态合计** | $16N$ | **1472 GB** |

**第二步：激活**

92B 量级取 $L=80, h=8192, n_h=64, d_k=128$。用 FA + ckpt 后每层只存 $bsh$：

$$
M_{\text{act/ckpt}} = L \cdot b \cdot s \cdot h \cdot 2 = 80 \cdot 8 \cdot 2048 \cdot 8192 \cdot 2 \approx 21.5 \text{ GB}
$$

（若不开 ckpt 且用标准 attention，激活可达数百 GB，attention 矩阵 $3bn_hs^2$ 在 $s=2048$ 时约 $80 \cdot 3 \cdot 8 \cdot 64 \cdot 2048^2 \cdot 2 \approx 322$ GB，是瓶颈——所以大模型训练 **FA + ckpt 是标配**。）

**第三步：单卡峰值显存（无分片，纯数据并行 DP/DDP）**

$$
M_{\text{peak}} = 1472 + 21.5 + \text{tmp} \approx 1500 \text{ GB}
$$

单卡 H100 80GB → **OOM 19 倍**。即 DDP（每卡一份完整训练态）**根本不可行**，92B 训练必须把训练态分片到多卡。

### 4.4 分片后单卡显存：要多少卡？

把 $16N$ 训练态用 [[Fully Sharded Data Parallel|FSDP]] / [[ZeRO (DeepSpeed)]] ZeRO-3 分片到 $P$ 张卡，每卡只存 $1/P$：

$$
M_{\text{per\_gpu}} = \frac{16N}{P} + M_{\text{act}} + M_{\text{comm}} + M_{\text{tmp}}
$$

注意**激活不分片**（每卡都要存自己 batch 的激活），通信缓冲 $M_{\text{comm}}$ 在 all-gather 时会瞬间拉一份全量大张量（峰值）。

**求卡数 $P$（H100 80GB，留 20% 余量给激活/通信/临时，可用 $\approx 64$GB）**：

$$
\frac{16N}{P} \le 64 \text{ GB} \implies P \ge \frac{16N}{64} = \frac{1472}{64} \approx 23
$$

即**最少 24 张 H100** 才能把 92B 训练态装下（纯 ZeRO-3/FSDP）。考虑激活峰值/通信临时，实际要 **32~64 张**起步。这是 92B 训练的**下限**。

### 4.5 不同并行策略下的显存账

| 策略 | 单卡训练态 | 单卡激活 | 92B 需卡数（粗估） |
|---|---|---|---|
| DDP（每卡全量） | $16N = 1472$ GB | $M_{\text{act}}$ | **不可行**（单卡 OOM） |
| ZeRO-1（分 opt state） | $2N+2N+12N/P$ | $M_{\text{act}}$ | $\frac{12N}{64} \approx 18$，实测 ~32 |
| ZeRO-2（分 opt+grad） | $2N+(2N+12N)/P$ | $M_{\text{act}}$ | $\frac{14N}{64} \approx 21$，实测 ~32 |
| **ZeRO-3 / FSDP**（全分片） | $16N/P$ | $M_{\text{act}}$ | $\frac{16N}{64} \approx 23$，实测 ~32-64 |
| FSDP + gradient ckpt | $16N/P$ | $Lbsh$（小） | ~24-32 |
| **+ TP**（权重也切） | $16N/(P_{\text{dp}} \cdot P_{\text{tp}})$ | 部分切 | 8卡节点内 TP8 + 跨节点 DP |
| + offload（CPU/NVMe） | $(16N)_{\text{部分卸载}}$ | $M_{\text{act}}$ | 卡数↓但带宽↓ |

详见 [[ZeRO (DeepSpeed)]] 的 ZeRO-1/2/3 分级、[[Megatron-LM]] 的 TP+PP+DP 三维。

## 5. 代码示例（可选）

### 5.1 训练显存估算函数

```python
def training_memory(N_bytes_params, optimizer='adam_amp',
                    use_grad_ckpt=True, use_flash_attn=True,
                    batch=8, seq=2048, n_layers=80, hidden=8192,
                    n_heads=64, bytes_per_elem=2):
    """估算训练峰值显存 (bytes).
    N_bytes_params: 参数量 (个数), optimizer: adam_amp/sgd/adafactor."""
    N = N_bytes_params
    # 1) 权重 + 梯度 (bf16)
    M_weight = 2 * N
    M_grad   = 2 * N
    # 2) optimizer state
    if optimizer == 'adam_amp':          # fp32 master + m + v
        M_opt = 12 * N
    elif optimizer == 'adam_fp32':       # 纯 fp32, 权重已 fp32, state=m+v
        M_opt = 8 * N; M_weight = 4*N; M_grad = 4*N
    elif optimizer == 'sgd_momentum':   # fp32 v
        M_opt = 4 * N
    elif optimizer == 'adafactor':      # 低秩
        M_opt = 2 * N
    else:                                 # sgd
        M_opt = 0
    M_state = M_weight + M_grad + M_opt
    # 3) 激活
    bsh = batch * seq * hidden
    if use_flash_attn:
        per_layer = 11 * bsh * bytes_per_elem
    else:
        per_layer = 11 * bsh * bytes_per_elem + \
                    3 * batch * n_heads * seq * seq * bytes_per_elem
    M_act = per_layer * n_layers
    if use_grad_ckpt:
        M_act = n_layers * bsh * bytes_per_elem   # 只存层边界
    # 4) 临时/通信 ~5%
    M_tmp = 0.05 * (M_state + M_act)
    return {
        'weight_GB': M_weight/1e9, 'grad_GB': M_grad/1e9,
        'opt_GB': M_opt/1e9, 'state_total_GB': M_state/1e9,
        'act_GB': M_act/1e9, 'tmp_GB': M_tmp/1e9,
        'peak_per_gpu_GB': (M_state + M_act + M_tmp)/1e9,
    }

# 92B, Adam AMP, FA + ckpt
r = training_memory(92e9)
for k, v in r.items():
    print(f'{k:20s}: {v:.1f} GB')
# state_total_GB:    1472.0 GB   ← 16N, 静态大头
# act_GB:             21.5 GB    ← FA+ckpt 压得很小
# peak_per_gpu_GB:  1572.2 GB   ← 单卡 OOM, 必须分片

# 分片到 P 卡后单卡
for P in [8, 16, 32, 64]:
    per_gpu = r['state_total_GB']/P + r['act_GB'] + r['tmp_GB']
    print(f'FSDP P={P:3d}: {per_gpu:.1f} GB/card '
          f'{"OK" if per_gpu < 80 else "OOM"}')
# P=  8: 207.5 OOM
# P= 16: 113.5 OOM
# P= 32:  67.5 OK   ← 32 卡起步
# P= 64:  44.5 OK
```

## 6. 与其他知识点的关系

- **上游（依赖）**: [[参数量计算]]（$N$ 是显存公式 $16N$ 的根输入）、[[Tensor]]/[[数值类型与精度]]（bytes/param 由精度决定）、[[mixed precision training]]（master 权重 + $16N$ 的来源）、[[optimizer state memory]]（$12N$ 的细拆）、[[gradient memory]]（$2N$）、[[activation memory]]（$M_{\text{act}}$）。
- **下游（应用）**: [[Fully Sharded Data Parallel|FSDP]] / [[ZeRO (DeepSpeed)]]（训练态分片，把 $16N$ 切到多卡）、[[Megatron-LM]]（TP+PP 切权重，省单卡训练态）、[[gradient checkpointing]]（减激活峰值）、[[offloading (CPU-NVMe)]]（把 optimizer state/激活卸到 CPU/NVMe）、[[GPU memory snapshot]]（实测峰值 vs 估算对照）、[[MFU与算术强度]]（显存装得下后算算力/时间）、[[FLOPs计算]]（$6ND$ 训练算力）。
- **对比 / 易混**:
  - **训练显存 vs 推理显存**：训练 $\approx 16N + M_{\text{act}}$，推理 $\approx 2N + M_{\text{KV}}$。训练多出 optimizer(12N)+梯度(2N)+激活，差 6-8 倍。详见 §2.2。
  - **$16N$ vs $12N$ vs $8N$**：取决于 optimizer 和精度路线（AMP master + Adam = 16N；纯 fp32 Adam = 12N；SGD = 4-8N）。报训练显存必须说清 optimizer + 精度。
  - **静态 vs 峰值**：$16N$ 是静态（不随 batch 变），峰值还要加激活+通信临时。大 batch/长 seq 时激活可能反超静态。

## 7. 常见误区与易错点

> [!warning] 误区 1：用权重显存当训练显存
> "92B bf16 = 184GB" 只是**权重**。训练态 $16N = 1472$GB，差 8 倍。报训练成本/卡数必须用 $16N$，不是 $2N$。

> [!warning] 误区 2：漏算 fp32 master 权重
> Adam + bf16 AMP 训练的 optimizer state 不只是 $m+v = 8N$，还有 **fp32 master 权重 $4N$**，共 $12N$。漏算会把训练态估成 $12N$（实际 $16N$），低估 33%。详见 [[mixed precision training]]。

> [!warning] 误区 3：以为单卡能靠"省 batch"装下 92B
> 即使 batch=1、seq=1（激活趋零），训练态 $16N = 1472$GB 仍远超单卡 80GB。**静态部分与 batch 无关**，省 batch 只省激活，省不了权重+梯度+optimizer。92B 训练**必须分片**，没有单卡方案。

> [!warning] 误区 4：DDP 和 FSDP 分不清
> [[Distributed Data Parallel|DDP]] = 每卡一份完整训练态（$16N$/卡），92B 单卡 OOM 不可行。[[Fully Sharded Data Parallel|FSDP]] = 训练态分片（$16N/P$/卡），92B 需 $\ge 24$ 卡。**超过单卡容量的模型必须用 FSDP/ZeRO-3，不是 DDP**。详见 [[Distributed Data Parallel]] 的适用边界。

> [!warning] 误区 5：激活估小了（忽略 attention 矩阵 $s^2$ 项）
> 不用 FlashAttention 时，attention 的 $QK^T$/softmax 是 $bn_hs^2$，长序列平方爆炸。$s=8192$ 时 92B 标准 attention 激活可达 **TB 级**。长序列训练**必须** FA + ckpt，否则激活反超 optimizer 成瓶颈。

> [!warning] 误区 6：分片后单卡显存只算 $16N/P$
> FSDP all-gather 时会瞬间拉一份**全量大张量**到单卡做计算（虽然算完即弃），峰值远高于 $16N/P + M_{\text{act}}$。还要留通信临时缓冲。预留 20% 余量，用 [[GPU memory snapshot]] 实测。

> [!warning] 误区 7：MoE 用激活参数算显存
> MoE 模型显存按**总参数**算（所有专家权重都要存），不是激活参数。DeepSeek-V3 总 671B 但激活 37B，显存按 671B 算（$16 \times 671 = 10.7$TB 训练态），算力按 37B 算。详见 [[参数量计算]] §3.5、[[MoE]]。

## 8. 延伸细节

### 8.1 8-bit Optimizer / 1-bit Adam 省显存

把 Adam 的 $m, v$ 从 fp32 量化到 int8（如 `bitsandbytes` 8-bit Adam），optimizer state 从 $12N$ 降到 $\approx 6N$（仍需 master $4N$ + 8bit $m,v \approx 2N$），训练态 $16N \to \approx 10N$。92B 从 1472GB → 920GB，省 38%。代价：精度略降、需特殊实现。LLM 训练 8-bit optimizer 已较常见（节省大、精度损失小）。

### 8.2 Adafactor / low-rank optimizer state

Adafactor 把 $m, v$ 的尾数为 $\{1, \sqrt{t}\}$ 因子分解，state 显存 $\approx 2N$（而非 $8N$）。训练态 $\approx 6N$（master $4N$ + state $2N$），92B 约 552GB。但 Adafactor 在大模型上稳定性不如 Adam，需仔细调参。

### 8.3 offloading：把 optimizer state 卸到 CPU/NVMe

[[offloading (CPU-NVMe)]] 把最大的 optimizer state（$12N$）卸到 CPU 内存或 NVMe SSD，单卡只留权重+梯度+激活（$4N + M_{\text{act}}$）。92B 单卡 $4N = 368$GB 仍超 80GB，故 offload 通常与 FSDP 配合（FSDP 切分后再 offload）。代价：每步 optimizer 更新要 CPU↔GPU 传 $12N$，PCIe 带宽瓶颈，训练变慢。适合"显存极紧但 CPU 内存大"的场景（如单机 8 卡训 70B）。

### 8.4 通信显存

FSDP/ZeRO-3 前向/反向时 all-gather 拉一份完整 layer 权重到单卡做计算，算完 reduce-scatter 释放。**峰值时单卡会同时持有：被 all-gather 的全量 layer 权重 + 自己分片的 optimizer + 激活 + 通信缓冲**。这就是"分片了但峰值仍可能 OOM"的来源。预分配缓冲（`forward_prefetch`、`reshard_after_forward`）可压峰值。详见 [[Fully Sharded Data Parallel|FSDP]]。

### 8.5 实测对照：用 GPU memory snapshot

纸笔估算是**上限**，实际 PyTorch 还有 framework overhead（CUDA context ~300MB、cuBLS workspace、临时张量碎片）。用 [[GPU memory snapshot]] 或 `torch.cuda.max_memory_allocated()` 抓真实峰值对照。经验：实际峰值 ≈ 估算 × 1.05~1.15。

### 8.6 训练态显存的"分母口诀"

快速估卡数口诀（H100 80GB，留余量按 64GB 可用）：

$$
P_{\min} \approx \frac{16N}{64\text{GB}} = \frac{N}{4\text{GB}}
$$

- 7B → $\approx 2$ 卡（实际单卡 FSDP 不够，需 ZeRO-2 或 2 卡 FSDP）
- 13B → $\approx 4$ 卡
- 70B → $\approx 18$ 卡（实际 32+ 卡，含激活/通信）
- 92B → $\approx 23$ 卡（实际 32-64 卡）
- 405B → $\approx 102$ 卡（实际数百卡，含 TP+PP）

这是"能装下"的下限，**不是"训得快"的推荐**（推荐卡数 = 算力目标 ÷ MFU ÷ 单卡峰值，见 [[MFU与算术强度]]）。

### 8.7 ZeRO 的三个阶段显存账

[[ZeRO (DeepSpeed)]] 的 ZeRO-1/2/3 逐级分片，单卡训练态：

| 阶段 | 分片对象 | 单卡训练态 | 92B / P=32 |
|---|---|---|---|
| ZeRO-0 | 不分（纯 DP） | $16N$ | 1472GB OOM |
| ZeRO-1 | optimizer state | $2N+2N+12N/P$ | $4N + 12N/32 = 403$GB OOM |
| ZeRO-2 | + gradient | $2N + (2N+12N)/P$ | $2N + 14N/32 = 390$GB OOM |
| ZeRO-3 | + 全部（权重也分） | $16N/P$ | $46$GB OK |

可见 92B 在 32 卡下，**只有 ZeRO-3/FSDP 能装下**（46GB/卡），ZeRO-1/2 仍 OOM。这正是 FSDP/ZeRO-3 是大模型训练标配的原因。
