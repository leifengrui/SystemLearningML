## 1. 一句话定义

**计算图（computation graph，计算图）** 是把一个复合运算拆成"节点 + 有向边"的**有向无环图（DAG）**：节点是张量或算子，边是数据流向；它让"前向算数值、反向算梯度"有了可执行的拓扑结构，是 [[autograd机制]] 和 [[backward过程]] 的物理载体。

> [!note] 解答：这和推理的"入图"是一个东西吗？CUDA Graph 呢？
> **不是一回事。** 它们都叫"图"，但处在**不同层级、解决不同问题**，新手极易混淆，下表对清楚：
>
> | 概念 | 所在层级 | 是什么 | 解决什么问题 | 生命周期 |
> |---|---|---|---|---|
> | **计算图 / autograd 图**（本笔记主题） | 框架/Python 逻辑层 | 前向执行时隐式记录的"op + 张量"DAG，存在每个张量的 `.grad_fn` 里 | **让自动微分可行**：反向时沿图回传梯度 | 动态图每次前向重建，`backward` 后默认释放 |
> | **CUDA Graph** | GPU 驱动/硬件层 | 把一段 GPU 上的 **kernel 启动序列**捕获成一个可重放对象，一次 CPU 调用就能重放整段 | **消除 CPU 端 kernel launch 开销 + 同步开销**，提吞吐 | 捕获一次，反复 replay，直到模型结构变 |
> | **推理"入图"/图捕获**（如 TensorRT build、ONNX 导出、`torch.compile` trace） | 部署/编译层 | 把模型的前向 op 序列**固化成一张静态图**，再做融合、内存规划、量化 | **全局优化 + 部署**：算子融合、常量折叠、序列化 | 编译一次，部署期固定用这张优化图 |
>
> 三个一句话区分：
> - 计算图 = "**记我做了什么，好反推梯度**"（为求导服务）。
> - CUDA Graph = "**把 GPU 要跑的 kernel 序列录下来，下次一键重放**"（为省 launch 开销服务，纯 GPU 执行层）。
> - 推理入图 = "**把模型固化成静态图，做全局优化/部署**"（为快和部署服务）。
>
> 它们会**叠用**但概念独立：例如 `torch.compile` 可能把模型编译成一张静态优化图（推理入图的范畴），再在 GPU 上用 CUDA Graph 重放执行；但 autograd 计算图仍在前向时隐式建立（只要 `requires_grad=True`）。所以"图"这个词在不同语境指不同对象，**看到"图"先问在哪个层级、解决什么**。详见 [[torch.compile]]、[[CUDA Graph]]（待建）、推理系统章。

## 2. 为什么需要它（动机与背景）

设想你要最小化一个损失：

$$
L = \text{MSE}(W x + b,\; y)
$$

手算梯度时，你必须把 $L$ 对 $W$、对 $b$ 的偏导一行行推出来。模型只有两层时还行，但 LLM 有上百层、上千亿参数，**人力根本不可能手算**。

于是问题变成：能不能让程序**自动**算任意复杂函数的导数？答案需要两步：

1. **把函数拆成基本算子**（加减乘除、matmul、激活），每个基本算子的导数已知；
2. **按组合顺序，用链式法则把基本算子的导数串起来**。

这个"拆 + 串"的结构，就是计算图。有了图，反向传播只是**沿图反向遍历、逐节点乘导数**，复杂度和前向同阶。

> [!note] 一句话总结
> 计算图存在的唯一目的：**让自动微分变得可行且高效**。没有图，"反向传播"无从谈起。

## 3. 核心概念详解

### 3.1 图的组成

- **节点（node）**：
  - **变量节点**：输入、参数、中间结果（张量）。
  - **算子节点（op）**：`+`、`*`、`matmul`、`relu`、`layernorm` 等。
- **边（edge）**：表示数据依赖 —— 谁是谁的输入。有方向，所以是有向图。
- **叶子节点（leaf）**：用户直接创建、不依赖其他节点的张量（通常是参数 `W, b` 和输入 `x`）。
- **根节点（root）**：最终要被求导的目标，通常是标量 `loss`。

> [!note] 解答：什么是算子？用户什么时候要自己写算子？做训练框架的人会感知算子吗？
> **算子（operator，op）** 是框架里**最小的一个可执行计算单元**，对应计算图里的"算子节点"。每个算子有两面：一个 **forward**（怎么算输出）和一个 **backward / vjp**（怎么把下游传来的梯度反推给输入，供 [[autograd机制]] 使用）。你写 `torch.matmul(a,b)`、`a + b`、`F.softmax`、`nn.Linear`（内部就是 matmul+add）调用的都是算子；调用后结果张量上的 `.grad_fn` 就是这个算子的反向函数指针。
>
> 算子大致分类：
> - **逐元素**：`add`、`mul`、`relu`、`tanh`（每个元素独立）
> - **规约**：`sum`、`mean`、`max`（多元素→少元素，对精度敏感，常保 fp32）
> - **BLAS / 线代**：`matmul`、`gemm`、`layernorm`、`softmax`（计算密集，tensor core 加速主战场）
> - **索引/搬运**：`gather`、`embedding`、`cat`、`reshape`
> - **通信**：`all_reduce`、`all_gather`（分布式里也建模成算子）
>
> **用户什么时候要自己写算子？** 一般三种动机：
> 1. **库里没有的自定义逻辑**：新激活函数、特殊数值稳定处理。
> 2. **算子融合换性能**：把 `layernorm + bias + residual + activation` 融成一个 kernel，少几次 kernel launch、少几次显存往返——这是大模型提速的常见手段（FlashAttention 本质就是一个手写融合算子）。
> 3. **手推比 autograd 默认更优的反向**：比如 softmax 反向有解析式，直接写比让 autograd 自动拆更省显存/更快。
> - 怎么写：继承 `torch.autograd.Function` 实现 `forward`+`backward`（纯 Python 可微即可）；要极致性能就写 **CUDA kernel** 再用 C++ 扩展或 **Triton** 绑进来。[[autograd机制]] §4 有 `Function` 的例子。
>
> **做训练框架 / 训推系统的人会感知算子吗？——深度感知，这是他们的核心工作单位。**
> - **模型算法人**：基本只把算子当 API 调（`F.softmax(x)`），不关心内核，只在缺功能时才想自写。
> - **框架/系统人**：关心每个算子的**计算量、访存量、是否可融合、走哪个后端 kernel、显存占用、是否数值稳定、能否低精度化**。瓶颈分析（见 [[compute vs communication bottleneck]]）就是定位"哪个算子慢"。他们常写融合算子、注册自定义 op、调 op dispatch。可以说**框架人看模型就是一串算子**，性能优化全在算子层做。
>
> 一句话：**算子 = 计算图节点 = (forward + backward) 的最小计算单元**；算法人调它、框架人优化它，FlashAttention、`torch.compile` 算子融合都是"把多个小算子合成一个大算子"的工程。详见 [[算子融合]]（8.1 已提及）、[[autograd机制]]。

### 3.2 前向 vs 反向

```
前向（forward）：  x, W, b  ──>  z = Wx+b  ──>  a = relu(z)  ──>  L = MSE(a, y)
反向（backward）：  L  <──  a  <──  z  <──  W, b   （沿边反向，每条边乘一个局部导数）
```

> [!note] 解答：看不懂这张图，展开讲一讲
> 这张图说的是：**前向从左到右算数值，反向从右到左算梯度，每过一个算子就乘一次它的局部导数（链式法则）**。光看箭头抽象，套一组真数字就懂了。
>
> **设：** $x=1.0,\; W=2.0,\; b=0.5,\; y=3.0$（目标），用单样本 MSE $L=(a-y)^2$。
>
> **前向（从左到右，用前面的值算后面的值）：**
>
> | 步骤 | 公式 | 代入 | 结果 |
> |---|---|---|---|
> | 1 | $z = Wx+b$ | $2.0\times1.0+0.5$ | $z=2.5$ |
> | 2 | $a = \text{relu}(z)$ | $z>0$ 原样 | $a=2.5$ |
> | 3 | $L = (a-y)^2$ | $(2.5-3.0)^2$ | $L=0.25$ |
>
> 前向走完，每个节点的"值"都有了。但训练要的是 $\partial L/\partial W$、$\partial L/\partial b$，得反向算。
>
> **反向（从右到左，每个节点从下游收到"loss 对下游的梯度"，乘自己这个算子的局部导数，得到"loss 对自己的梯度"，再传给上游）：**
>
> | 步骤 | 在哪个节点 | 下游传来的梯度 | 本算子局部导数 | 相乘 = 传给上游的梯度 |
> |---|---|---|---|---|
> | 起点 | $L$ | $\partial L/\partial L = 1$ | — | — |
> | 1 | $L=(a-y)^2$ | $\bar L=1$ | $\partial L/\partial a = 2(a-y)=2(2.5-3)=-1$ | $\bar a = 1\times(-1)=-1$ |
> | 2 | $a=\text{relu}(z)$ | $\bar a=-1$ | $\partial a/\partial z = 1$（因 $z>0$） | $\bar z = -1\times1=-1$ |
> | 3a | $z=Wx+b$（对 $W$） | $\bar z=-1$ | $\partial z/\partial W = x=1$ | $\bar W = -1\times1 = \mathbf{-1}$ |
> | 3b | $z=Wx+b$（对 $b$） | $\bar z=-1$ | $\partial z/\partial b = 1$ | $\bar b = -1\times1 = \mathbf{-1}$ |
>
> 最终 $W.\text{grad}=-1,\; b.\text{grad}=-1$。优化器做梯度下降：$W \leftarrow W - \eta\cdot(-1) = W+\eta$，$b$ 同理——**$W,b$ 往使 $L$ 减小的方向走**，这就是训练。
>
> **图里那行"每条边乘一个局部导数"的意思**：反向时每经过一条边（即一个算子），就把"下游送来的梯度"乘上"这个算子输出对输入的导数"，得到"对上游节点的梯度"，一路乘到参数。这就是链式法则 $\frac{\partial L}{\partial W}=\frac{\partial L}{\partial a}\frac{\partial a}{\partial z}\frac{\partial z}{\partial W}$ 在图上的具体走法。
>
> 一句话：**前向左→右算"值是什么"，反向右→左算"梯度是什么"；每过一个节点乘一次局部导数，乘到参数就是 `.grad`。** 自动能这套流程的就是 [[autograd机制]]。

### 3.3 静态图 vs 动态图

| 项 | 静态图（Static Graph） | 动态图（Dynamic Graph） |
|---|---|---|
| 代表框架 | TensorFlow 1.x、JAX | PyTorch、TF 2.x(eager) |
| 建图时机 | **先建图，后执行**（compile-then-run） | **边执行边建图**（define-by-run） |
| 每次前向 | 图固定，重复用同一张图 | 每次前向**新建**一张图 |
| 控制流 | 要用 `tf.cond`/`tf.while_loop` 特殊算子 | 直接写 Python `if`/`for`/`while` |
| 调试 | 难（图和代码分离，报错难定位） | 易（就是普通 Python，可断点） |
| 优化空间 | 大（可全局优化、算子融合、内存规划） | 小（每步图不同，难全局优化） |
| 适合 | 大规模部署、固定模型结构 | 研究、动态结构（变长序列、RL rollout） |

> [!tip] PyTorch 的折中
> PyTorch 默认动态图（易用、好调试），同时提供 `torch.compile`（原 TorchScript/TorchDynamo）把动态图**即时编译**成静态优化图，兼顾两者。现代 LLM 训练常用 `torch.compile` 加速。

### 3.4 动态图"每步重建"的含义

PyTorch 里你写的每行前向代码，**执行时**就把对应算子记录进一个隐式图（存在张量的 `.grad_fn` 里）。函数跑完，图就建好；`backward()` 跑完，默认**图被释放**，下次前向再建一张新图。这就是 define-by-run。

### 3.5 retain_graph

默认反向后释放图是为了省内存。但有时一次反向不够（如多次 backward、或要二阶导），需 `retain_graph=True` 保留图。

## 4. 数学原理 / 公式

### 链式法则在图上的体现

设前向链：$u = f(x),\; v = g(u),\; L = h(v)$。则：

$$
\frac{\partial L}{\partial x} = \frac{\partial L}{\partial v}\frac{\partial v}{\partial u}\frac{\partial u}{\partial x}
$$

在计算图上，这就是**从 $L$ 反向走到 $x$，把沿途每条边的局部导数相乘**。图里一个节点可能有多个后继（如 $W$ 同时影响 $z_1$ 和 $z_2$），则把各路径的导数**求和**（多元链式法则）：

$$
\frac{\partial L}{\partial x} = \sum_{\text{后继 } s} \frac{\partial L}{\partial s}\frac{\partial s}{\partial x}
$$

> [!note] 这正是梯度累加的来源
> 一个参数被多条路径影响时，反向传播把各路径梯度**加**到它的 `.grad` 上 —— 这和"batch 累加"是同一回事。

### 为什么反向比正向高效（reverse-mode AD）

对 $n$ 个输入、$m$ 个输出的函数：

- **正向模式**：一次前向可算出 $L$ 对某一个输入的导数，算全输入要 $O(n)$ 次遍历。
- **反向模式**：一次反向可算出 $L$（标量）对**所有**输入的导数，只需 $O(1)$ 次遍历（常数倍前向开销）。

损失 $L$ 通常是标量，而参数有几十亿个，所以深度学习**必然用反向模式**。复杂度 $\approx$ 前向的 2~3 倍。

## 5. 代码示例

```python
import torch

# 动态图：边执行边建图
x = torch.tensor([1.0, 2.0, 3.0])
W = torch.tensor([[1.0, 0.0, -1.0], [0.0, 1.0, 0.0]], requires_grad=True)
b = torch.tensor([0.1, 0.2], requires_grad=True)

# 前向 —— 每行都往隐式图里加节点
z = W @ x + b          # 节点：matmul, add
a = torch.relu(z)      # 节点：relu
L = a.sum()            # 节点：sum（标量根节点）

# 查看图结构
print(z.grad_fn)              # <AddBackward0> —— z 的反向函数
print(z.grad_fn.next_functions)  # 指向上游节点 (matmul, b)
print(a.grad_fn)              # <ReluBackward0>
print(L.grad_fn)              # <SumBackward0>

# 控制流：动态图天然支持 Python if/for
def f(t):
    if t.mean() > 0:        # 每次前向按数据决定走哪条路，图随数据变
        return t * 2
    else:
        return t - 1

# 反向：沿图反推梯度
L.backward()                 # 默认反向后释放图
print(W.grad, b.grad)        # 梯度已填回叶子节点 .grad

# 多次 backward 需要 retain_graph
L2 = (W @ x + b).sum()
L2.backward(retain_graph=True)   # 保留图
L2.backward()                   # 第二次反向才不会报错
```

## 6. 与其他知识点的关系

- **上游（依赖）**: [[Tensor]]（图节点就是张量）。
- **下游（应用）**: [[autograd机制]]（图是 autograd 的数据结构）、[[backward过程]]（反向传播就是遍历计算图）、[[梯度与Jacobian]]（链式法则在图上的具体形式）。
- **对比 / 易混**:
  - **静态图 vs 动态图**：见 3.3 表格。
  - **计算图 vs 神经网络**：网络是一次前向跑出来的图的一个实例，不同 batch 可能图不同（动态结构）。

## 7. 常见误区与易错点

> [!warning] 误区清单
> 1. **以为图是"固定的网络结构"**：动态图里，每次前向可能因 `if`/变长序列而建出不同图，图随数据变。
> 2. **`backward()` 后还试图再用同一张图**：默认图被释放，再次反向会报 `Trying to backward through the graph a second time`，要用 `retain_graph=True`。
> 3. **以为 `no_grad` 关闭的是"建图"**：对，`with torch.no_grad():` 下前向**不建图**，所以省内存、不记梯度，但也不能反向。推理/评估时常用。
> 4. **in-place 操作破坏图**：`x.add_(1)` 这类原地改可能让反向时用的中间值被覆盖而报错或算错。
> 5. **跨图引用**：把上一个图里的张量直接用于下一个图，可能因为 `grad_fn` 指向已释放的图而出错，必要时 `.detach()` 切断。

## 8. 延伸细节

### 8.1 算子融合与静态图优势

静态图能在"全局看到整张图"后做优化：把 `matmul + bias + relu` 融合成一个 kernel，减少 kernel launch 和访存。这是 `torch.compile`、XLA、TensorRT 的核心收益来源。

### 8.2 计算图与梯度检查点

[[gradient-checkpointing]]（梯度检查点）通过**不存中间激活、反向时重算前向**来换显存，本质是改写计算图的"哪些节点需要缓存激活"。

### 8.3 vmap / 函数式变换

JAX 的精髓是把计算图当成可变换的一等对象，支持 `vmap`（自动批量化）、`grad`（自动微分）、`jit`（编译）组合。PyTorch 的 `torch.func`（原 functorch）也提供类似能力，对单样本代码自动批量化。

### 8.4 反向模式 vs 正向模式的取舍

- 参数多、损失标量 → **反向模式**（深度学习全部用这个）。
- 输出多、输入少（如算 Jacobian 的一列）→ 正向模式更划算，PyTorch 可用 `torch.autograd.functional.jvp`。
