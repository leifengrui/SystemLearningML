# 数值一致性测试

> **所属章节**: [[测试]]
> **所属模块**: [[20-可靠性与开源工程]]
> **别名**: 数值一致性 / atol rtol / 容差比较 / bit-level 一致 / 跨设备比较 / 浮点一致性 / 训推一致性
> **难度**: 中高（需懂 IEEE 754 浮点、[[数值类型与精度]]、归约顺序、[[mixed precision training]]、训推一致性）


## 1. 一句话定义

**数值一致性测试**是 ML 测试里专门对付"浮点非精确"的一类——ML 计算几乎全是浮点，**两个本应相等的浮点数几乎不会 bit-level 相同**（因浮点非结合、归约顺序变、GPU/CPU 算法不同），故用**容差比较** `torch.allclose(a, b, atol, rtol)`：$|a-b|\le\text{atol}+\text{rtol}\cdot|b|$，**atol**（绝对容差，absolute tolerance）容许小数值附近的绝对误差、**rtol**（相对容差，relative tolerance）容许大数值附近的相对误差；**bit-level 一致**要求二进制完全相同（最严，需确定性运算，难达成因浮点非结合律——$(a+b)+c\ne a+(b+c)$）；**跨设备比较**（GPU vs CPU、不同 GPU 架构、bf16 vs fp32）是 ML 特有难题——同一算子在不同 backend 的实现/归约顺序不同，结果有合法数值差；典型场景是**训推一致性测试**（rollout 用 vLLM 算的 logp vs trainer 用 FSDP 算的 logp，同 $\theta$ 应相同但 TIM 不一致，见 [[rollout train reference logprob一致性]]）；**为什么难 bit-level**——浮点运算**顺序敏感**（reduction 的树形 vs 线性、并行归约的切分粒度变），同输入不同顺序得不同 bit；测试用例设计要 cover **边界值（大数/小数/NaN/Inf/零/下溢/溢出）**；与 [[单元测试与分布式测试]] 互补——后者测逻辑/形状/切分，本条专测数值容差与跨后端一致性。

> [!note] 三句话定位
> - **是什么**：用 atol/rtol 容差比较两浮点张量是否"足够相等"；bit-level 一致要求二进制同（难）；跨设备/GPU/CPU/精度比较有合法差。
> - **为什么**：浮点非结合、归约顺序变、不同 backend 算法不同，同输入得不同 bit，但功能正确——必须容差比较，否则假阳。
> - **训推一致性**：rollout（vLLM）vs trainer（FSDP）同 θ 算 logp 应同，TIM 不一致污染 RL ratio——是本条在 RL 的特化，见 [[rollout train reference logprob一致性]]。


## 2. 为什么需要它（动机与背景）

### 2.1 浮点非精确的本质

IEEE 754 浮点数是**有限精度实数近似**——fp32 仅 23 位尾数（~7 位十进制），fp16 仅 10 位（~3 位），bf16 仅 7 位（~2 位）。每次运算有**舍入误差**（round-off），误差累积。两个"数学上等价"的计算路径因**运算顺序/算法不同**，舍入路径不同，结果 bit 不同——但都在合法误差范围，**功能都对**。用 `==` 比较几乎必失败，必须容差。

### 2.2 浮点非结合律（核心难点）

$$(a+b)+c \ne a+(b+c) \quad \text{(一般情况)}$$

例：$a=1, b=10^{-8}, c=-1$（fp32）。
- $(a+b)+c = (1+10^{-8})+(-1) = 1.00000001 + (-1) = 10^{-8}$（$1+10^{-8}$ 在 fp32 能表示，再减 1 得 $10^{-8}$）。
- $a+(b+c) = 1+(10^{-8}-1) = 1+(-0.99999999) = 0.00000001$（$10^{-8}-1$ 在 fp32 因精度丢成 $-1$，再 $1+(-1)=0$，丢失 $10^{-8}$）。

两路径数学等价，结果不同（甚至一个 0 一个非 0）。**归约顺序变**（树形 vs 线性、并行切分粒度变）→ 每次舍入路径不同 → bit 不同。GPU 的并行归约（warp/block/grid）顺序随配置变，CPU 串行归约不同，故 GPU vs CPU 的 sum 结果 bit 不同。

### 2.3 跨后端的合法差异

- **cuDNN 的 convolution**：多种算法（Winograd、FFT、implicit GEMM），选不同算法结果 bit 不同。
- **cuBLAS 的 GEMM**：tile 切分、kernel 选择随 shape 变，归约顺序变，bit 不同。
- **CPU（MKL/OpenBLAS）vs GPU（cuBLAS）**：实现不同，bit 不同。
- **不同 GPU 架构**（Ampere vs Hopper）：tensor core 的 bf16 累加精度/路径不同，bit 不同。
- 这些都是**合法差**（allclose 过），非 bug。

### 2.4 训推一致性的 ML 特例

RL 框架的 rollout（vLLM 推理框架，BF16 forward + 优化 kernel）与 trainer（训练框架，FP32 master + BF16 forward）算同 $\theta$ 的 logp，**数值不一致**（TIM，训推不一致）。这让 PPO/GRPO 的 ratio $\rho=\exp(\log\pi_\theta-\log\pi_{old})$ 在 $\theta_{new}=\theta_{old}$ 时偏离 1（应等于 1），污染 IS 修正→梯度有偏→训练崩。这是数值一致性测试在 RL 的特化场景，需专门测同 $\theta$ 下两框架 logp 的 `allclose`。详见 [[rollout train reference logprob一致性]]。


## 3. 核心概念详解

### 3.1 atol / rtol 容差比较

`torch.allclose(a, b, atol=1e-5, rtol=1e-4)`：

$$|a - b| \le \text{atol} + \text{rtol}\cdot|b|$$

- **atol**（绝对容差）：固定阈值，容许**小数值附近**的误差（$|b|$ 小时 rtol 项小，靠 atol 兜底）。如 `atol=1e-5` 容许 $|a-b|<10^{-5}$。
- **rtol**（相对容差）：与 $|b|$ 成正比，容许**大数值附近**的相对误差（$|b|$ 大时 atol 不够，rtol 放大）。如 $|b|=100$，`rtol=1e-4` 容许 $|a-b|<0.01$。
- 两者**相加**——小数值靠 atol，大数值靠 rtol，覆盖全范围。

```python
import torch
a = torch.tensor([1.0, 100.0, 1e-8])
b = torch.tensor([1.00001, 100.01, 2e-8])
torch.allclose(a, b, atol=1e-5, rtol=1e-4)
# 1.0 vs 1.00001: |1e-5| <= 1e-5 + 1e-4*1 = 1.1e-4  OK
# 100 vs 100.01: |0.01| <= 1e-5 + 1e-4*100 = 0.01  OK (临界)
# 1e-8 vs 2e-8: |1e-8| <= 1e-5 + 1e-4*1e-8 ≈ 1e-5  OK (atol 兜底小数)
```

### 3.2 bit-level 一致（最严）

bit-level 一致 = `a.equal(b)`（二进制完全同，连最后一位 ULP 同）。要求：① 确定性运算（无原子 add 的 race、无随机算法），② 同 backend 同算法同顺序。**极难达成**——

- cuDNN 默认选最快算法（非确定），`torch.use_deterministic_algorithms(True)` 强制确定性（可能慢或不支持）。
- atomic add（如 scatter_add、attention 的 softmax 归约）多线程归约顺序非确定，bit 不一致。
- 跨 GPU/CPU 几乎不可能 bit-level（算法/顺序根本不同）。

bit-level 一致只用于**极严场景**（如 RL 的 ratio 一致性要求、跨步复现 bug）。

### 3.3 跨设备比较（GPU vs CPU）

```python
x = torch.randn(1000, 1000)
y_cpu = x.cpu() @ x.cpu().T        # CPU (MKL)
y_gpu = (x.cuda() @ x.cuda().T).cpu()  # GPU (cuBLAS)
torch.allclose(y_cpu, y_gpu, atol=1e-3, rtol=1e-3)  # 通常 True (容差内)
# bit-level: torch.equal(y_cpu, y_gpu)  False (几乎必)
```

- 归约顺序/算法不同 → bit 不同，但容差内 True。
- 大矩阵误差累积，atol/rtol 要放宽（fp32 ~1e-3，bf16 ~1e-1）。
- 跨架构（Ampere vs Hopper）类似——同 CUDA 算但 tensor core 路径不同。

### 3.4 跨精度比较（bf16 vs fp32）

```python
x = torch.randn(1024, 1024)
y_fp32 = x @ x.T                          # fp32 参考
y_bf16 = x.to(torch.bfloat16) @ x.to(torch.bfloat16).T  # bf16
torch.allclose(y_fp32, y_bf16.float(), atol=1e-2, rtol=1e-2)  # True (bf16 精度 ~2 位)
```

bf16 仅 7 位尾数（~2 位十进制），误差大，atol/rtol 放宽到 ~1e-2。详见 [[数值类型与精度]]、[[mixed precision training]]。

### 3.5 训推一致性测试（TIM）

```python
# 同 θ 下, rollout (vLLM) vs trainer (FSDP) 算的 logp 应相等
# 实际 TIM 不一致 (数值差)
logp_rollout = vllm_model(input_ids).log_prob    # vLLM BF16
logp_trainer = fsdp_model(input_ids).log_prob    # FSDP FP32 master + BF16 forward
torch.allclose(logp_rollout, logp_trainer, atol=1e-3, rtol=1e-2)
# 期望 True (θ 相同应等), 实际可能 False -> TIM 污染 ratio
```

TIM 的治理靠——① 统一精度（两端都用 BF16 forward + FP32 master，或都 BF16），② actor 重算 old_log_prob（trainer 用 θ_old 重新 forward 算 logp 当锚点，弃用 rollout 算的），③ 统一算子实现。详见 [[rollout train reference logprob一致性]]。

### 3.6 归约顺序的影响（bit 不一致的根）

```
sum 线性 (CPU 串行):  (((a0+a1)+a2)+a3)+...  顺序固定
sum 树形 (GPU 并行):  (a0+a1)+(a2+a3) 等树形归约, 顺序随切分变
sum atomic add (GPU): 多线程原子加, 顺序完全非确定

同输入, 三者结果 bit 不同 (浮点非结合)
```

- CPU `torch.sum` 串行归约（顺序固定）。
- GPU `torch.sum` 并行归约（block 树形 + atomic，顺序随配置变）。
- `torch.use_deterministic_algorithms(True)` 强制 GPU 用确定算法（如不 atomic，串行归约），慢但可复现。

### 3.7 测试用例设计（边界值）

| 边界 | 为什么 | 测什么 |
|---|---|---|
| 大数 | 溢出/大相对误差 | rtol 项主导 |
| 小数（近零） | 下溢/atol 兜底 | atol 项主导 |
| 零 | 除零/eps | 数值稳定（LayerNorm 的 $\sqrt{var+\epsilon}$） |
| NaN/Inf | 传播/除零 | 检测错误传播 |
| 极大极小混 | 抵消/catastrophic cancellation | $(a+b)+c$ 顺序敏感 |
| 负零 | $+0 \ne -0$（IEEE） | 极少但理论存在 |


## 4. 数学原理 / 公式

### 4.1 IEEE 754 浮点表示

fp32：1 符号位 + 8 指数位 + 23 尾数位。值 $= (-1)^s \times 1.m \times 2^{e-127}$。精度 ~7 位十进制，范围 $\pm 3.4\times 10^{38}$。

- **机器 epsilon**（$\varepsilon_{\text{mach}}$）：1 与下一个可表示浮点数之差，fp32 $\approx 1.19\times 10^{-7}$。
- **ULP**（unit in the last place）：尾数最低位，相对误差 $\le 0.5$ ULP。
- 每次运算舍入到最近可表示值，误差 $\le 0.5$ ULP。

### 4.2 容差公式的推导

两数 $a, b$，设 $b$ 为参考真值，$a$ 为计算值，相对误差 $\delta = |a-b|/|b|$。

- 若 $|b|$ 大：$\delta$ 主导，要 rtol $\ge \delta$。
- 若 $|b|$ 小（近零）：$\delta$ 可能爆（分母小），但 $|a-b|$ 也小，靠 atol 兜底。
- 组合：$|a-b| \le \text{atol} + \text{rtol}\cdot|b|$ —— atol 兜小数，rtol 兜大数，全范围覆盖。

经验值（PyTorch 默认 `atol=1e-8, rtol=1e-5` 用于 fp32，但 ML 常放宽）：
- fp32 单算子：`atol=1e-5, rtol=1e-4`。
- fp32 跨设备：`atol=1e-4, rtol=1e-3`（归约顺序差累积）。
- bf16/fp16：`atol=1e-2, rtol=1e-2`（精度低）。
- 训推一致性：`atol=1e-3, rtol=1e-2`（两端框架差）。

### 4.3 误差累积的 $O(\sqrt{n})$ 模型

$n$ 个 fp32 数求和，随机舍入误差假设独立，累积误差标准差 $\sim \sqrt{n}\cdot\varepsilon_{\text{mach}}$（随机游走）。故大矩阵 sum 的容差 $\propto \sqrt{n}$，$n=10^6$ 时 $\sim 10^{-3.5}$，需 atol~1e-3。若舍入有偏（非随机），误差 $\sim n\cdot\varepsilon$（线性，更差）。

### 4.4 浮点非结合的误差界

$(a+b)+c$ 与 $a+(b+c)$ 的差 $\le 2\varepsilon\cdot(|a|+|b|+|c|)$（两次舍入）。一般小，但**catastrophic cancellation**（$a \approx -c$，$a+c$ 接近零，丢失有效位）时误差放大几个数量级。归约顺序优化（如 Kahan 求和、配对求和）减误差但仍非 bit-level。

### 4.5 bit-level 不可能的数学根

bit-level 一致要求两路径的**舍入序列完全相同**——每一步的中间值可表示值相同。但 GPU 并行归约的中间值（部分和的舍入）依赖归约树结构，CPU 串行的不同。除非**强制确定算法 + 同 backend + 同顺序**，否则 bit-level 不可能。`torch.use_deterministic_algorithms(True)` 是逼近的官方手段，但部分 op（atomic add）不支持。

### 4.6 TIM 的 ratio 污染

PPO ratio $\rho=\exp(\log\pi_\theta-\log\pi_{old})$。$\theta_{new}=\theta_{old}$ 时 $\rho$ 应 $=1$。但 TIM 让 $\log\pi_\theta^{\text{trainer}} \ne \log\pi_{old}^{\text{rollout}}$（同 θ 下数值差 $\delta$），$\rho=\exp(\delta)\ne 1$。若 $\delta\sim 10^{-2}$，$\rho\sim 1.01$，clip 误触发（$\rho$ 出 $[1-\varepsilon,1+\varepsilon]$），梯度有偏。这是数值一致性测试在 RL 的直接后果——TIM 治理是 ratio 正确的前提。详见 [[rollout train reference logprob一致性]]。


## 5. 代码示例（可选）

### 5.1 atol/rtol 容差比较

```python
import torch

a = torch.tensor([1.0, 100.0, 1e-8, -1.0])
b = torch.tensor([1.0001, 100.01, 1.5e-8, -1.0001])

# 默认 (atol=1e-8, rtol=1e-5) 太严, 大数 fail
print(torch.allclose(a, b))  # False (100 vs 100.01 超严)

# 放宽
print(torch.allclose(a, b, atol=1e-4, rtol=1e-3))  # True

# bit-level (几乎必 False)
print(torch.equal(a, b))  # False

# 逐元素看差异
diff = (a - b).abs()
tol = 1e-4 + 1e-3 * b.abs()
print(diff <= tol)  # tensor([True, True, True, True])
```

### 5.2 跨设备（CPU vs GPU）一致性

```python
import torch

torch.manual_seed(0)
x = torch.randn(2048, 2048)

# CPU 参考 (MKL)
y_cpu = x @ x.T

# GPU (cuBLAS)
y_gpu = (x.cuda() @ x.cuda().T).cpu()

# bit-level 必不同
print(torch.equal(y_cpu, y_gpu))  # False

# 容差内 (归约顺序差, 误差 ~1e-3)
print(torch.allclose(y_cpu, y_gpu, atol=1e-3, rtol=1e-3))  # True

# 最大相对误差
rel_err = ((y_cpu - y_gpu).abs() / (y_cpu.abs() + 1e-8)).max()
print(f"max rel err: {rel_err:.2e}")  # ~1e-4
```

### 5.3 跨精度（bf16 vs fp32）

```python
import torch

torch.manual_seed(0)
x = torch.randn(4096, 4096)

y_fp32 = x @ x.T                                   # fp32 参考
y_bf16 = (x.to(torch.bfloat16) @ x.to(torch.bfloat16).T).float()  # bf16

# bf16 ~2 位精度, atol/rtol 放宽到 1e-2
print(torch.allclose(y_fp32, y_bf16, atol=1e-2, rtol=1e-2))  # True

# 误差量级
rel_err = ((y_fp32 - y_bf16).abs() / (y_fp32.abs() + 1e-8)).mean()
print(f"mean rel err: {rel_err:.2e}")  # ~1e-3
```

### 5.4 训推一致性（TIM 检测）

```python
import torch

def trainer_logp(input_ids, model_fsdp):
    # trainer: FP32 master + BF16 forward
    with torch.no_grad():
        return model_fsdp(input_ids).log_prob  # BF16 forward

def rollout_logp(input_ids, vllm_model):
    # rollout: vLLM BF16 forward + 优化 kernel
    return vllm_model(input_ids).log_prob

# 同 θ (weight synced)
input_ids = torch.randint(0, 32000, (4, 128))
logp_t = trainer_logp(input_ids, model_fsdp)
logp_r = rollout_logp(input_ids, vllm_model)

# θ 相同应 bit-level 一致 (理想), 实际 TIM 有差
if not torch.allclose(logp_t, logp_r, atol=1e-3, rtol=1e-2):
    print("TIM detected: rollout vs trainer logp 不一致")
    diff = (logp_t - logp_r).abs()
    print(f"max diff: {diff.max():.2e}, mean: {diff.mean():.2e}")
    # ratio ρ = exp(logp_t - logp_r) 在 θ 相同时应 =1, 实际偏离
    ratio = torch.exp(logp_t - logp_r)
    print(f"ratio mean: {ratio.mean():.4f} (应=1), max: {ratio.max():.4f}")
```

### 5.5 边界值测试用例

```python
import torch
import pytest

@pytest.mark.parametrize("x,tol", [
    (torch.tensor([1e30, 1e30]), 1e2),       # 大数 (溢出风险)
    (torch.tensor([1e-30, 1e-30]), 1e-30),   # 小数 (下溢)
    (torch.tensor([0.0, 0.0]), 1e-8),        # 零 (除零/eps)
    (torch.tensor([1.0, -1.0]), 1e-8),        # 抵消 (catastrophic)
    (torch.tensor([float('inf')]), None),     # Inf
    (torch.tensor([float('nan')]), None),      # NaN (应检测传播)
])
def test_norm_stability(x, tol):
    if tol is None:
        # Inf/NaN: 输出应不爆 (或正确报错)
        with pytest.raises((ValueError, OverflowError)):
            out = (x - x.mean()) / x.std()
    else:
        out = (x - x.mean()) / (x.std() + 1e-8)  # +eps 防除零
        assert torch.isfinite(out).all(), "got Inf/NaN"
```

## 6. 与其他知识点的关系

- **上游（依赖）**: IEEE 754 浮点、[[数值类型与精度]]（fp32/fp16/bf16 的精度边界）、[[mixed precision training]]（混合精度的舍入路径）、归约算法（树形/线性/atomic）
- **下游（应用）**: [[单元测试与分布式测试]]（数值断言用 allclose）、[[rollout train reference logprob一致性]]（训推一致性是 TIM 的特化）、CI 的数值回归、模型量化一致性测试
- **对比 / 易混**:
  - **atol/rtol vs ==**：浮点必须容差，`==` 几乎必失败。
  - **atol/rtol vs bit-level**：前者容许合法差，后者要求二进制同（难）。
  - **数值一致性 vs 功能正确**：allclose 过不代表算法对（可能都错同一偏差），还需功能测（[[单元测试与分布式测试]]）。
  - **TIM vs staleness**：TIM 是同 θ 下数值差（错误偏离），staleness 是 θ 不同步的正确偏离——两者正交，详见 [[rollout train reference logprob一致性]]、[[importance sampling与off-policy correction]]。


## 7. 常见误区与易错点

> [!warning] 误区1：用 == 比较浮点
> 错。浮点几乎必不等（舍入差）。用 `torch.allclose(a, b, atol, rtol)`。

> [!warning] 误区2：atol/rtol 越严越好
> 错。太严假阳（合法数值差报错），太松假阴（真 bug 漏过）。按精度/场景选——fp32 ~1e-5、bf16 ~1e-2、跨设备 ~1e-3。

> [!warning] 误区3：bit-level 一定能达成
> 难。浮点非结合 + 归约顺序变 + 跨 backend 算法不同，除非强制确定性算法 + 同 backend + 同顺序。GPU 的 atomic add 几乎不可能 bit-level。

> [!warning] 误区4：GPU vs CPU 结果应一致
> 错。归约顺序/算法不同，bit 必不同，但容差内 True（合法）。这是浮点特性，非 bug。

> [!warning] 误区5：只测中等数值
> 漏边界。大数（溢出）、小数（下溢）、零（除零）、NaN/Inf（传播）、抵消（catastrophic cancellation）都要测，这些是 bug 高发区。

> [!warning] 误区6：随机种子不固定测数值
> 假阳/假阴。数值测试固定 seed 保证复现，否则 flaky。`torch.manual_seed` + cuda + numpy + random 全固定。

> [!warning] 误区7：TIM 当成 staleness
> 错。TIM 是同 θ 下数值差（错误），staleness 是 θ 不同步的正确偏离。前者治 TIM（精度/算子统一），后者治 sync 频率。两者正交。详见 [[rollout train reference logprob一致性]]。

> [!tip] 实践：容差分层
> 按场景选容差——fp32 单算子 `atol=1e-5, rtol=1e-4`；跨设备 `1e-3`；bf16 `1e-2`；训推一致性 `1e-3~1e-2`。CI 默认跑严，nightly 放宽跑跨设备/精度。

> [!tip] 实践：测边界值
> 用例必含大数/小数/零/NaN/Inf/抵消组合，这些是数值 bug 高发区。parametrize 多组边界值跑矩阵。

> [!tip] 实践：确定性算法逼近 bit-level
> 需 bit-level 时 `torch.use_deterministic_algorithms(True)` + `CUBLAS_WORKSPACE_CONFIG` + 固定 seed，逼近可复现。注意部分 op 不支持（atomic add），`warn_only=True` 降级。


## 8. 延伸细节

### 8.1 catastrophic cancellation（灾难性抵消）

两接近相反的数相加（$a \approx -b$），$a+b$ 接近零，**丢失有效位**。例 fp32：$a=1.0000001, b=-1.0$，$a+b=1.0000001-1.0=10^{-7}$，但 $a$ 与 $b$ 各 ~7 位精度，结果只剩 ~1 位有效。LayerNorm 的 $x-\text{mean}$ 若 mean 接近 x 大小，易抵消。需高精度累加（FP32 master 累加 BF16）。

### 8.2 Kahan 求和（减误差）

Kahan 求和补偿舍入误差：维护一个补偿项 $c$，每次 $y = a - c$（补偿上轮舍入），$t = s + y$，$c = (t - s) - y$（本轮舍入），$s = t$。误差从 $O(n)\varepsilon$ 降到 $O(1)\varepsilon$。但慢，GPU 并行难用（需顺序）。PyTorch 的 `torch.sum` 不用 Kahan（并行归约），误差 $\sim O(\sqrt n)\varepsilon$（随机假设）。

### 8.3 atomic add 的非确定性

GPU 的 `index_add`/`scatter_add`/attention 的 softmax 用 atomic add 并行累加，**归约顺序非确定**（多线程竞争顺序变）→ 同输入不同 run 得不同 bit。`torch.use_deterministic_algorithms(True)` 会强制串行（或排序后累加），慢但确定。ML 训练一般容忍（误差小），但 RL ratio 一致性严，需注意。

### 8.4 训推一致性的 TIM 治理（详）

TIM 的根——vLLM（BF16 forward + 优化 kernel，如 fused softmax/kernel）与 trainer（FP32 master + BF16 forward，aten 标准算子）的**实现/精度不同**。治理：

1. **统一精度**：两端都 BF16 forward + FP32 master（或都纯 BF16）。减精度差。
2. **统一算子**：两端用同一实现（如 trainer 也用 vLLM 的 fused kernel，或 rollout 用 aten 标准）。
3. **actor 重算 old_log_prob**：trainer 用 $\theta_{old}$ 重新 forward 算 logp 当锚点，弃用 rollout 算的——rollout 的 logp 只用于生成，不进 ratio。verl/OpenRLHF 主流做法。
4. **MoE 路由回放**：MoE 模型训推路由不一致需 replay，详见 [[rollout train reference logprob一致性]] §R3。

### 8.5 量化一致性测试

模型量化（INT8/INT4）后，量化前后的输出应 `allclose`（容差放宽，如 INT8 `atol=1e-1`）。测：fp32 参考 vs 量化推理，各层误差累积不爆。量化感知训练（QAT）的目的是让训练态就适应量化误差，减推理时的精度损失。

### 8.6 跨框架一致性（PyTorch vs JAX）

同一模型 PyTorch 与 JAX 实现，同输入同 seed，输出应 `allclose`（容差 ~1e-3，归约/算法不同）。用于验证跨框架移植正确性。bit-level 几乎不可能（两框架算子实现不同）。

### 8.7 与 [[单元测试与分布式测试]] 的协同

- [[单元测试与分布式测试]] 测**逻辑/形状/切分**对不对，数值断言用 `allclose`（容差由本条定）。
- 本条测**数值容差/跨设备/精度**的合理性与边界。
- 两者共同构成 ML 数值正确性测试——前者管"算的逻辑对"，后者管"数值差是否合法"。多 rank 一致性测（[[单元测试与分布式测试]] §3.4）的容差选择也由本条指导（跨 rank 归约顺序变，放宽 atol）。
