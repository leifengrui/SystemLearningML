## 1. 一句话定义

**单元测试与分布式测试**是 ML 工程的测试双支柱——**单元测试**用 **pytest** 测单个函数/算子/模块的正确性（如测一个 LayerNorm 算子：给定输入，输出是否与参考实现对齐、形状对、device 对、dtype 对）；**分布式测试**用 **`torchrun` 起多 rank** 测 [[Distributed Data Parallel|DDP]]/[[Tensor Parallel|TP]] 的**通信与并行切分正确性**——多 rank 各算一片后合并，应与单 rank 全量一致（**多 rank 一致性**），验证并行切分无误（梯度同步对、TP 切 attention/mlp 对、all-reduce/all-gather 顺序对）；与普通软件测试的核心区别——ML 测**数值正确性（浮点容差）+ 形状 + 设备 + dtype**，不只测逻辑对错；测试策略分层——**小规模快速单测**（毫秒级，CI 每次 commit 跑）+ **大规模回归测试**（分钟级，nightly/PR 跑），用 **fixture/mock** 复用环境与数据，**GPU CI 用小卡/限卡**控成本（如 T4/单卡跑多 rank 模拟多机）。[[推理引擎源码|vLLM]]/Megatron 的测试体系都遵循这套——pytest 单测覆盖算子、torchrun 测分布式通信、nightly 跑端到端回归。

> [!note] 三句话定位
> - **是什么**：pytest 测单算子/单函数的数值形状正确性；torchrun 起多 rank 测 DDP/TP 的通信与切分正确性（多 rank 合并应与单 rank 一致）。
> - **为什么**：ML 算子是数值密集，错一点（梯度同步漏一行）训练崩；分布式切分错（TP 切错维度）结果错乱但单测抓不到，必须多 rank 测。
> - **与普通软件测试区别**：测数值（浮点容差）+ 形状 + device + dtype；CI 受 GPU 成本限制，分快速单测 + nightly 回归。


## 2. 为什么需要它（动机与背景）

### 2.1 ML 算子易错，错一点崩训练

- LayerNorm 的 eps 位置错（先减均值再除方差 vs 先归一再减）→ 训练发散。
- 梯度同步漏一个 rank（DDP 的 reducer 漏 bucket）→ 某卡梯度旧 → 收敛错。
- TP 切 attention 头切错维度（列切 vs 行切）→ 结果错乱但 shape 对（单测过，多 rank 才暴露）。
- 这些错**单测抓不到**，必须数值对比 + 多 rank 一致性测。

### 2.2 分布式测试的必要性

单卡算对不代表多卡并行算对——切分（DDP 分数据、TP 分权重、SP 分序列）的切边如果错，单 rank 各自的片是对的，但**合并后与单卡全量不一致**。这种错只在多 rank 场景暴露，单测盲区。故 DDP/TP 的实现必配 torchrun 多 rank 测试，对齐单卡参考。

### 2.3 CI 的 GPU 成本约束

GPU 贵（A100 ~$3/h，H100 ~$10/h），CI 不能像 CPU 项目跑几百并行。策略：① **小卡 CI**（T4/L4 便宜），② **单卡跑多 rank 模拟多机**（`torchrun --nproc_per_node=4` 在一卡上起 4 rank，每 rank `set_device(0)`，逻辑跑通虽无真并行但测通信正确性），③ **nightly/PR 才跑大规模真多卡**（成本可控）。


## 3. 核心概念详解

### 3.1 pytest 基础（单元测试）

```python
# test_norm.py
import torch
import pytest

def layernorm(x, eps=1e-5):
    mean = x.mean(-1, keepdim=True)
    var = x.var(-1, unbiased=False, keepdim=True)
    return (x - mean) / torch.sqrt(var + eps)

def test_layernorm_shape():
    x = torch.randn(2, 3, 4)
    out = layernorm(x)
    assert out.shape == x.shape, f"shape mismatch {out.shape} vs {x.shape}"

def test_layernorm_value():
    x = torch.randn(2, 3, 4)
    ref = torch.nn.functional.layer_norm(x, [4])
    out = layernorm(x)
    assert torch.allclose(out, ref, atol=1e-5, rtol=1e-4), "value mismatch"

@pytest.mark.parametrize("dtype", [torch.float32, torch.float16, torch.bfloat16])
def test_layernorm_dtype(dtype):
    x = torch.randn(2, 3, 4, dtype=dtype)
    out = layernorm(x)
    assert out.dtype == dtype
```

- **断言**：`assert` 测值/形状/dtype，失败给消息。
- **parametrize**：一组参数跑多次（不同 dtype/shape）。
- **mark**：`@pytest.mark.gpu`（标需 GPU）、`@pytest.mark.slow`（标慢，CI skip）。
- **fixture**：复用 setup（建模型/数据/临时文件），scope=`function`/`module`/`session`。

### 3.2 ML 测试的四检查项

| 检查项 | 例子 | 普通软件是否关注 |
|---|---|---|
| **数值正确性** | 输出与参考 `allclose(atol, rtol)` | 否（普通只测逻辑对错） |
| **形状** | `out.shape == expected` | 是 |
| **设备** | `out.device == torch.device("cuda:0")` | 否 |
| **dtype** | `out.dtype == torch.float16` | 否 |

ML 单测**四项都要测**——形状对但 dtype 错（fp32 vs fp16）会让下游 mixed precision 崩；device 对但数值在 CPU/GPU 不一致（[[数值一致性测试]]）也是 bug。

### 3.3 分布式测试（torchrun 多 rank）

```python
# test_ddp.py
import os, torch
import torch.distributed as dist
import torch.nn as nn

def setup():
    dist.init_process_group("nccl" if torch.cuda.is_available() else "gloo")
    rank = dist.get_rank()
    if torch.cuda.is_available():
        torch.cuda.set_device(rank % torch.cuda.device_count())

def test_ddp_grad_sync():
    setup()
    rank = dist.get_rank()
    world = dist.get_world_size()
    torch.manual_seed(0)   # 各 rank 同 seed -> 模型同权重
    model = nn.Linear(8, 8).cuda()
    ddp = nn.parallel.DistributedDataParallel(model, device_ids=[rank])
    x = torch.randn(4, 8).cuda() + rank   # 各 rank 不同数据 (数据并行)
    loss = ddp(x).sum()
    loss.backward()
    # 关键: DDP 应 all-reduce 梯度 -> 各 rank 的 model.grad 相同
    grad_rank0 = ddp.module.weight.grad.clone()
    dist.all_reduce(grad_rank0, op=dist.ReduceOp.SUM)
    grad_rank0 /= world
    # 各 rank 的 grad 应等于 mean (all-reduce 后)
    assert torch.allclose(model.weight.grad, grad_rank0, atol=1e-5), "DDP grad not synced"
```

跑：`torchrun --nproc_per_node=2 test_ddp.py`，2 rank 起来，init_process group，DDP 包 model，backward 后验梯度是否 all-reduce 同步。

### 3.4 多 rank 一致性（核心）

**多 rank 各算一片 → 合并应 = 单 rank 全量**。是分布式切分正确性的金标准。

```
单 rank (参考):
  x = [d0, d1, d2, d3]  (全量)
  out_single = model(x)

多 rank DDP (每 rank 一片数据):
  rank0: x0 = [d0, d1], out0 = model(x0)
  rank1: x1 = [d2, d3], out1 = model(x1)
  all_gather: out_multi = [out0, out1]  (沿 batch 维拼)
  
断言: out_multi == out_single  (allclose)  -> 切分+同步无误
```

- **DDP 一致性**：分数据并行，all-gather 输出与单卡全量一致。
- **TP 一致性**：切权重，单卡全量 vs 多卡各算一片 all-gather 合并一致。
- **SP 一致性**：切序列，单卡全量 vs 多卡各算一段拼接一致。
- 任何切分实现的正确性都靠这个**多 rank 合并 = 单 rank 全量**测。

### 3.5 单卡模拟多 rank（CI 技巧）

GPU CI 受限——一卡跑 4 rank 模拟多机：

```bash
torchrun --nproc_per_node=4 --rdzv_endpoint=localhost:0 test_ddp.py
# 4 rank 同挤一卡 (set_device(0)), 无真并行但测通信逻辑
# 真多卡留 nightly
```

- 4 rank 同卡，每 rank `set_device(0)`（共享 context）。
- NCCL 在同卡多 rank 仍工作（同设备 communicator）。
- 测的是**切分/同步逻辑**对不对，不是性能。
- 真多卡/多机 nightly 或 PR 队列跑。

### 3.7 测试分层

| 层 | 范围 | 耗时 | CI 策略 |
|---|---|---|---|
| **单元测试** | 单算子/函数 | 毫秒~秒 | 每次 commit 必跑 |
| **集成测试** | 模块组合（一 layer + backward） | 秒~分钟 | commit/PR 跑 |
| **分布式测试** | 多 rank 通信/切分 | 秒~分钟（小规模） | PR 跑（单卡多 rank） |
| **端到端回归** | 小训练/推理跑通 | 分钟~小时 | nightly / PR 大改时 |
| **大规模回归** | 多卡多机长跑 | 小时+ | nightly |


## 4. 数学原理 / 公式

### 4.1 数值容差（atol/rtol）的选择

ML 算子测数值正确性用容差比较（详见 [[数值一致性测试]]）：

$$|a - b| \le \text{atol} + \text{rtol}\cdot|b|$$

- **fp32 参考**：`atol=1e-5, rtol=1e-4`（PyTorch 默认）。
- **fp16/bf16**：`atol=1e-3, rtol=1e-2`（精度低容差大）。
- **跨设备**（CPU 参考 vs GPU）：`atol=1e-3`（不同归约顺序误差累积）。
- 选错容差——太严假阳（合法数值差报错），太松假阴（真 bug 漏过）。

### 4.2 DDP 梯度同步的数学验证

DDP 的 backward 对各 rank 的梯度做 **all-reduce mean**：

$$g_{\text{synced}} = \frac{1}{W}\sum_{r=0}^{W-1} g_r$$

单 rank 全量参考梯度：

$$g_{\text{single}} = \nabla_\theta \frac{1}{N}\sum_{i=0}^{N-1} \ell_i$$

其中 $N$ 全量样本，$W$ rank 数，每 rank $N/W$ 样本。若 DDP 正确，$g_{\text{synced}} = g_{\text{single}}$（allclose）。测：单卡全量 backward 算 $g_{\text{single}}$，多卡各算 $g_r$ all-reduce mean 得 $g_{\text{synced}}$，断言 `allclose(g_synced, g_single)`。

### 4.3 TP 切分的数学一致性

设 $Y = XW$，TP 切 $W$ 按列 $\text{col}$：$W = [W_0 | W_1]$，rank 0 算 $Y_0 = XW_0$，rank 1 算 $Y_1 = XW_1$，all-gather $Y = [Y_0 | Y_1]$。单卡参考 $Y_{\text{single}} = XW$。若切分正确，`allclose([Y_0,Y_1], Y_single)`。若切错维度（行切当列切）shape 对但值错，多 rank 一致性测暴露。

### 4.4 测试覆盖的统计意义

设单测覆盖 $P$ 比例代码，bug 漏过概率 $\propto (1-P)$。分布式逻辑复杂度 $C$（切分/同步/通信顺序），bug 期望数 $\propto C\cdot(1-P)$。ML 分布式 $C$ 大（DDP/TP/SP/MoE 路由），需更高 $P$——故分布式测试不能省，nightly 必跑。


## 5. 代码示例（可选）

### 5.1 pytest 单元测试（含 fixture + parametrize + mark）

```python
# test_norm.py
import torch
import torch.nn as nn
import pytest

@pytest.fixture(params=[(2,3,4), (1,1,16), (8,16,32)])
def x(request):
    return torch.randn(*request.params)

@pytest.mark.parametrize("dtype", [torch.float32, torch.bfloat16])
def test_layernorm(x, dtype):
    x = x.to(dtype)
    m = nn.LayerNorm(x.shape[-1]).to(dtype)
    out = m(x)
    # 1. shape
    assert out.shape == x.shape
    # 2. device
    assert out.device == x.device
    # 3. dtype
    assert out.dtype == dtype
    # 4. 数值 (与手算参考比)
    mean = x.mean(-1, keepdim=True).to(torch.float32)
    var = x.var(-1, unbiased=False, keepdim=True).to(torch.float32)
    ref = ((x.float() - mean) / torch.sqrt(var + 1e-5)).to(dtype)
    assert torch.allclose(out, ref, atol=1e-2, rtol=1e-2)
```

跑：`pytest test_norm.py -v`。

### 5.2 torchrun 分布式测试

```python
# test_ddp_grad_sync.py
import os, torch
import torch.distributed as dist
import torch.nn as nn

def setup():
    dist.init_process_group("gloo" if not torch.cuda.is_available() else "nccl")
    rank = dist.get_rank()
    if torch.cuda.is_available():
        torch.cuda.set_device(rank)

def teardown():
    dist.barrier()
    dist.destroy_process_group()

def test_ddp_grad_consistency():
    setup()
    rank, world = dist.get_rank(), dist.get_world_size()
    torch.manual_seed(42)
    model = nn.Linear(8, 8)
    if torch.cuda.is_available():
        model = model.cuda()
        torch.cuda.set_device(rank)
    ddp = nn.parallel.DistributedDataParallel(
        model, device_ids=[rank] if torch.cuda.is_available() else None
    )
    # 各 rank 不同数据 (数据并行)
    torch.manual_seed(rank)
    x = torch.randn(4, 8)
    if torch.cuda.is_available(): x = x.cuda()
    loss = ddp(x).sum()
    loss.backward()
    # 收集各 rank 梯度 -> 应相同 (all-reduce mean 后)
    grads = [torch.zeros_like(model.weight.grad) for _ in range(world)]
    dist.all_gather(grads, model.weight.grad)
    for g in grads:
        assert torch.allclose(g, grads[0], atol=1e-5), "grad not synced across ranks"
    teardown()

if __name__ == "__main__":
    test_ddp_grad_consistency()
```

跑：`torchrun --nproc_per_node=2 test_ddp_grad_sync.py`。

### 5.3 多 rank 合并 vs 单 rank 全量（TP 一致性）

```python
# test_tp_consistency.py
import torch, torch.distributed as dist
from megatron.core.tensor_parallel import ColumnParallelLinear  # 示例

def test_tp_vs_single():
    dist.init_process_group("nccl")
    rank, world = dist.get_rank(), dist.get_world_size()
    torch.cuda.set_device(rank)
    torch.manual_seed(0)
    x_single = torch.randn(4, 8, device="cuda")
    W = torch.randn(8, 16, device="cuda")
    Y_single = x_single @ W                      # 单 rank 全量参考

    # TP: 切 W 按列, 每 rank 算一片
    col = W.chunk(world, dim=1)[rank]
    Y_part = x_single @ col                        # 各 rank 一片
    Y_parts = [torch.zeros_like(Y_part) for _ in range(world)]
    dist.all_gather(Y_parts, Y_part)
    Y_multi = torch.cat(Y_parts, dim=1)           # 拼接
    assert torch.allclose(Y_multi, Y_single, atol=1e-4, rtol=1e-3), "TP split wrong"
```

- **上游（依赖）**: [[torch.distributed]]（分布式测试的通信底座）、[[Distributed Data Parallel]]（DDP 测的对象）、[[rank与world size]]（多 rank 身份）、pytest 框架

- **对比 / 易混**:
  - **单元测试 vs 集成测试**：单算子 vs 模块组合；单测快、抓局部 bug，集成测慢、抓组合 bug。
  - **单 rank 单测 vs 多 rank 分布式测**：前者测算子逻辑，后者测切分/通信逻辑；多 rank 一致性是后者的金标准。
  - **ML 测试 vs 普通软件测试**：ML 加测数值（容差）+ 形状 + device + dtype；普通只测逻辑对错。
  - **与 [[数值一致性测试]] 互补**：本条测逻辑/切分/形状，[[数值一致性测试]] 专测浮点容差/跨设备一致性。


## 7. 常见误区与易错点

> [!warning] 误区1：单测过了多卡就对
> 错。切分（DDP 分数据/TP 分权重/SP 分序列）的切边错误单测抓不到——单 rank 各自的片对，合并错。必须多 rank 一致性测。

> [!warning] 误区2：用 == 比较浮点
> 错。浮点非精确，`==` 几乎必失败。用 `torch.allclose(a, b, atol, rtol)`。详见 [[数值一致性测试]]。

> [!warning] 误区3：只测 fp32
> 不够。生产用 bf16/fp16，需 parametrize 多 dtype 测——低精度下 bug（溢出/下溢/eps 位置）fp32 不暴露。

> [!warning] 误区4：分布式测试要真多机
> 不必。单卡多 rank（`--nproc_per_node=4` 同卡）测通信逻辑，CI 友好。真多机 nightly。

> [!warning] 误区5：测试不测形状/dtype/device
> 漏检。形状对但 dtype 错（fp32 vs fp16）下游崩；device 错（CPU vs CUDA）runtime 崩。四项全测。

> [!warning] 误区6：CI 跑所有测试
> 太慢太贵。分层——快速单测每次 commit、分布式 PR、端到端/大规模 nightly。用 mark 分组。

> [!warning] 误区7：随机种子不固定
> 测试用随机输入但不固定 seed → 偶发失败（flaky）。`torch.manual_seed(0)` 固定，复现性好。数值一致性测尤其敏感。

> [!tip] 实践：ML 测试四件套
> ① pytest 单测（算子级，四检查项）；② torchrun 分布式测（多 rank 一致性）；③ nightly 端到端（小训练/推理跑通）；④ 性能 benchmark（[[benchmark与regression test]]）。四层缺一不可。

> [!tip] 实践：单卡多 rank CI
> GPU 贵，CI 用单卡多 rank 模拟（`--nproc_per_node=4` 同卡），测通信/切分逻辑。真多卡 nightly 或 PR 关键改动跑。

> [!tip] 实践：fixture 复用 + parametrize 矩阵
> fixture 复用 model/data setup（scope 控生命周期），parametrize 跑 dtype×shape×device 矩阵，覆盖率高。


## 8. 延伸细节

### 8.1 Megatron / vLLM 的测试体系

- **Megatron-LM**：`tests/` 下 pytest 单测（attention/mlp 算子）+ torchrun 分布式测（TP/PP/EP 切分一致性）+ nightly 端到端（GPT 训练收敛）。CI 跑 `--nproc_per_node=8` 单卡模拟（或真 8 卡 PR）。
- **vLLM**：`tests/` pytest 单测（kernel/sampling/scheduler）+ 多 GPU 测（`test_multigpu.py` 起多 rank 测 TP）+ nightly 端到端吞吐/延迟。用 `pytest -m distributed` 分组。
- 两者都遵循"快速单测 + 分布式测 + nightly 回归 + benchmark"分层。

### 8.2 弹性训练的测试（[[elastic training]]）

弹性训练（节点增减、rank 重启）的测试需模拟故障——kill 某 rank、加 rank、torchrun `--rdzv` 重新 rendezvous。测的是**故障后能否续训、状态是否一致**。比普通分布式测更复杂，需 chaos 测试框架（如 pytest + 模拟 kill）。

### 8.3 NCCL/CUDA 错误的测试

分布式测试会遇 NCCL 超时、CUDA OOM、nccl-tests 通信错。测试要：① 设 NCCL 超时（`NCCL_TIMEOUT`）防 hang；② OOM 用 `pytest.raises` 测异常路径；③ 通信错用 mock 模拟。CI 环境 NCCL 版本/驱动要固定。

### 8.4 数值测试的种子与确定性

测试数值一致性要**固定种子**（`torch.manual_seed`、`torch.cuda.manual_seed_all`、`random.seed`、`numpy.seed`）保证复现。但有些 op 非确定性（cuDNN 某些算法、atomic add 的归约顺序），需 `torch.use_deterministic_algorithms(True)`（可能慢或部分 op 不支持）。详见 [[数值一致性测试]]。

### 8.5 mock 与 stub（分布式隔离）

不真起多进程测分布式逻辑——mock `dist` 的 all_reduce/all_gather 返回预设值，测调用者逻辑（如 DDP 的 reducer 何时调 all_reduce）。适合 CI 无 GPU/无多 rank 场景，测的是**调用正确性**而非**通信正确性**。真通信验证仍需 torchrun。

### 8.6 与 [[benchmark与regression test]] 的分工

- **本条（功能测试）**：测对错（数值/形状/切分逻辑对不对）。
- **[[benchmark与regression test]]**：测快慢（吞吐/延迟/显存）。
- 两者互补——功能对但性能退化也要抓。CI 跑功能测 + nightly 跑 benchmark。

### 8.7 与 [[数值一致性测试]] 的协同

本条的数值断言（`allclose(atol, rtol)`）的具体容差选择、跨设备对比、bit-level 一致性由 [[数值一致性测试]] 详述。本条侧重"测什么"（算子/分布式逻辑），[[数值一致性测试]] 侧重"怎么测数值"（容差/设备/精度）。两者共同构成 ML 数值正确性测试。
