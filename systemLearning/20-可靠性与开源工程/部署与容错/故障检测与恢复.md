# 故障检测与恢复

> **所属章节**: [[部署与容错]]
> **所属模块**: [[20-可靠性与开源工程]]
> **别名**: 故障检测与恢复 / health check / watchdog / 心跳 / 自动重启 / checkpoint 恢复 / fault tolerance / self-healing / RTO / RPO / GPU XID error / ECC error
> **难度**: 中高（需懂 [[elastic training]]、[[NCCL hang与timeout]]、[[async distributed checkpoint]]、GPU 错误码、Python 线程/watchdog）

## 1. 一句话定义

**故障检测与恢复** 是训练系统在**节点崩/通信 hang/GPU 报错/磁盘满**等故障发生时**自动发现 → 自动重启/缩容 → 从 checkpoint 恢复续训**的整套机制——分**检测**（watchdog 监控训练步超时、heartbeat 心跳、NCCL timeout、GPU XID/ECC 内核日志、节点 liveness）与**恢复**（K8s 重启 Pod、torchrun 弹性 re-rendezvous、从最近 ckpt 重载、跳过坏数据样本）两段。大模型训练几千卡天、单次故障人工介入太慢（凌晨崩没人管，几小时空跑烧 Millions），必须**自动恢复**——把"故障 → 人工介入 → 重启"的小时级人工链路压缩到"故障 → 检测 → 自动 ckpt 恢复"的分钟级自愈。**检测**靠 [[metrics logging tracing]] 采集的 metrics + watchdog 线程；**恢复**靠 [[elastic training]] 的 re-rendezvous + ckpt；**关键指标**是 **RTO**（Recovery Time Objective，恢复时间目标，故障到恢复训练的时间）与 **RPO**（Recovery Point Objective，数据丢失量，由 ckpt 间隔决定，最多丢一个 ckpt 间隔的步数）。**核心 trade-off**：ckpt 越密 RPO 越小但 IO 阻塞越多（用 [[async distributed checkpoint]] 缓解）；watchdog 超时阈值越短检测越快但易误报（正常 backward 偶尔卡顿被误判 hang）。

> [!note] 三句话定位
> - **是什么**：watchdog + heartbeat + NCCL timeout + GPU 日志监控做"检测"；K8s 重启 Pod / torchrun elastic re-rendezvous / ckpt 恢复做"恢复"，自动自愈不靠人。
> - **为什么**：大模型训练几小时到几天，单节点故障人工介入太慢，几千卡天烧 Millions；必须自动检测 + 秒级到分钟级恢复。
> - **指标**：RTO（恢复时长，分钟级目标）、RPO（丢的步数，由 ckpt 间隔决定，max=一个 ckpt 间隔）。两者是故障治理的"SLA"。


## 2. 为什么需要它（动机与背景）

### 2.1 大模型训练的故障不可避免

几千卡集群跑几天，单卡/单节点的 MTBF（平均故障间隔）是**小时级**——GPU ECC 单比特错、XID error、网卡掉线、节点 OOM、磁盘满、内核 panic、电源毛刺。几千卡同时跑，**几乎必然有节点在故障**。人工盯 → 凌晨崩没人管 → 几小时空跑。成本 = 卡数 × 时间 × 单卡时薪，几千卡几小时 = 几万到几十万美金。**必须自动**。

### 2.2 NCCL hang 是静默死锁

最阴险的故障是 **NCCL hang**——某 rank 的 allreduce 卡住（网卡丢包、GPU kernel 死锁），其他 rank 永远等。无报错、无崩溃、进程活着、占用 GPU 但**零吞吐**。靠"看吞吐曲线变平"或 watchdog 超时才能发现。详见 [[NCCL hang与timeout]]。watchdog 是发现 NCCL hang 的主力。

### 2.3 GPU 错误码要主动捞

GPU 故障不总是崩进程——很多是**内核日志里报 XID error**（如 XID 79 = "fallen off the bus"，GPU 掉 PCIe 总线）或 **ECC 双比特错**（不可纠正，显存坏了）。这些不直接杀进程，但下一步 kernel 就会 illegal memory access 或挂。要主动**监控 `dmesg` / NVML 的 XID/ECC**，提前把节点标坏、迁任务、重启。

### 2.4 人工介入的代价

故障 → 值班收到告警 → 上线确认 → 手动 kill job → 手动起 → 手动跑恢复脚本 → 等加载。链路几小时。期间卡空跑。**几千卡天烧 Millions**。自动恢复把链路压到"检测秒级 + 恢复分钟级"。这是"自愈"的 ROI。

### 2.5 与 elastic training 的配合

恢复不是"原地重启同一节点"——节点可能真坏了（硬件故障）。**恢复 = 缩容到健康节点 + re-rendezvous + ckpt reload**。这要 [[elastic training]] 的成员动态变化能力。固定训练恢复要手动改 world_size 重启，elastic 自动。所以"故障检测 + elastic + ckpt"是三位一体的容错栈。


## 3. 核心概念详解

### 3.1 检测层：watchdog / heartbeat / NCCL timeout / GPU 日志

```
检测层:
  ├─ watchdog (Python 线程, 监控训练步间隔超时)
  │     每 N 秒检查"上次 step 时间戳", 若超阈值 (如 300s) -> 判 hang -> raise
  ├─ heartbeat (各 worker 定期上报"我还活着"到 coordinator)
  │     coordinator 看谁心跳停 -> 判该 worker 死
  ├─ NCCL timeout (init_process_group(timeout=) / allreduce 超时)
  │     allreduce 卡超 NCCL_TIMEOUT -> 抛 RuntimeError
  └─ GPU 日志监控 (dmesg / NVML / nvidia-smi -q)
        XID error (XID 79 GPU 掉总线, XID 48 ECC 双比特错, ...)
        ECC 错误计数
        GPU util/mem/温度异常
```

- **watchdog**：Python 起一个**后台线程**，每隔 interval 检查"主训练线程最近一次完成 step 的时间戳"。若 `now - last_step_time > threshold`（如 300s）判 hang，raise 中断。抓 NCCL hang（进程活但卡）最有效。threshold 是 trade-off——太短易误报（正常长 backward 偶尔超），太长发现慢。
- **heartbeat**：各 worker 定期（如每 5s）上报心跳到 coordinator（如 K8s etcd、Ray GCS、自研 daemon）。coordinator 看 worker 心跳停超阈值判该 worker 死。是**进程级** liveness（watchdog 是**进程内** liveness）。K8s 的 livenessProbe 是 heartbeat 的内置版。
- **NCCL timeout**：`init_process_group(backend="nccl", timeout=timedelta(seconds=600))`。allreduce 卡超 timeout 抛 `RuntimeError: NCCL error`。是**集合通信级** liveness。但注意 NCCL 默认 timeout 长（10min+），且 hang 时 timeout 可能不触发（卡在 kernel 不回超时），所以不能只靠 NCCL timeout。
- **GPU 日志监控**：主动捞 `dmesg` 里的 `NVRM: Xid` 行、`nvidia-smi -q` 的 ECC 计数、NVML 事件。XID 码语义（部分）：
  - **XID 79**：GPU "fallen off the bus"（掉 PCIe），节点 GPU 不可用，必须重启节点。
  - **XID 48/63**：ECC 双比特错（不可纠正显存错），GPU 显存坏，下个 kernel 挂。
  - **XID 31/43**：GPU 内存页错误（illegal memory access），通常是 kernel bug 或指针错。
  - **XID 13**：GPU 丢失（硬件故障/驱动重载）。
  详见 NVIDIA XID 文档（部分码待核实最新语义）。

### 3.2 恢复层：重启 / elastic re-rendezvous / ckpt / 跳坏样本

```
恢复层:
  ├─ 重启故障 worker
  │     K8s: Pod 挂 -> controller 起新 Pod (新节点或同节点)
  │     torchrun elastic: agent 检测 worker 崩 -> re-rendezvous (缩容)
  │     Ray: actor 死 -> Ray 重启 actor (PlacementGroup 重排)
  ├─ 从最近 ckpt 恢复
  │     load ckpt -> model/optimizer/sampler state -> 从 step=N 继续
  │     异步 ckpt: 不阻塞训练, 见 [[async distributed checkpoint]]
  ├─ 跳过坏数据样本
  │     某样本 OOM/抛错 -> catch + log + skip -> 继续下个 batch
  │     防单个坏样本炸整 job
  └─ 节点标记坏 + 迁移
        故障节点从调度池摘 (K8s cordon/taint) -> 新任务不调上去 -> 维修后归队
```

- **重启故障 worker**：进程崩了由上层重启。K8s 是 Pod controller 自动起新 Pod；torchrun elastic 是 agent + re-rendezvous；Ray 是 actor 重启。**重启后新 worker rank 可能变**（rendezvous 重新分），需从 ckpt 恢复对齐。
- **从 ckpt 恢复**：见 [[elastic training]] §3.5。全员 reload ckpt → 模型/优化器/sampler state → 从 step=N 继续。ckpt 间隔决定 RPO（§4.2）。
- **跳坏样本**：某样本（损坏数据/超长 prompt 触发 OOM）让训练崩。catch 异常 + log + skip 该 batch → 继续下个。是"数据级容错"。但要监控 skip 率（太高说明数据 pipeline 有系统性问题）。
- **节点摘池**：故障节点（XID 79/ECC 双比特）硬件坏了，重启该节点也还会再坏。要从调度池摘（K8s `cordon` + `taint` 让新 Pod 不调上）+ 工单维修 + 修好归队。防止"重启 Pod 调回坏节点"循环崩。

### 3.3 常见故障类型与处理

| 故障类型 | 表征 | 检测手段 | 恢复手段 |
|---|---|---|---|
| NCCL hang | allreduce 卡、进程活但零吞吐 | watchdog 步超时、NCCL timeout、吞吐曲线变平 | elastic re-rendezvous、重启 worker |
| GPU OOM | `CUDA out of memory` | 异常抛出、mem 监控 | 重启 worker + 降 batch/减 grad accumulation |
| CUDA illegal memory access | `XID 31/43`、`illegal memory access` | GPU 日志、异常 | 重启 worker（疑坏 GPU 则摘节点） |
| GPU 掉总线 | `XID 79`、`XID 13` | dmesg 监控 | 重启节点（不是 Pod）、节点摘池维修 |
| ECC 双比特错 | `XID 48/63`、ECC 计数升 | NVML/nvidia-smi -q | 摘节点维修（显存坏不可恢复） |
| 节点掉线 | 心跳停、SSH 不通 | heartbeat 超时 | elastic 缩容、摘节点 |
| 磁盘满 | ckpt 写失败、`No space left` | 磁盘监控告警 | 清旧 ckpt、扩容 PVC |
| 坏数据样本 | OOM/解析错某 batch | 异常 catch | skip 该 batch |
| 进程崩 | exit code ≠ 0 | K8s Pod 状态、torchrun agent | 重启 worker、elastic |

### 3.4 RTO / RPO

```
故障发生 ----[检测时间]---- 检测到 ----[恢复时间]---- 恢复训练
 |<-------------- RTO (Recovery Time Objective) ------------>|
 |                                                            |
 |--丢的步数 = RPO (Recovery Point Objective) --|              |
   (从最近 ckpt 到故障, 最多一个 ckpt 间隔)
```

- **RTO**：故障发生到恢复训练的总时长 = 检测时间 + 决策时间 + 重启/re-rendezvous 时间 + ckpt load 时间 + NCCL init 时间。目标分钟级（大模型几十 GB ckpt load 是大头，用 [[async distributed checkpoint]] 或 RAM disk 压）。
- **RPO**：故障丢失的训练进度 = 从最近 ckpt 到故障点的步数。最多一个 ckpt 间隔（见 [[elastic training]] §4.3）。ckpt 越密 RPO 越小但 IO 阻塞越多。SLA 形式："RPO ≤ 10 步、RTO ≤ 5 min"。

### 3.5 与 elastic training 的关系

故障恢复的"成员变更"段由 [[elastic training]] 承担：

```
检测到 worker 死 (watchdog/heartbeat)
  -> torchrun elastic agent 收到 worker 崩信号
  -> re-rendezvous (剩余 worker 重组, 见 [[elastic training]] §3.2)
  -> 从 ckpt 恢复 (见 [[elastic training]] §3.5)
  -> 缩容续训 (新 world_size = 原来减 1, 若 >= min_nodes)
```

故障检测是 elastic 的**触发器**，elastic 是恢复的**执行器**。两者配合才是完整自愈。固定训练（无 elastic）的恢复要手动改 world_size 重启，慢且易错。


## 4. 数学原理 / 公式

### 4.1 watchdog 超时判定

设训练主线程每次完成 step 时更新时间戳 $t_{\text{last}}$。watchdog 线程每 interval $\Delta$ 秒检查：

$$\text{if } t_{\text{now}} - t_{\text{last}} > \theta \quad \Rightarrow \quad \text{判 hang}, \text{ raise}$$

$\theta$ 是 hang 阈值（如 300s）。trade-off：

- $\theta$ 小 → 检测快，但易误报（正常长 backward/数据加载偶超）。
- $\theta$ 大 → 误报少，但真 hang 时发现慢（多烧卡）。

设正常 step 耗时分布 $P(\tau)$，误报率 $P_{\text{FP}} = P(\tau > \theta)$。漏报（真 hang 但未达 $\theta$）率取决于 hang 发生到检测的延迟 $= \theta + \Delta$。选 $\theta$ 是在 $P_{\text{FP}}$ 与检测延迟间 trade-off。

### 4.2 RPO = ckpt 间隔

ckpt 间隔 $T_{\text{ckpt}}$（按步数 $\lceil T_{\text{ckpt}}/\tau \rceil$ 步或按时间 $T_{\text{ckpt}}$ 秒）。故障时丢的步数：

$$\text{RPO} \in [0, \lceil T_{\text{ckpt}} / \tau \rceil], \quad \mathbb{E}[\text{RPO}] \approx \frac{T_{\text{ckpt}}}{2\tau} \text{ 步}$$

（假设故障均匀发生在两次 ckpt 间）。SLA "RPO ≤ N 步" → $T_{\text{ckpt}} \le N \tau$。ckpt 越密 RPO 越小，但每 $T_{\text{ckpt}}$ 停一次写盘，IO 开销 $T_{\text{save}}$ 占比 $T_{\text{save}} / (T_{\text{ckpt}} + T_{\text{save}})$。用 [[async distributed checkpoint]] 让 ckpt 写盘与训练重叠，减 IO 阻塞。

### 4.3 RTO 分解

$$\text{RTO} = t_{\text{detect}} + t_{\text{restart}} + t_{\text{rdzv}} + t_{\text{ckpt\_load}} + t_{\text{nccl\_init}} + t_{\text{sampler}}$$

- $t_{\text{detect}}$：watchdog/heartbeat 判故障时间（$\le \theta + \Delta$）。
- $t_{\text{restart}}$：K8s 起新 Pod / torchrun agent 重启 worker（秒级到十秒级）。
- $t_{\text{rdzv}}$：re-rendezvous（秒级）。
- $t_{\text{ckpt\_load}}$：**大头**，大模型 ckpt 几十 GB，从 NFS/PVC 读分钟级。用 sharded ckpt（每 rank 只读自己 shard，见 [[async distributed checkpoint]]）或 RAM disk 压。
- $t_{\text{nccl\_init}}$：NCCL communicator 重建（秒到十秒级，IB 握手慢）。
- $t_{\text{sampler}}$：DistributedSampler 重切（毫秒级）。

大模型 RTO 主要被 $t_{\text{ckpt\_load}}$ 主导。减 RTO 的杠杆：① sharded ckpt（并行读）；② RAM disk 缓存 ckpt；③ 异步 ckpt 不阻塞。SLA "RTO ≤ 5 min" 要把 $t_{\text{ckpt\_load}}$ 压到分钟内。

### 4.4 几千卡天成本与自愈 ROI

设 $N$ 卡训练，单卡时薪 $c$（如 H100 ~$2/h spot / $4/h on-demand），故障频率 $\lambda$（次/小时/节点），人工介入耗时 $T_{\text{human}}$（小时），自动 RTO $T_{\text{auto}}$（小时）。单次故障成本：

$$C_{\text{manual}} = N \cdot c \cdot T_{\text{human}}, \quad C_{\text{auto}} = N \cdot c \cdot T_{\text{auto}}$$

自愈节省率 $= (T_{\text{human}} - T_{\text{auto}}) / T_{\text{human}}$。如 $N=2048$、$c=2$、$T_{\text{human}}=2\text{h}$、$T_{\text{auto}}=1/12\text{h}(5\text{min})$：单次故障 $C_{\text{manual}} = 8192$，$C_{\text{auto}} \approx 341$，单次省 ~$7851$。集群 MTBF 几小时一次 → 月省几十万。这是"自愈"的硬 ROI。


## 5. 代码示例（可选）

### 5.1 watchdog 超时检测（Python 线程）

```python
import threading, time, datetime

class TrainingWatchdog:
    """监控训练步间隔超时, 判 NCCL hang / 卡死."""
    def __init__(self, timeout_sec=300, check_interval=10):
        self.timeout = timeout_sec
        self.interval = check_interval
        self.last_step_time = time.time()
        self._stop = threading.Event()
        self._thread = threading.Thread(target=self._run, daemon=True)

    def update(self):
        """主训练线程每完成一步调一次."""
        self.last_step_time = time.time()

    def _run(self):
        while not self._stop.is_set():
            elapsed = time.time() - self.last_step_time
            if elapsed > self.timeout:
                # 判 hang: 进程活但零吞吐
                raise TimeoutError(
                    f"Training hang: no step in {elapsed:.0f}s "
                    f"(> {self.timeout}s). Likely NCCL hang. "
                    f"Last step at {datetime.datetime.fromtimestamp(self.last_step_time)}"
                )
            time.sleep(self.interval)

    def start(self): self._thread.start()
    def stop(self): self._stop.set()

# 用法
wd = TrainingWatchdog(timeout_sec=300, check_interval=10)
wd.start()
for step, batch in enumerate(loader):
    train_step(batch)
    wd.update()              # 完成一步, 告 watchdog 我还活着
```

### 5.2 K8s livenessProbe（heartbeat 的内置版）

```yaml
# Pod spec 加 livenessProbe, K8s 周期探活, 挂了重启 Pod
spec:
  containers:
  - name: trainer
    image: myregistry/train:cuda12.1
    livenessProbe:
      httpGet:
        path: /health        # 训练脚本起个 HTTP server, 返回最近 step 时间
        port: 8080
      initialDelaySeconds: 600   # 训练启动期不探 (load ckpt 慢)
      periodSeconds: 30          # 每 30s 探一次
      failureThreshold: 10      # 连续 10 次失败 (300s 无响应) 判死, 重启 Pod
```

训练脚本里起个轻量 HTTP server，`/health` 返回"最近 step 时间戳"，K8s 探到"超阈值"判死、重启 Pod。这是把 watchdog 逻辑外包给 K8s。

### 5.3 GPU XID 错误监控（捞 dmesg）

```python
import subprocess, re

BAD_XIDS = {79, 48, 63, 13, 31, 43}   # 掉总线 / ECC 双比特 / GPU 丢 / illegal mem access

def check_gpu_errors():
    """捞 dmesg 的 NVRM XID 行, 判 GPU 硬件故障."""
    out = subprocess.run(["dmesg", "--color=never", "-T"],
                         capture_output=True, text=True).stdout
    for line in out.splitlines():
        m = re.search(r"Xid (\d+)", line)
        if m and int(m.group(1)) in BAD_XIDS:
            return {"xid": int(m.group(1)), "line": line, "action": "drain_node"}
    return None

# 周期跑 (如每 60s), 发现 XID -> 标节点坏 -> 触发迁移 + 摘池
```

### 5.4 跳坏数据样本

```python
for step, batch in enumerate(loader):
    try:
        loss = model(batch)
        loss.backward()
        optimizer.step()
        optimizer.zero_grad()
    except (RuntimeError, torch.cuda.OutOfMemoryError) as e:
        # 坏样本 (OOM / 解析错 / 超长) -> 跳过, 不炸整 job
        print(f"[skip] step {step} bad batch: {e}")
        torch.cuda.empty_cache()
        if "out of memory" in str(e).lower():
            torch.cuda.synchronize()  # 清坏 CUDA state
        continue
    wd.update()
```

### 5.6 ckpt 周期保存 + 恢复入口

```python
def save_ckpt(step, model, optimizer, path):
    if dist.get_rank() == 0:
        torch.save({
            "step": step,
            "model": model.state_dict(),
            "optimizer": optimizer.state_dict(),
            "world_size": dist.get_world_size(),
        }, path)
    dist.barrier()

def maybe_recover(path):
    """启动时若有 ckpt, 恢复. 用于 elastic re-rendezvous 后."""
    if os.path.exists(path):
        ckpt = torch.load(path, map_location="cuda")
        model.load_state_dict(ckpt["model"])
        optimizer.load_state_dict(ckpt["optimizer"])
        print(f"[recover] from step {ckpt['step']}, "
              f"old world={ckpt['world_size']} -> new world={dist.get_world_size()}")
        return ckpt["step"]
    return 0
```


## 6. 与其他知识点的关系

- **上游（依赖）**: [[metrics logging tracing]]（metrics 是检测的基础——watchdog 看吞吐、heartbeat 看 liveness、告警基于 metrics）、GPU 硬件知识（XID/ECC 码）、[[NCCL hang与timeout]]（NCCL 级检测与根因）、[[Python multiprocessing与GIL]]（watchdog 后台线程模型）。
- **下游（应用）**: [[elastic training]]（恢复的执行器——re-rendezvous + ckpt 恢复）、[[async distributed checkpoint]]（减 RPO/RTO 的 ckpt 后端）、[[RL worker故障恢复]]（RL 框架的 rollout worker 容错借鉴同样的检测/恢复设计）、大模型训练平台的自愈栈。
- **对比 / 易混**:
  - **故障检测 vs 故障恢复**：前者是"发现"（watchdog/heartbeat/log），后者是"修"（重启/elastic/ckpt）。分开设计但配合用。
  - **watchdog vs heartbeat vs NCCL timeout**：watchdog 是**进程内**步超时检测（抓 NCCL hang 最有效）；heartbeat 是**进程间** liveness（coordinator 看谁掉线）；NCCL timeout 是**集合通信级**（allreduce 卡超）。三者层次不同，互补不替代。
  - **故障恢复 vs elastic training**：前者是整套"检测+恢复"理念，后者是"成员动态变化"的执行机制。故障恢复的"成员变更"段由 elastic 承担。前者管"检测+决策+恢复全链路"，后者管"成员动态"子问题。
  - **同步恢复 vs 异步恢复**：同步恢复是"故障→全员等 ckpt 完→恢复"，RTO 大；异步恢复用 [[async distributed checkpoint]] 让 ckpt 写盘与训练重叠，减 RTO。后者是大模型标配。


## 7. 常见误区与易错点

> [!warning] 误区 1：只靠 NCCL timeout 抓 hang
> 不够。NCCL hang 时进程卡在 kernel，timeout 可能不触发（NCCL 的 timeout 在某些 hang 模式下不回超时）。必须配 watchdog 看步间隔超时。NCCL timeout 是兜底不是主力。

> [!warning] 误区 2：watchdog 阈值越小越好
> 不是。阈值太小（如 30s）易误报——正常长 backward、数据加载偶卡、GC 暂停都可能超。典型 300s（5min）。要在"发现快"与"误报少"间 trade-off。

> [!warning] 误区 3：故障节点重启 Pod 就好
> 不一定。XID 79（掉总线）/ECC 双比特是**硬件故障**，重启 Pod 调回同节点还会再坏。必须**摘节点**（K8s cordon/taint）+ 工单维修。只重启 Pod 是"循环崩"。

> [!warning] 误区 4：ckpt 越密越好（RPO 越小）
> trade-off。ckpt 密 RPO 小但 IO 阻塞多（每 $T_{\text{ckpt}}$ 停一次写盘）。大模型 ckpt 几十 GB，同步 ckpt 每次停分钟级。用 [[async distributed checkpoint]] 让写盘与训练重叠，才能既密又不阻塞。

> [!warning] 误区 5：RTO = ckpt load 时间
> 不全是。RTO = 检测 + 重启 + re-rendezvous + ckpt load + NCCL init + sampler。ckpt load 是大头但不是全部。减 RTO 要全链路压（sharded ckpt 减 load、预连接减 NCCL init、watchdog 减 detect）。

> [!warning] 误区 6：故障检测靠"人看告警"
> 太慢。几千卡凌晨崩，人收告警→上线→确认→手动重启要几小时。必须自动——watchdog 直接 raise 触发 elastic re-rendezvous，不等人。人只做"事后复盘+修坏节点"。

> [!warning] 误区 7：跳坏样本 = 数据偷工
> 不是。跳坏样本是"数据级容错"（防单坏样本炸整 job）。但要监控 skip 率——若 >0.1% 说明数据 pipeline 有系统问题（不是偶发）。skip 是兜底不是常态。

> [!warning] 误区 8：heartbeat 间隔越短越好
> 不是。heartbeat 太密（如每 1s）增加 coordinator 压力 + 网络开销。典型 5-10s。配合"连续 N 次没心跳"判死，防单次网络抖动误判。


## 8. 延伸细节

### 8.1 DCGM exporter 采 GPU metrics

NVIDIA 的 **DCGM exporter** 是个 DaemonSet，跑在每 GPU 节点，采 GPU metrics（util、显存、温度、ECC 错误计数、XID 事件、功耗）暴露成 Prometheus 格式。是 GPU 故障检测的**数据源**——告警规则基于 DCGM metrics（如 `DCGM_FI_DEV_XID_ERRORS > 0` 或 `DCGM_FI_DEV_ECC_SBE_VOL_TOTAL > threshold`）。详见 [[metrics logging tracing]] §3.6。

### 8.2 ncu/nsys 辅助根因分析

故障复盘时用 [[nsys (Nsight Systems)]] 和 [[ncu (Nsight Compute)]] 抓 timeline/kernel metric，看是哪个 kernel 卡、哪个 rank 慢、通信瓶颈在哪。是"事后分析"工具，不是实时检测。[[torch profiler]] 也能抓训练步 timeline。

### 8.3 nccl-tests 验证通信健康

[[nccl-tests]]（NCCL 集合通信基准工具）跑 all_reduce/all_gather 基准，看带宽/延迟是否正常。故障节点常表现为 nccl-tests 带宽异常（如比同集群其他节点低 5x）。是"节点健康体检"工具——节点修好后跑 nccl-tests 验证才归队。

### 8.4 RL 框架的故障恢复

[[RL worker故障恢复]] 里 RL 的 rollout worker（跑环境推理）故障率高于 trainer（模型推理轻、环境多样易崩）。检测/恢复设计借鉴本文——rollout worker 心跳、崩了重起 actor、从 rollout ckpt 恢复轨迹。GRPO/PPO RL 框架多内建类似机制。

### 8.5 抢占式实例的优雅退出

云 spot 实例被回收前通常给**提前告警**（如 AWS Spot Instance Interruption Notice 提前 2min、GCP preemptible 提前 30s）。训练系统要订阅这个告警，在回收前**主动存 ckpt + 优雅退出**，把被动故障变成主动 checkpoint + elastic 缩容，RPO 几乎为 0。这是 spot + elastic + 检测的高级配合。
