# 多模态数据加工

> **所属章节**: [[数据加工]]
> **所属模块**: [[18-集群网络存储与数据系统]]
> **别名**: 多模态数据加工 / multimodal data processing / 图文数据加工 / 视频数据加工 / 数据血缘
> **难度**: 中-高（需懂 [[WebDataset]] shard 格式 + [[Parquet]]/[[Arrow]] 列存 + [[Spark执行模型]] / [[Ray Data]] 分布式加工 + MinHash 去重）


## 1. 一句话定义

**多模态数据加工** 是图文 / 视频 / 音频等多模态预训练数据的加工流程——① **解码**（图片/视频/音频解码成可处理格式、处理损坏/截断文件）② **过滤**（质量过滤：分辨率/长宽比/水印/NSFW/OCR 文本质量；启发式 + 模型分类）③ **去重**（**MinHash/LSH 近似去重**、图文对去重；精确去重代价高用近似）④ **质量评估**（**CLIP score** 评估图文对齐、人工抽检、模型打分）⑤ **数据血缘**（记录每个样本来源、加工步骤、过滤规则、可追溯可复现）。**多模态加工难**是因为数据异构（图/文/视频/音频格式各异、解码开销大）、去重跨模态（图文对去重比单文本去重复杂）、质量难自动评（CLIP score 不完美、需人工抽检）。与单模态文本加工区别：文本只需清洗去重，多模态要**解码 + 跨模态对齐**。是 LLM/VLM 预训练数据管道的核心环节，与 [[WebDataset]]（shard 存储）、[[Ray Data]]（流式加工）、[[Parquet]]（结构化存储）结合，常与 [[流批一体数据加工]] 配合做全量批 + 增量流。

> [!note] 三句话定位
> - **是什么**：多模态预训练数据加工五步——解码 → 过滤 → 去重 → 质量评估 → 数据血缘。
> - **为什么难**：数据异构（格式各异、解码贵）、去重跨模态、质量难自动评。
> - **与文本加工区别**：文本只清洗去重；多模态要解码 + 跨模态对齐（CLIP score 等）。

## 2. 为什么需要它（动机与背景）

### 2.1 多模态预训练的崛起

LLM 之外，**VLM（Vision-Language Model，如 CLIP、BLIP、LLaVA、GPT-4V）**需要海量**图文对**训练——OpenAI CLIP 用 4 亿对、LAION-5B 有 50 亿对、常见数据集 PB 级。**视频/音频**（如 Video-LLaMA、AudioLM）也要海量多模态数据。

这些数据来自**网络爬取**（Common Crawl 图片 + alt text、YouTube 视频 + 字幕），**原始质量极差**：

- 损坏/截断文件（图片下了一半、视频 codec 错）。
- 低质量图（低分辨率、长宽比极端、水印、广告 banner）。
- NSFW（色情/暴力）内容。
- 图文不对齐（alt text 与图无关、机器生成的 SEO 垃圾文本）。
- 大量重复（同一图/文被多站点转载）。

**直接喂训练**会让模型学噪声、浪费算力、甚至触发合规风险。**必须加工**。

### 2.2 单模态文本加工不够用

单模态文本加工（如 LLM 纯文本预训练）流程相对简单：

- 清洗（去 HTML、去乱码、语言识别）。
- 去重（MinHash/LSH 近似去重，文本 shingling）。
- 质量过滤（perplexity、启发式规则）。

但**多模态加工多了两步关键**：

- **解码**：图片/视频/音频要解码成 tensor 才能处理——解码开销大（CPU/GPU 算力、内存）、损坏文件处理。
- **跨模态对齐评估**：图与文是否对齐？单看图或单看文都不够，需 CLIP score 等跨模态模型评估。

这两步使多模态加工比文本加工复杂得多。

### 2.3 加工流水线五步

业界（LAION、DataComp、OpenCLIP）总结出五步流水线：

```
爬取的图文对 (raw)
  |
  v
① 解码 (decode) ---- 处理损坏/截断、转 tensor
  |
  v
② 过滤 (filter) ---- 分辨率/长宽比/水印/NSFW/OCR 文本质量
  |
  v
③ 去重 (dedup) ---- MinHash/LSH 近似去重 (图 hash + 文 hash)
  |
  v
④ 质量评估 (quality) - CLIP score / 人工抽检 / 模型打分
  |
  v
⑤ 数据血缘 (lineage) - 记录来源/步骤/规则, 可追溯复现
  |
  v
训练 shard (WebDataset)
```

## 3. 核心概念详解

### 3.1 解码（decode）

把图片/视频/音频**从压缩格式解码成 tensor**：

- **图片**：JPEG/PNG/WebP → RGB tensor (H×W×3)。库：PIL、OpenCV、`torchvision.io`。
- **视频**：MP4/MKV → frame tensor (T×H×W×3) + 音频流。库：ffmpeg、`decord`、`pyav`。
- **音频**：MP3/WAV → waveform (samples) / spectrogram。库：`librosa`、`torchaudio`。

**处理损坏/截断文件**：网络爬取的文件常下到一半、HTTP 中断、codec 错。解码失败要**捕获异常、跳过或标记坏样本**，不能让一坏样本挂整个 pipeline。

```python
from PIL import Image, UnidentifiedImageError
import io

def safe_decode(byte_data):
    try:
        img = Image.open(io.BytesIO(byte_data))
        img.load()   # 强制解码（PIL 懒, load 才真解）
        return img.convert("RGB")
    except (UnidentifiedImageError, OSError, ValueError) as e:
        return None   # 坏样本, 跳过
```

> [!warning] 误区：PIL `Image.open` 立刻解码
> PIL 是**懒解码**——`open` 只读 header、`load` 或 `convert` 才真解。漏 `load()` 会把坏样本漏过到下游（训练时才爆）。务必 `load()` 或 `convert`。

### 3.2 过滤（filter）

解码后按**质量规则**过滤：

| 维度 | 规则 | 实现 |
|---|---|---|
| **分辨率** | 短边 < 200px 丢（太低训不出细节） | `img.size` |
| **长宽比** | 极端长宽比丢（如 100×10000 banner） | `w / h` 阈值 |
| **水印** | 有水印丢（视觉污染） | 水印检测模型 / 频域 |
| **NSFW** | 色情/暴力丢（合规） | NSFW 分类模型（如 `NudeNet`） |
| **OCR 文本质量** | 图内文字或 alt text 太短/乱码丢 | OCR + 启发式 |
| **重复帧（视频）** | 连续相同帧丢 | frame diff |
| **黑屏/纯色屏** | 视频黑屏段丢 | pixel variance |

**启发式 + 模型分类**结合：启发式快但粗（分辨率、长宽比）、模型分类准但贵（NSFW、水印）。生产 pipeline 两层都跑。

```python
def quality_filter(img, alt_text):
    w, h = img.size
    if min(w, h) < 200: return False                    # 分辨率
    if not (0.5 <= w/h <= 2.0): return False            # 长宽比
    if len(alt_text) < 5 or len(alt_text) > 200: return False  # 文本长度
    if is_nsfw(img): return False                       # NSFW 模型
    if has_watermark(img): return False                 # 水印模型
    return True
```

### 3.3 去重（dedup）：MinHash / LSH

多模态去重比文本复杂——既要**图去重**（同一图不同分辨率/压缩）、又要**文去重**（同一 caption 多次出现）、还要**图文对去重**（同一对整体重复）。

**精确去重**（hash 整样本）代价低但**抓不住**轻微变化（图压缩、resize 后 hash 变）。**近似去重**用 **MinHash / LSH（Locality Sensitive Hashing）**——对样本做 shingling、取最小 hash 签名、相似样本签名碰撞。

#### MinHash 原理

对集合 $S$（如文本的 n-gram 集合、图的感知 hash 特征），用 $k$ 个独立 hash 函数 $h_1, \dots, h_k$：

$$\text{minhash}_i(S) = \min_{x \in S} h_i(x), \quad i = 1, \dots, k$$

签名 $\text{sig}(S) = (\text{minhash}_1, \dots, \text{minhash}_k)$。

**Jaccard 相似度估计**：

$$\hat{J}(S_1, S_2) = \frac{|\{i : \text{minhash}_i(S_1) = \text{minhash}_i(S_2)\}|}{k} \approx J(S_1, S_2) = \frac{|S_1 \cap S_2|}{|S_1 \cup S_2|}$$

**理论保证**：$\mathbb{P}[\text{minhash}_i(S_1) = \text{minhash}_i(S_2)] = J(S_1, S_2)$。所以 $k$ 个 minhash 中相同比例无偏估计 Jaccard。

#### LSH（Locality Sensitive Hashing）

MinHash 签名是 $k$ 维向量，两两比 $O(n^2)$。**LSH** 把签名分 $b$ 段、每段 $r$ 个（$k = b \cdot r$）——只要**任一段完全相等**即视为候选重复对（candidate pair），再精比。降低比较复杂度到 $O(n)$。

```
sig = [m1, m2, ..., mk]  (k 维)
分 b 段, 每段 r:  [m1..mr] | [m_{r+1}..m_{2r}] | ... | [m_{k-r+1}..mk]

两样本 sig, sig':
  任一段 [m_{i*r+1}..m_{(i+1)*r}] 完全相等 -> 候选对
  候选对再精比 Jaccard 阈值
```

**参数权衡**：$r$ 小 → 召回高（抓得多）、误报多；$r$ 大 → 召回低、精确。$b$ 大 → 召回高、慢。

#### 图去重与图文对去重

- **图去重**：图的**感知 hash（pHash）**——DCT 变换后取低频系数的 hash。resize/压缩不变，近似去重。或 CLIP embedding + LSH。
- **图文对去重**：图 pHash + 文 MinHash 联合签名，两者都近似才算重复。

```python
import hashlib

def text_minhash(text, k=128, ngram=5):
    tokens = text.lower().split()
    shingles = {" ".join(tokens[i:i+ngram]) for i in range(len(tokens)-ngram+1)}
    sig = []
    for i in range(k):
        h = hashlib.blake2b(digest_size=8, salt=str(i).encode())
        min_h = min(int(h.hexdigest(), 16) for s in shingles)
        sig.append(min_h)
    return sig   # 128 维签名

def image_phash(img):
    # 感知 hash (DCT 低频)
    import imagehash
    return str(imagehash.phash(img))

def is_dup(sig_a, sig_b, threshold=0.8):
    j = sum(1 for x, y in zip(sig_a, sig_b) if x == y) / len(sig_a)
    return j >= threshold
```

### 3.4 质量评估（quality evaluation）

去重后剩的样本还要评估**图文对齐质量**——这是多模态独有（单文本无此步）。

| 方法 | 语义 | 代价 |
|---|---|---|
| **CLIP score** | 图文 cosine 相似度（CLIP image emb · text emb） | 一次 CLIP forward |
| **人工抽检** | 人工标 图文是否对齐 | 贵、慢、抽样 |
| **模型打分** | 训一个质量分类器（如 LAION aesthetic predictor） | 训练一次、推理快 |
| **perplexity** | 文本的语言模型困惑度 | LM forward |

**CLIP score** 是主流——CLIP 预训练后图与文对齐在 embedding 空间，cosine 高=对齐好。DataComp 等用 CLIP score 过滤低质图文对。

```python
import torch
import open_clip

model, _, preprocess = open_clip.create_model_and_transforms("ViT-L-14", pretrained="openai")
tokenizer = open_clip.get_tokenizer("ViT-L-14")

def clip_score(image, text):
    img_t = preprocess(image).unsqueeze(0)
    text_t = tokenizer([text])
    with torch.no_grad():
        img_emb = model.encode_image(img_t)
        text_emb = model.encode_text(text_t)
        img_emb /= img_emb.norm(dim=-1, keepdim=True)
        text_emb /= text_emb.norm(dim=-1, keepdim=True)
        return (img_emb @ text_emb.T).item()   # cosine 相似度
```

> [!note] 补充：CLIP score 不完美
> CLIP score 高不代表"图与文真实对齐"——CLIP 学的是 web 数据的弱对齐，可能把"图里有 cat 字"和图里真有猫都打高分。需人工抽检补。DataComp 实验显示 CLIP score过滤 + 下游任务评估能涨点，但非银弹。

### 3.5 数据血缘（data lineage）

**数据血缘**记录每个样本：

- 来源（URL、crawl batch、原始文件）。
- 加工步骤（解码、过滤规则、去重匹配、质量分）。
- 规则版本（filter/dedup 规则的代码 hash）。
- 时间戳。

价值：**可追溯**（出问题反查）、**可复现**（同规则再跑一致）、**合规审计**（版权/隐私投诉可定位删样本）。

### 3.6 多模态加工难在哪

| 难点 | 原因 | 对策 |
|---|---|---|
| **数据异构** | 图/文/视频/音频格式各异、metadata 不一 | 统一 schema（[[Arrow]]） |
| **解码开销大** | 图片/视频解码 CPU 重、视频尤甚 | 分布式 [[Ray Data]] actor 池、GPU 解码 |
| **去重跨模态** | 图 pHash + 文 MinHash 联合、LSH 参数 | 联合签名、调 LSH 参数 |
| **质量难自动评** | CLIP score 不完美、需人工抽检 | 自动 + 抽检 + 下游评估 |
| **版权/合规** | 来源广、NSFW/隐私风险 | 血缘可追溯 + 主动过滤 |

## 4. 数学原理 / 公式

### 4.1 Jaccard 相似度

两集合 $S_1, S_2$ 的 Jaccard：

$$J(S_1, S_2) = \frac{|S_1 \cap S_2|}{|S_1 \cup S_2|} \in [0, 1]$$

$J=1$ 完全相同、$J=0$ 完全不同。文本去重用 n-gram 集合算 $J$。

### 4.2 MinHash 无偏估计

**定理**：$\mathbb{P}[\text{minhash}(S_1) = \text{minhash}(S_2)] = J(S_1, S_2)$。

**证明**：设两集合的并 $U = S_1 \cup S_2$、交 $I = S_1 \cap S_2$。在 $U$ 上做均匀随机 hash，最小元等概率落 $U$ 任一元。最小元落在 $I$ 当且仅当 $\text{minhash}(S_1) = \text{minhash}(S_2)$（即最小元同时是两集合最小）。

$$\mathbb{P}[\text{min} \in I] = \frac{|I|}{|U|} = J(S_1, S_2) \quad \blacksquare$$

所以 $k$ 个独立 minhash 中相同比例 $\hat{J}$ 是 $J$ 的无偏估计、方差 $\text{Var}(\hat{J}) = J(1-J)/k$（伯努利）。

### 4.3 LSH 召回率

签名分 $b$ 段、每段 $r$ 个。两样本段 $i$ 完全相等概率 $J^r$（每 minhash 独立、都相等概率 $J^r$）。任一段相等（候选对）概率：

$$P_{\text{cand}}(J) = 1 - (1 - J^r)^b$$

要 $P_{\text{cand}} \ge 1-\delta$（召回率），解参数。$r$ ↑ 召回↓、$b$ ↑ 召回↑。典型 $b=20, r=5$，$J=0.5$ 时 $P_{\text{cand}} \approx 1 - (1-0.03)^{20} \approx 0.46$。

### 4.4 感知 hash 不变性

图 pHash 用 DCT（离散余弦变换）低频系数。低频对 resize/压缩/轻微变化鲁棒——

$$\text{pHash} = \text{threshold}(\text{DCT}(\text{resize}(img, 32\times32))_{\text{top-left } 8\times8})$$

resize 把图归一 32×32、DCT 取低频、阈值化成二进制 hash。相似图 hash 汉明距离小。

### 4.5 CLIP score

$$\text{CLIP\_score}(I, T) = \cos(E_I(I), E_T(T)) = \frac{E_I(I) \cdot E_T(T)}{\|E_I(I)\| \|E_T(T)\|} \in [-1, 1]$$

$E_I, E_T$ 是 CLIP 的 image/text encoder。高 cosine = 图文对齐好。一般阈值 > 0.2-0.3 算合格（DataComp 实验）。

## 5. 代码示例（可选）

### 5.1 完整 MinHash 去重（datasketch 库）

```python
from datasketch import MinHash, MinHashLSH

def build_minhash(text, num_perm=128):
    m = MinHash(num_perm=num_perm)
    tokens = text.lower().split()
    for i in range(len(tokens) - 4):
        m.update(" ".join(tokens[i:i+5]).encode())   # 5-gram shingle
    return m

# LSH 索引（分 b 段、每段 r）
lsh = MinHashLSH(threshold=0.8, num_perm=128)

samples = [
    {"id": "s1", "text": "a cat sitting on the mat"},
    {"id": "s2", "text": "a cat sitting on the mat"},   # 重复
    {"id": "s3", "text": "a dog running in the park"},
]

for s in samples:
    mh = build_minhash(s["text"])
    dup = lsh.query(mh)             # 查近似重复
    if dup:
        print(f"{s['id']} dup of {dup}, skip")
        continue
    lsh.insert(s["id"], mh)         # 新样本入索引
    print(f"{s['id']} kept")
```

### 5.2 Ray Data 流式多模态加工（分布式）

```python
import ray
from PIL import Image
import open_clip

ray.init()

model, _, preprocess = open_clip.create_model_and_transforms("ViT-L-14", pretrained="openai")
tokenizer = open_clip.get_tokenizer("ViT-L-14")

class MultimodalProcessor:
    """Ray Data actor: 解码+过滤+CLIP score"""
    def __init__(self):
        self.model = model.cuda()
        self.model.eval()
    def __call__(self, batch):
        results = []
        for img_bytes, text in zip(batch["image"], batch["text"]):
            # ① 解码
            img = safe_decode(img_bytes)
            if img is None: continue
            # ② 过滤
            if not quality_filter(img, text): continue
            # ④ 质量评估 (CLIP score)
            img_t = preprocess(img).unsqueeze(0).cuda()
            tok = tokenizer([text]).cuda()
            import torch
            with torch.no_grad():
                ie = self.model.encode_image(img_t).norm(dim=-1)
                te = self.model.encode_text(tok).norm(dim=-1)
                score = (ie * te).sum(-1).item()
            if score < 0.2: continue
            results.append({"image": img_bytes, "text": text, "score": score})
        return results

ds = ray.data.read_webdataset("s3://bucket/raw-*.tar")  # [[WebDataset]] shard
pipeline = ds.map_batches(MultimodalProcessor,
                          strategy=ray.data.ActorPoolStrategy(16),  # 16 actor 复用 GPU 模型
                          batch_size=64)
pipeline.write_webdataset("s3://bucket/clean-*.tar")   # 写回 shard
```

### 5.4 Spark 批加工（离线全量）

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import udf, col
from pyspark.sql.types import BooleanType, FloatType

spark = SparkSession.builder.appName("multimodal-batch").getOrCreate()

df = spark.read.parquet("s3://bucket/raw/")   # 全量历史 (batch)

@udf(BooleanType())
def filter_udf(img_bytes, text):
    img = safe_decode(img_bytes)
    if img is None: return False
    return quality_filter(img, text)

@udf(FloatType())
def clip_udf(img_bytes, text):
    return clip_score(safe_decode(img_bytes), text)

clean = df.filter(filter_udf(col("image"), col("text"))) \
          .withColumn("score", clip_udf(col("image"), col("text"))) \
          .filter(col("score") > 0.2)

clean.write.mode("overwrite").parquet("s3://bucket/clean/")
```

> [!tip] 实践：Spark 批 vs Ray Data 流
> 全量历史一次加工（如重训前的数据清洗）用 **Spark 批**（shuffle 优化深、SQL 生态、Crank 跑得快）。流式增量加工（新爬数据持续来）用 **[[Ray Data]] 流**（低延迟、actor 池 GPU 解码）。两者结合即 [[流批一体数据加工]]。

## 6. 与其他知识点的关系

- **上游（依赖）**: [[WebDataset]]（shard 存储与流式读）、[[Parquet]]（结构化存储、列存）、[[Arrow]]（跨进程零拷贝传 tensor）、[[Spark执行模型]]（批加工）、[[Ray Data]]（流式加工）、[[流批一体数据加工]]（批流统一）。
- **下游（应用）**: VLM 预训练（CLIP/BLIP/LLaVA）、Video-LLM 训练、多模态 RLHF、数据合规审计。
- **对比 / 易混**:
  - **多模态 vs 单模态文本加工**：前者要解码 + 跨模态对齐（CLIP score）；后者只清洗去重。
  - **MinHash vs 精确 hash**：前者近似（抓轻微变化、容忍 resize）、后者精确（只抓完全相同）。
  - **图 pHash vs CLIP embedding**：前者传统 DCT 低频、快但粗；后者深度语义、准但贵。
  - **CLIP score vs 人工评估**：前者自动、快但弱对齐；后者准但贵。生产两者结合。

## 7. 常见误区与易错点

> [!warning] 误区 1：CLIP score 高 = 图文真对齐
> 不一定。CLIP 学的是 web 弱对齐，可能把"图里有 cat 文字"和图里真有猫都打高分。CLIP score 是启发式过滤，非金标准。需人工抽检 + 下游任务评估补。

> [!warning] 误区 2：去重只用图 hash
> 不够。图去重抓不住同一 caption 配不同图、文去重抓不住同一图配不同文。**图文对联合去重**（图 pHash + 文 MinHash）才完整。

> [!warning] 误区 3：MinHash 精确等于 Jaccard
> MinHash 是**估计**（$\hat{J} \approx J$），有方差。$k$ 小时误差大。要 $k \ge 128$ 才稳定。LSH 召回也不完美（漏相似对）。

> [!warning] 误区 4：解码用 PIL `open` 即解码
> PIL 懒。`Image.open` 只读 header，要 `load()` / `convert()` 才真解码。漏 `load()` 会让坏样本漏到下游训练爆。

> [!warning] 误区 5：数据血缘是可选
> 不是。版权/隐私投诉来时**必须能定位删样本**、加工规则变更要可复现。血缘是合规与可复现的基础，不能省。

> [!warning] 误区 6：过滤越严越好
> 过严会**丢有用样本**、数据量不足。DataComp 实验显示**适度过滤**（如 CLIP score 阈值 0.2-0.3）最优，过严反伤下游性能。需下游任务评估调阈值。

> [!tip] 实践：加工 pipeline 调优
> (1) 解码慢 → Ray Data actor 池 + GPU 解码（如 `nvidia/dali`）。(2) 去重慢 → MinHash 并行（[[Spark执行模型]] shuffle 分布式）。(3) 质量分贵 → CLIP score actor 池批量算、cache 复用。

## 8. 延伸细节

### 8.1 LAION-5B / DataComp 实践

- **LAION-5B**：50 亿图文对、PB 级。用 CLIP score + 安全过滤 + 去重，开源。
- **DataComp**：系统实验过滤策略（CLIP score 阈值、去重强度）对下游性能影响，发现**适度过滤 + 强去重**最优。

### 8.2 视频加工特殊点

视频比图复杂——**抽帧策略**（均匀抽 / 场景抽）、**帧去重**（连续相同帧）、**时序对齐**（caption 与哪段帧对应）、**codec 多样**（H.264/H.265/VP9）。常用 `decord`/`pyav` 高性能抽帧、ffmpeg 转码。

### 8.3 GPU 加速解码

CPU 解码图片/视频慢。**NVIDIA DALI**（Data Loading Library）GPU 解码 + 增广 pipeline，与训练 step 重叠、消除 I/O 阻塞。Ray Data 可集成 DALI actor 池。

### 8.4 与训练数据格式

加工后写 shard（[[WebDataset]] tar）或 [[Parquet]]（图存 bytes / URL、文存 text、metadata 存 schema）。训练时 [[Ray Data]] 流式读喂 trainer。血缘写单独 Parquet 索引。

### 8.5 数据合规与版权

- **版权**：爬取的图/文可能受版权，血缘记录 URL 便于删除投诉。
- **隐私**：人脸、车牌、个人信息需脱敏（人脸检测 + 模糊）。
- **NSFW**：色情/暴力内容需模型过滤，避免违规训练数据。

详见 LAION 的 "LAION-5B → NSFW filtering" 与 DataComp 的"filtering ablations"。
