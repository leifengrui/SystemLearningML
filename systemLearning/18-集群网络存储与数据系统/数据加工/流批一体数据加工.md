# 流批一体数据加工

> **所属章节**: [[数据加工]]
> **所属模块**: [[18-集群网络存储与数据系统]]
> **别名**: 流批一体 / stream-batch unification / 流批统一 / unifying stream and batch / 流批一体架构
> **难度**: 中（需懂 [[Spark执行模型]] 批模型 + 流处理基础 + LLM 训练/RL 数据流）


## 1. 一句话定义

**流批一体数据加工** 是**同一套 API / 同一引擎**既跑**批处理**（全量历史数据、有限数据集、一次性算完）又跑**流处理**（增量实时数据、无界数据流、持续算）的架构范式——**Apache Flink 是代表**（DataStream API 统一、批是流的有限子集、bounded stream = batch、unbounded stream = streaming），Spark Structured Streaming 是微批（micro-batch）近似。核心动机是 **AI 数据管道要同时满足离线训练全量批 + 在线服务/RL 增量实时流**，流批统一降低开发成本（一套代码、一套引擎、一套语义）；关键技术是 **checkpoint/savepoint 实现 exactly-once**（精确一次）、**水位线 watermark 处理乱序**、**增量与全量**（全量重建 vs 增量更新、CDC change data capture）。与 [[Spark执行模型]] 的批为主对比，Spark Structured Streaming 是微批（用批模型套流、延迟高）、Flink 是真流（事件驱动、低延迟）；在 LLM 数据管道的应用：预训练数据离线批（全量清洗 tokenize 一次跑完）、RLHF 采样增量流（实时产 trajectory 增量喂 trainer）。

> [!note] 三句话定位
> - **是什么**：同一套 API/引擎既跑批（全量历史）又跑流（增量实时），Flink 代表（批是流的有限子集）。
> - **为什么 AI 要**：训练要全量历史批、在线服务/RL 要增量实时流，统一降低开发成本。
> - **与 Spark 对比**：Spark 批为主、Structured Streaming 是微批近似；Flink 是真流（事件驱动、低延迟）。

## 2. 为什么需要它（动机与背景）

### 2.1 传统：批流两套

传统大数据栈是**两套系统**：

- **批处理（batch）**：Spark/Hadoop MR，跑全量历史数据，T+1（次日）出结果。延迟高但准确、算力充分。
- **流处理（stream）**：Storm/早期 Flink，实时增量数据，秒级延迟。低延迟但窗口/乱序难处理。

问题：**同业务两套代码、两套引擎、两套语义**——批算完的数据与流算的不一致（不同聚合语义、不同 window 定义），开发维护成本高。比如"今日 UV"批算一次、流算一次、数字对不上。

### 2.2 流批一体（unification）

Flink 提出**批是流的有限子集**：所有数据都是"事件流"，批就是"有限事件流"（bounded stream）、流就是"无限事件流"（unbounded stream）。**一套 DataStream API**、一套算子（map/filter/keyBy/window）、一套语义（event time、watermark、exactly-once）覆盖批和流。

- 写一份代码，参数切批/流：bounded source = 批、unbounded source = 流。
- 同语义：批的 window 与流的 window 一致（事件时间、watermark）。
- 同引擎：Flink 引擎同时跑批和流，批优化（block partition、sort）自动应用。

### 2.3 AI 数据管道要流批统一

LLM/AI 数据管道天然要流批一体：

- **预训练**：全量历史文本（几 TB），离线批清洗、去重、tokenize 一次跑完（[[Spark执行模型]] 批，或 Flink bounded）。
- **RLHF/在线 RL**：实时采样产 trajectory、增量喂 trainer（流，Flink unbounded）。
- **数据更新**：新数据持续来（爬新网页、新对话），流式增量加工、与历史全量合并。
- **在线特征**：推理时实时特征（用户行为流），需流式算。

**一套引擎覆盖训练批 + 在线流**，降低开发成本、保证数据语义一致。

### 2.4 增量与全量

- **全量重建**：定期把全部历史数据重跑一遍（如每周重训 LLM），结果准但慢、资源大。
- **增量更新**：只处理新来的增量数据（CDC、流），merge 到存量。快但需保证一致性（exactly-once、乱序处理）。

**流批一体**让两者**统一**：批 = 一次性全量、流 = 持续增量，都是"事件流"。

## 3. 核心概念详解

### 3.1 bounded vs unbounded stream

| 类型 | 数据 | 处理模式 | 典型 |
|---|---|---|---|
| **bounded stream（有限流）** | 已结束、有边界 | 批（一次性算完） | 历史日志、Parquet 文件 |
| **unbounded stream（无限流）** | 持续来、无边界 | 流（持续算） | Kafka、CDC、实时日志 |

Flink 统一：source 给 bounded → 批模式、source 给 unbounded → 流模式。**同一份算子代码**两种都能跑。

### 3.2 event time vs processing time

- **processing time**：数据**到处理引擎**的时间戳。受网络/调度延迟影响，乱序严重。
- **event time**：数据**产生**的时间戳（业务时间，如用户点击发生时间）。稳定、按业务语义。

流处理优先 **event time**——窗口按 event time 分、watermark 处理乱序（晚到的数据仍归正确窗口）。批因数据全在、按 event time 排序就行。

### 3.3 watermark（水位线）

**watermark** 是流处理的核心机制——声明"**event time ≤ T 的数据都已到**"，允许引擎推进窗口（T 之前的窗口可输出）。

- watermark 越激进 → 延迟低但**可能丢晚到数据**。
- watermark 越保守 → 完整但**延迟高**（窗口迟迟不输出）。

`allowedLateness` 容忍晚到数据再更新窗口。详见 Flink 文档"Watermarks"。

### 3.4 window（窗口）

流是无限的，聚合要切片——**window**：

| 窗口 | 语义 |
|---|---|
| **tumbling**（滚动） | 不重叠固定窗（每 1min 一个） |
| **sliding**（滑动） | 重叠固定窗（每 30s 起一个 1min 窗） |
| **session**（会话） | 按活跃间隔动态分（用户连续行为一段 session） |

批因数据全在、window 退化为按 event time 分组聚合。

### 3.5 checkpoint / savepoint（exactly-once）

**exactly-once**（精确一次语义）是流批一体的关键——故障后恢复、**每条数据恰好处理一次**（不丢不重）。

- **checkpoint**：周期性把算子状态（窗口、offset）快照到可靠存储（HDFS/S3）。故障后从 checkpoint 恢复、状态一致。
- **savepoint**：手动的 checkpoint（升级/迁移用）。
- **Chandy-Lamport 算法**：Flink 用异步 barrier 分布式快照、不停作业、低开销。

```
source offset 1000, 窗口状态 {window1: sum=42}
  | checkpoint barrier 触发
  v
快照到 S3: {offset=1000, state={window1: 42}}
  | 故障
  v
从快照恢复: source 从 offset 1000 重读, state 从 42 起 -> 不丢不重 (exactly-once)
```

### 3.6 CDC（Change Data Capture）

**CDC** 把数据库 binlog/redo log 作为流——表的**增量变更**（insert/update/delete）作为事件流。Flink 读 CDC 流实时同步表到下游、做增量 ETL。

```sql
-- Flink SQL 读 MySQL CDC
CREATE TABLE orders (
  id INT, amount DECIMAL, ts TIMESTAMP(3),
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH (
  'connector' = 'mysql-cdc',
  'hostname' = '...', 'database-name' = '...', 'table-name' = '...'
);

-- 流式聚合 (实时输出每分钟订单额)
SELECT window_start, window_end, SUM(amount)
FROM TABLE(TUMBLE(TABLE orders, DESCRIPTOR(ts), INTERVAL '1' MINUTE))
GROUP BY window_start, window_end;
```

### 3.7 与 Spark 对比

| 维度 | Spark | Flink |
|---|---|---|
| 模型 | 批为主、Structured Streaming 微批 | 真流（事件驱动） |
| 延迟 | 秒-分钟（微批） | 毫秒-秒（事件） |
| 批流统一 | 微批近似（流 = 连续小批） | 原生统一（批是有限流） |
| 状态管理 | 弱（批无状态） | 强（RocksDB 状态后端、增量 checkpoint） |
| exactly-once | Structured Streaming 支持 | 原生支持（Chandy-Lamport） |
| window | Structured Streaming 支持 | event time + watermark 完整 |
| 适合 | 离线 ETL、批 SQL、批 ML | 实时 ETL、实时特征、在线聚合 |

**Structured Streaming** 是 Spark 的流批统一尝试——把流当"连续小批"（micro-batch），每小批跑一次批引擎。延迟较高（秒级），但写法与批一致（同 DataFrame API）。**Flink 是真流**（事件驱动、毫秒级）。

### 3.8 在 LLM 数据管道的应用

| 场景 | 批/流 | 引擎 |
|---|---|---|
| 预训练数据离线清洗、tokenize | 批（全量历史） | Spark / Flink bounded |
| 持续爬新数据增量加工 | 流（CDC / 文件流） | Flink unbounded |
| RLHF 采样产 trajectory 增量喂 trainer | 流（rollout 实时产） | Flink / [[Ray Data]] |
| 在线推理实时特征 | 流（用户行为） | Flink |
| 全量重训（每周全量） | 批 | Spark / Flink bounded |
| 增量微调（持续新数据 fine-tune） | 流 | Flink + [[Ray Data]] 喂 trainer |

**流批一体的价值**：训练批 + 在线流用一套引擎、一套语义、一套代码——降低维护成本、保证数据一致。

## 4. 数学原理 / 公式

### 4.1 exactly-once 的状态机

设 source offset $o$、算子状态 $S$、输出 $Y$。每条数据 $i$ 处理：

$$S_{t+1} = f(S_t, i_t), \quad Y_t = g(S_t, i_t), \quad o_{t+1} = o_t + 1$$

checkpoint 在时刻 $T_c$ 快照 $(o_{T_c}, S_{T_c})$。故障恢复从快照重放 $o_{T_c}+1$ 起：

$$S'_{T_c} = S_{T_c}, \quad o' = o_{T_c} + 1, \quad \text{replay from here}$$

若每条恰好重放一次（不丢不重），则**输出恰好等于无故障情况**——exactly-once 保证。

### 4.2 watermark 与乱序

设 watermark $W(t)$ 在 processing time $t$ 声明"event time $\le W(t)$ 的数据已到"。窗口 $[T_s, T_e)$ 在 $W(t) \ge T_e$ 时输出。

- $W(t)$ 越激进（接近 $t$）→ 窗口早输出、低延迟，但晚到数据（event time < $W(t)$）丢弃。
- $W(t)$ 越保守（远小 $t$）→ 窗口晚输出、高延迟，但完整。

典型 $W(t) = \max_{\text{seen}} \text{event\_time} - \Delta$（$\Delta$ 容忍乱序）。

### 4.3 微批 vs 真流的延迟

Spark Structured Streaming 微批：每批处理间隔 $\Delta$、延迟 $\ge \Delta$（每 $\Delta$ 跑一次）。

Flink 真流：每事件到达即处理、延迟 $\approx$ 网络延迟（毫秒级）。

| 模型 | 延迟 | 吞吐 |
|---|---|---|
| 微批（Spark SS） | $\ge \Delta$（秒级） | 高（批优化） |
| 真流（Flink） | 毫秒级 | 中（每事件开销） |

### 4.4 增量 vs 全量代价

设全量数据 $D$、增量 $\Delta D$、重算代价 $C$。

- **全量重建**：每周期重算 $C(D)$，资源 $O(D)$。
- **增量更新**：每周期只算 $C(\Delta D)$，资源 $O(\Delta D)$，但需维护状态 $S$、merge 代价 $O(S)$。

当 $\Delta D \ll D$ 时增量快 $|\Delta D|/|D|$ 倍，但状态管理复杂、需 exactly-once 保证。

## 5. 代码示例（可选）

### 5.1 Flink SQL 流批一体

```sql
-- 同一份 SQL, source 不同即批/流

-- 批 (bounded source: Parquet 文件)
CREATE TABLE batch_source (
  id INT, ts TIMESTAMP(3), amount DECIMAL
) WITH ('connector' = 'parquet', 'path' = 'hdfs:///history/');

-- 流 (unbounded source: Kafka)
CREATE TABLE stream_source (
  id INT, ts TIMESTAMP(3), amount DECIMAL,
  WATERMARK FOR ts AS ts - INTERVAL '5' SECOND
) WITH ('connector' = 'kafka', 'topic' = 'orders', ...);

-- 同一份聚合 SQL, 改 source 即批/流
SELECT
  window_start, window_end, SUM(amount) AS total
FROM TABLE(TUMBLE(TABLE stream_source, DESCRIPTOR(ts), INTERVAL '1' MINUTE))
GROUP BY window_start, window_end;
```

### 5.2 Flink DataStream API

```java
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();

// bounded source -> 批模式
DataStream<Event> batch = env.readTextFile("hdfs:///history/*.log");

// unbounded source -> 流模式
DataStream<Event> stream = env.addSource(new FlinkKafkaConsumer<>("orders", ...));

// 同一份算子 (event time + window + watermark)
DataStream<Result> result = stream
    .assignTimestampsAndWatermarks(
        WatermarkStrategy.<Event>forBoundedOutOfOrderness(Duration.ofSeconds(5))
            .withTimestampAssigner((e, t) -> e.ts))
    .keyBy(e -> e.userId)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new MyAgg());

env.execute("my job");
```

### 5.3 Spark Structured Streaming（微批）

```python
from pyspark.sql import SparkSession
from pyspark.sql.functions import window, sum

spark = SparkSession.builder.appName("ss-demo").getOrCreate()

# 流 source (Kafka)
df = spark.readStream.format("kafka").option("kafka.bootstrap.servers", "...").load()

# 同 DataFrame API (与批一致)
agg = df.groupBy(window(df.ts, "1 minute")).agg(sum("amount").alias("total"))

# 启动流查询 (微批, 每 trigger 跑一次)
query = agg.writeStream.format("console").trigger(processingTime="10 seconds").start()
query.awaitTermination()
```

## 6. 与其他知识点的关系

- **上游（依赖）**: [[Spark执行模型]]（批模型对照）、[[Spark shuffle与partition]]（shuffle 是批的瓶颈）、[[Parquet]]/[[Arrow]]（批存储格式）、Kafka/CDC（流 source）。
- **下游（应用）**: LLM 预训练数据批清洗、RLHF 增量 trajectory 流、在线推理实时特征、[[多模态数据加工]]（流式加工多模态）、[[trajectory generation]]（RL 经验流）。
- **对比 / 易混**:
  - **Flink vs Spark Structured Streaming**：前者真流（事件驱动、毫秒）、原生流批统一；后者微批（秒级）、批模型套流近似。
  - **流批一体 vs Lambda 架构**：前者一套引擎统一；后者**两套**（批层 + 速度层，需 merge）。
  - **exactly-once vs at-least-once**：前者精确一次（状态一致、输出幂等）；后者至少一次（可能重复，需下游幂等）。
  - **bounded vs unbounded**：前者批、后者流。Flink 统一为"事件流"。

## 7. 常见误区与易错点

> [!warning] 误区 1：Spark Structured Streaming = Flink 真流
> 不等价。Spark SS 是**微批**（每 trigger 跑一次批），延迟秒级。Flink 是真流（事件驱动），延迟毫秒级。低延迟实时选 Flink。

> [!warning] 误区 2：流批一体 = 一份代码无脑复用
> 流和批语义有差异——流要处理 watermark/乱序/晚到、批因数据全在不要。一份代码但需参数调（窗口、watermark）适配流场景。

> [!warning] 误区 3：增量更新一定比全量快
> 当 $\Delta D \to D$（大部分数据变化）、状态 merge 代价大时，增量可能慢于全量。增量适合"小变更 + 大存量"。

> [!warning] 误区 4：exactly-once = 不重不丢
> exactly-once 指**应用层状态**精确一次。**输出到外部**（如 Kafka、DB）需幂等 sink 或两阶段提交，否则下游可能见重复。Sink 要配 `exactly-once` connector。

> [!warning] 误区 5：watermark 越激进越好
> 激进 watermark 窗口早输出、低延迟，但**丢晚到数据**（event time 落后的数据）。要按业务乱序程度调（容忍乱序 $\Delta$）。

> [!tip] 实践：选 Spark SS 还是 Flink
> 延迟秒级够、与批 DataFrame 复用 → Spark SS。延迟毫秒、复杂窗口/watermark、强状态 → Flink。AI 训练数据：批用 Spark、实时 RL 用 Flink + [[Ray Data]]。

## 8. 延伸细节

### 8.1 Lambda vs Kappa 架构

- **Lambda**：批层（Spark MR）+ 速度层（流）+ 服务层（merge）。两套代码、merge 复杂。
- **Kappa**：**只流**——批是流的重放（bounded source = 历史 + 流 source = 实时）。一套代码、一套引擎。Flink 推动 Kappa。

流批一体本质是 Kappa 范式。

### 8.2 Flink 状态后端

- **MemoryStateBackend**：状态存内存（小、快、不持久，测试用）。
- **FsStateBackend**：状态存内存 + checkpoint 写文件。
- **RocksDBStateBackend**：状态存 RocksDB（本地盘）、增量 checkpoint、支持超大状态（TB 级）。生产默认。

### 8.3 流表双算（Flink Table API）

Flink Table API/SQL 是流批一体的统一入口——同 SQL 既跑批（bounded source）又跑流（unbounded source + 动态表）。"动态表"概念：流是表的 changelog、表是流的物化视图。详见 Flink "Dynamic Tables"。

### 8.4 与 LLM 数据管道集成

- **预训练批**：Spark/Flink bounded 跑全量清洗 tokenize → Parquet → [[Ray Data]] 流式读喂 trainer。
- **RLHF 流**：rollout worker 实时产 trajectory → Flink 流式 collect → [[Ray Data]] 喂 RL trainer。
- **持续 fine-tune**：新数据流式来 → Flink 增量处理 → 持续 fine-tune（如 DPO 持续新偏好对）。

详见 [[trajectory generation]]、[[RL训推一体框架]]。
