# 并行文件系统

> **所属章节**: [[存储]]
> **所属模块**: [[18-集群网络存储与数据系统]]
> **别名**: 并行文件系统 / Parallel FS / Lustre / GPFS / IBM Storage Scale / WekaIO / BeeGFS / 共享文件系统 / 分布式文件系统
> **难度**: 中高（需懂 [[本地NVMe与对象存储]] + 多节点并发读 + POSIX + [[拓扑感知]]）


## 1. 一句话定义

**并行文件系统**是专为 **HPC/AI 大规模并发 IO** 设计的**共享文件系统**——把数据分散到多个存储节点（OST/NSD），客户端**并发直连多个存储节点**读写，聚合带宽随存储节点数线性扩展（单集群可达 **10–100+ GB/s 聚合读**），并**挂载为 POSIX 目录树**让所有训练节点看到同一命名空间。三大代表：**Lustre**（开源、客户端多、OST 分片、大文件顺序读强、HPC 王者）、**GPFS / IBM Storage Scale**（商用、POSIX 强一致、元数据强、企业级稳定）、**Weka**（对象后端 + 本地 NVMe 缓存层、AI 工作负载友好、低延迟）。它解决了 **NFS 单机元数据/单链路瓶颈**——AI 训练多 worker 并发读 shard 时，NFS 的单 metadata server 与单 TCP 流撑不住百万小文件与高并发，必须并行 FS 把元数据与数据流分散。与 [[本地NVMe与对象存储]] 互补：并行 FS 是"多节点共享 + 并发"的温热层，本地 NVMe 是节点内最快热层，对象存储是底冷层。典型部署：并行 FS 共享挂载到所有训练节点的同一路径（如 `/mnt/lustre`），worker 直接从共享路径读 shard。

> [!note] 三句话定位
> - **是什么**：多存储节点 + 多客户端并发的共享 POSIX 文件系统，聚合带宽随节点数扩展。
> - **为什么需要**：AI 训练多 worker 并发读 shard / 小文件，NFS 单点元数据与单链路成瓶颈，必须并行分散。
> - **与 [[本地NVMe与对象存储]] 关系**：本地 NVMe 单节点快但不共享，对象存储共享但慢，并行 FS 是"共享 + 够快"的折中热层。


## 2. 为什么需要它（动机与背景）

### 2.1 NFS 在 AI 训练下崩溃

NFS 是单 server + 单链路（或少量 HA）：元数据（open/stat/readdir）全走单 metadata server，数据流单 TCP。AI 训练 batch 大、worker 多（8 卡 ×8 worker = 64 并发读），每个 worker 打开 shard / 读小图都是一次元数据 RPC。**NFS 元数据锁争用 + 单链路带宽**让吞吐塌到几十–几百 MB/s，GPU 空转。这是并行 FS 存在的根本理由。

### 2.2 多 worker 并发读 shard

数据并行 + DataLoader 多 worker 时，训练节点同时读不同 shard / 同一 shard 不同 offset。并行 FS 让每个 worker 的读请求**分散到不同 OST/NSD（数据分片到多存储节点）**，多存储节点并发供数据，聚合带宽线性扩展。

### 2.3 小文件多的元数据压力

预训练语料若没打包成 [[WebDataset]] tar，百万小文件的 `open` 元数据请求打爆单 MDS。并行 FS 用**分布式元数据服务**（GPFS 多 NSD server、Lustre DNE 多 MDS）把元数据也分散，扛住高并发元数据。

### 2.4 共享命名空间

训练框架配置里写一个路径（`/mnt/lustre/data/train-00000.tar`），所有节点都能访问，不用每节点复制数据。这比"每节点本地 NVMe 各存一份"省存储省同步。是**多节点共享热数据**的首选。

### 2.5 与对象存储的差距

对象存储虽共享，但无 POSIX、LIST 慢、延迟高。训练热路径要 `mmap`、随机 seek、readdir 列目录，对象存储做不到。并行 FS 给完整 POSIX，应用代码无感（像本地盘用）。

> [!warning] 误区：并行 FS 一定比对象存储快
> 大对象顺序读时并行 FS 聚合带宽高；但**百万小文件随机读**时并行 FS 的元数据/寻道开销也不小，可能不如**先打包成大 shard 顺序读**。并行 FS 不解决"小文件多"的根本问题，仍需 shard 化。


## 3. 核心概念详解

### 3.1 Lustre

- **架构**：`MGS`（管理配置）+ `MDS/MDC`（元数据服务，管文件名/目录/布局）+ `OST`（Object Storage Target，数据分片存储块，每个 OST 是一个块设备）。客户端 `llite` 挂载，文件按 **`stripe`** 分散到多 OST。
- **分片**：一个大文件切成 `stripe_size`（默认 1 MB，可调到几十 MB–GB）的块，轮询分布到多 OST。客户端并发直连多 OST 读，聚合带宽 = OST 数 × 单 OST 带宽。
- **强项**：**大文件顺序读**（HPC checkpoint、AI shard tar 读）吞吐极高；客户端规模大（几千–几万 client）。
- **弱项**：元数据单 MDS（可 DNE 多 MDS 扩）；小文件元数据压力；POSIX 一致性偏弱（分布式锁 DLM）。
- **开源**：社区版 + 商业支持（DDN EXA、OpenSFS）。HPC 中心标配。

### 3.2 GPFS / IBM Storage Scale

- **架构**：`NSD`（Network Shared Disk，存储节点暴露的块设备）+ 多 NSD server + 分布式 token manager。所有节点可同时挂载，**POSIX 强一致**（token 协议保证字节级一致）。
- **强项**：**强一致 POSIX**、元数据强、企业级稳定（自愈、快照、副本）、文件集（fileset）/ 策略分层。
- **商用**：付费，IBM Power/Linux。金融/政企/大型企业多用。
- **AI 场景**：IBM Watson 当年用 GPFS 训练；现 Storage Scale 推 AI 友好配置（SSD tier + 策略）。

### 3.3 WekaIO (WekaFS)

- **架构**：**对象后端**（S3/On-prem 对象存储做持久层）+ **本地 NVMe 缓存层**（每个计算节点装本地 NVMe 做热缓存，热数据在本地 NVMe 命中，未命中的回对象存储）。软件定义，无专用存储硬件。
- **强项**：**AI 工作负载友好**——热 shard 命中本地 NVMe 给 GB/s + μs–低 ms 延迟，冷回对象存储；POSIX 共享；横向扩展到数百 GB/s。
- **弱项**：本地 NVMe 缓存命中率敏感；对象后端回源慢。

### 4 其他

- **BeeGFS**：开源，轻量，中小集群常用，元数据 + 存储分离，易部署。
- **CephFS**：基于 RADOS 的 POSIX FS，元数据池 + 数据池，云原生友好但延迟偏高。

### 3.5 机械与 SSD 混合分层

并行 FS 普遍支持**分层策略**（GPFS policy、Lustre HSM、Weka tiering）：

```
热数据 (当前 epoch shard, 刚写 ckpt)  → SSD tier (GB/s, 贵)
冷数据 (历史 shard, 旧 ckpt)         → 机械盘 tier (便宜, 容量大)
                                      或 对象存储后端
```

策略按访问频率/年龄自动迁移。机械盘容量大便宜但带宽低（~200 MB/s/盘），SSD tier 带宽高但贵。热放 SSD，冷放机械/对象存储，兼顾容量与带宽。

### 3.6 为什么 AI 训练要并行 FS

| 痛点 | 并行 FS 怎么解 |
|---|---|
| 多 worker 并发读 shard | 数据分片到多 OST/NSD，多存储节点并发供数据 |
| 小文件元数据瓶颈 | 分布式元数据服务，扛高并发 open/stat |
| 多节点要同一份数据 | POSIX 共享挂载，所有节点同路径 |
| 要 POSIX（mmap/seek/readdir） | 完整 POSIX，应用无感 |

### 3.7 典型部署

```
[存储节点: 多 OST/NSD, SSD+机械分层]
        |  (InfiniBand / RoCE / 高速以太网)
[训练节点 0..N: 共享挂载 /mnt/lustre]
   worker 直接读 /mnt/lustre/datasets/train-*.tar
```

所有训练节点挂载同一路径，DataLoader 从共享路径读 shard。可与本地 NVMe 缓存配合：先把当前 epoch 的 shard 预取到本地 NVMe（命中本地更快），并行 FS 做未命中的回源与共享。

### 3.8 三类存储横向对比（复用并补全）

| 维度 | [[本地NVMe与对象存储\|本地 NVMe]] | [[本地NVMe与对象存储\|对象存储]] | 并行文件系统 |
|---|---|---|---|
| 共享 | ❌ 节点本地 | ✅ 远端 | ✅ POSIX 共享挂载 |
| 聚合带宽 | 单盘 3–14 GB/s | 并发到 GB/s+（分 prefix） | **10–100+ GB/s**（随 OST 线性扩） |
| 延迟 | 10–100 μs | 10–100 ms | ms 级 |
| POSIX | ✅ | ❌ | ✅ |
| 容量 | 4–60 TB/节点 | 无限 | PB 级 |
| 小文件 | 强 | 差 | 中（依赖 MDS 规模） |
| 成本 | 贵 | 便宜 | 中（自建+商用许可） |


## 4. 数学原理 / 公式

### 4.1 聚合带宽模型

设 $N_{\text{OST}}$ 个存储节点（OST/NSD），单节点带宽 $BW_1$，客户端足够多时聚合带宽：

$$BW_{\text{agg}} \approx \min\big(N_{\text{OST}}\cdot BW_1,\; BW_{\text{network}}\big)$$

随 OST 数线性增长，直到网络互联带宽成上限。例：16 OST × 6 GB/s = 96 GB/s，受 IB 网络限。这就是为什么并行 FS 能比单 NVMe 快一个数量级。

### 4.2 stripe 与聚合读

文件大小 $F$，stripe_size $S$，OST 数 $K$。文件分 $\lceil F/S \rceil$ 个 stripe 轮询分布。客户端并发读 $K$ 个 OST：

$$T_{\text{read}}(F) \approx \frac{F}{BW_{\text{agg}}} \approx \frac{F}{K\cdot BW_1}$$

比单 OST 读 $\frac{F}{BW_1}$ 快 $K$ 倍。要求 stripe_size 足够大使每请求不碎（否则元数据/小请求开销吃掉并发优势）。AI shard tar 几百 MB–1 GB，stripe_size 设几十 MB–几百 MB 最优。

### 4.3 元数据吞吐

小文件读瓶颈在元数据。设 MDS 能处理 $R_{\text{meta}}$ 次 open/s，训练有 $W$ 个 worker 各并发打开文件，每文件需 $n_{\text{meta}}$ 次 RPC（open+stat+readdir）：

$$\text{并发元数据需求} = W \cdot n_{\text{meta}}\cdot f_{\text{open}}$$

若超 $R_{\text{meta}}$ 则排队。单 MDS 的 $R_{\text{meta}}$ 量级几万/s（待核实，与硬件强相关），多 MDS（DNE/多 NSD）线性扩。这也是**把数据打包成大 shard 减少文件数**的根本原因。

### 4.4 与 GPU 喂饱的带宽匹配

单卡数据消耗 $\approx \frac{\text{batch}\times\text{seq}\times\text{bytes/sample}}{T_{\text{step}}}$。8 卡 ×8 worker，若每 step 100 ms 消耗 1 GB，需 80 GB/s 读。单本地 NVMe 7 GB/s 不够，并行 FS 80 GB/s 正好。这是大规模训练用并行 FS 而非纯本地盘的算账依据。

```python
      .decode()
      .to_tuple("txt", "cls"))
for text, label in ds:
    ...
```

## 6. 与其他知识点的关系

- **上游（依赖）**: [[InfiniBand]] / [[RoCE]]（并行 FS 数据走高速网络）、[[RDMA与GPUDirect]]（客户端可 RDMA 读）、[[拓扑感知]]（数据本地性）、[[本地NVMe与对象存储]]（分层互补）。
- **下游（应用）**: AI 训练数据加载（多 worker 共享读 shard）、[[WebDataset]]（shard 放并行 FS）、checkpoint 集中存储、[[Ray Data]] / [[Spark执行模型]]（shuffle 数据落并行 FS）。
- **对比 / 易混**:
  - **并行 FS vs [[本地NVMe与对象存储\|对象存储]]**：前者 POSIX 共享 + 强一致 + 高聚合带宽；后者便宜无限容量但无 POSIX + 慢。热共享用并行 FS，冷仓用对象存储。
  - **Lustre vs GPFS vs Weka**：Lustre 开源大文件顺序读强 HPC 王者；GPFS 商用强一致企业稳；Weka 对象后端 + 本地 NVMe 缓存 AI 友好。
  - **并行 FS vs 本地 NVMe**：并行 FS 共享但延迟 ms（网络往返），本地 NVMe 节点内 μs 但不共享。最优是并行 FS + 本地 NVMe 缓存层（如 Weka 架构）。


## 7. 常见误区与易错点

> [!warning] 误区 1：挂了并行 FS 就不用 shard 化
> 错。百万小文件并行 FS 的元数据/寻道开销也大。仍要把样本打包成大 shard（[[WebDataset]] tar / [[Parquet]]）顺序读，并行 FS 只是让多节点共享 + 多存储节点供数据。

> [!warning] 误区 2：Lustre 和 GPFS 差不多
> 差异明显：Lustre 元数据偏单 MDS（DNE 扩）、POSIX 一致性靠 DLM 较弱、开源；GPFS token 强一致、商用稳。AI 场景读多写少都行，强一致写场景（如 ckpt 多写）GPFS 更稳。

> [!warning] 误区 3：并行 FS 写 ckpt 多节点同时写同一文件
> POSIX 写同一文件需锁，多节点并发写同文件会争用/损坏。ckpt 应每 rank 写不同分片文件（[[distributed checkpoint]] / [[state_dict分片]]），不要并发写同一文件。

> [!warning] 误区 4：stripe_size 默认就好
> 默认 1 MB 对大 shard 读偏小，元数据/小请求开销大。AI shard tar 建议调到几十 MB–1 GB，让每 stripe 充满带宽。

> [!tip] 实践：热数据预取到本地 NVMe
> 即使用并行 FS，当前 epoch 的 shard 仍建议预取到训练节点本地 NVMe（命中本地 GB/s + μs 延迟），并行 FS 做未命中回源。这是 Weka 架构思路，也可手动用 DataLoader prefetch + 本地缓存实现。

> [!note] 补充：RDMA 直读
> 现代 Lustre/GPFS 客户端支持 RDMA 读（数据不经服务端 CPU 直接到客户端内存，甚至 GPUDirect 直接到 GPU 显存），延迟降、CPU 省。配 InfiniBand 时务必开。


## 8. 延伸细节

### 8.1 元数据服务扩展

Lustre DNE（Distributed Namespace）把目录分散到多 MDS，元数据并行；GPFS 多 NSD server 同理。元数据吞吐随 MDS 数扩展，是扛百万小文件的关键（但根本解仍是 shard 化）。

### 8.2 一致性模型

- Lustre：DLM 分布式锁，写互斥、读共享，最终一致较弱（适合 HPC checkpoint 读多写少）。
- GPFS：token 协议字节级强一致，适合多节点并发写同文件集（但同文件并发写仍要小心）。
- Weka：POSIX 强一致（写经本地 NVMe → 对象后端，元数据强）。

### 8.3 与对象存储的融合

GPFS Storage Scale 支持 S3 对象后端 tier；Weka 原生对象后端。趋势是"并行 FS 前端 POSIX + 对象存储后端持久 + 本地 NVMe 缓存"，三者融合成统一存储栈。

### 8.4 内容来源

Lustre/GPFS/Weka 架构特性来自各自官方文档（lustre.org、IBM Storage Scale 文档、WekaIO 文档）与 HPC 存储工程实践。聚合带宽数据量级（数十–百 GB/s）来自大型 HPC 中心实测报告，具体随集群规模。stripe_size、DNE、RDMA 直读等特性稳定可信；个别 MDS 元数据吞吐数字标"待核实"（强硬件相关）。
