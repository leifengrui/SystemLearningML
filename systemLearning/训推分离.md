# 训推分离

> **所属章节**: [[训推分离]]（本章总览 / 第六章 §20）
> **所属模块**: [[06-训推系统]]
> **难度**: 中（架构概念，需懂 [[learner worker split]] 与推理服务基础）

## 1. 一句话定义

**训推分离（PT-PD 分离 / Training-Inference Disaggregation）** 是把大模型生命周期的**训练侧（PT，[[policy training (PT)]]）**与**部署侧（PD，[[policy deployment (PD)]]）**在**进程 / 设备 / 框架**上物理解耦的架构：PT 用 [[learner worker split]]（learner 训练 + [[rollout worker]] 采样）产出权重 $\theta$，PD 用 [[inference worker]] 服务用户，两端经 [[weight sync mechanism]] 单向推送 $\theta$。与 [[learner worker split]] 的关键区别：split 是**训练内部**采与训的分离（都在 PT 侧），训推分离是**训练与部署之间**的分离（PT↔PD）。两个"分离"都涉及 weight sync，但层次不同——口诀 **"split 拆采与训，训推分离拆训与用"**。

> [!note] 别名 / 撞名警告
> - 训推分离 = PT-PD disaggregation = policy training/deployment separation = 训练-推理解耦。
> - **撞名**：[[PD分离]]（推理系统里 Prefill-Decode Disaggregation 预填充-解码分离）也简称"PD 分离"，但那是**推理内部**两阶段分离，**不是**本条。本条 PD = **Policy Deployment**（策略部署）。两者完全不同层次，见到"PD 分离"先问"是推理内部 prefill/decode，还是训练/部署？"——本条是后者。
> [!note] 解答：异步训练算不算一种训推分离？
> 用户问："异步其实也属于一种训推分离？"——**直觉很准，但答案要分两层**：抽象形式上"是"，但在本库的严格语义层次上"不是本条训推分离(PT-PD)"，而是它的**内层子结构**的一个运行模式。下面把三层"分离"摆清楚。
>
> ## 一、为什么直觉上像——抽象形式确实同构
> 把"训推分离"抽象成一句话模式：**"会更新 $\theta$ 的计算"与"只用 $\theta$ 做前向(inference)的计算"在进程/设备上分离，中间用 weight sync 衔接**。按这个抽象：
>
> | | 训练侧（更新 θ） | 推理侧（用 θ 前向） | 衔接 |
> |---|---|---|---|
> | **训推分离(PT-PD，本条)** | learner 训练 | inference worker 服务用户 | weight sync（版本级，低频） |
> | **异步训练** | learner 训练 | **rollout worker 生成 trajectory**（也是前向！） | weight sync（step 级，高频） |
>
> 两者的"推理侧"都在跑**前向计算**，且 RLHF 里 rollout worker 用的就是 vLLM 这类**推理引擎**——所以用户感觉"异步也是一种训推分离"在**形式层面完全成立**：都是"训/推解耦 + weight sync"的骨架。
>
> ## 二、为什么严格说不是本条——三个层次要分清
> 关键在"**推理服务谁**"和"**分离在哪一层**"：
>
> | 维度 | 训推分离 PT↔PD（本条） | learner↔rollout split | 异步训练 |
> |---|---|---|---|
> | **分离的两端** | 训练 vs **部署服务用户** | 训练 vs **采样采数据** | （是 split 的**运行模式**） |
> | **两端在哪** | PT 与 PD，**跨阶段** | **都在 PT 内部** | 都在 PT 内部 |
> | **推理侧服务谁** | **终端用户**（要回答好） | **learner**（要采到训练数据） | learner |
> | **θ 更新频率** | 版本级（部署期冻结） | step 级（频繁变） | step 级（频繁变） |
> | **weight sync 频率** | 低频（版本升级） | 高频（每 N step） | 高频 + 容忍 stale |
> | **产物** | 用户回答 | trajectory 训练数据 | trajectory |
> | **有无 stale data** | PD 落后 PT（服务落后，不训崩） | worker 用旧 θ 采→ratio 失真（会训崩） | stale 是核心代价 |
> | **目的** | 训用解耦、各自最优 | 采训解耦、提吞吐 | 采训并行流水线、提吞吐 |
>
> 口诀升级：
> - **训推分离 = 拆"训"与"用(服务用户)"** —— 跨阶段，θ 在用侧冻结；
> - **split = 拆"训"与"采(为训采数据)"** —— 训练内部，θ 在采侧频繁变；
> - **异步 = split 的并行流水线运行模式**（vs 同步互相等），不是新的"分离"层次。
>
> 三者**嵌套**：训推分离(PT-PD) ⊃ split(learner↔rollout)，split 又有同步/异步两种运行模式。异步是 split 的**时序模式**，而 split 是训推分离 PT 内部的**子架构**。所以异步在"训推分离"的**大树里**，但不是树根那个"训推分离"本身。
>
> ## 三、但有真实的技术重叠——这才是直觉的来源
> 用户的直觉之所以对，是因为 RLHF 的 rollout worker **真的在跑推理引擎**：
> - verl/OpenRLHF 的 rollout worker 用 **vLLM/SGLang** 生成（和 PD 侧同一个推理栈）；
> - 所以"训推分离里 PD 侧用到的推理优化（KV cache、continuous batching、量化）"**在 rollout 侧原样复用**；
> - 极端形态：AReaL / verl fully_async 把 rollout worker 与 learner **物理分到不同 GPU 池**（ disaggregated generation），这时 rollout 池长得**几乎和 PD 池一模一样**——区别只剩"产出喂 learner 还是喂用户"。
>
> 即：**架构形式同构 + 推理技术栈复用**，是用户直觉"异步也是一种训推分离"的真正根源。这是**技术复用与形式同构**，不等于**架构语义等同**。
>
> ## 四、一句话定论
> **抽象地看，异步训练与训推分离同属"训/推解耦 + weight sync"骨架，形式同构、推理栈复用**；但**在本库的严格分类里，异步是 learner-worker split 的运行模式，split 是训推分离(PT-PD)内部子架构**——异步在训推分离的大树里、却不是那个"训推分离"本身。差别锚点一句话：**"推理侧服务谁"——服务用户=训推分离，服务 learner=split（异步是其模式）**。
>
> 详见 [[asynchronous training]]（异步是 split 的运行模式 + stale 代价）、[[learner worker split]]（采训分离）、本条 §3.2（训推分离 vs split 对比表）。三者关系链：**训推分离 ⊃ split ∋ {同步, 异步}**。
## 2. 为什么需要它（动机与背景）

训练（PT）与部署（PD）的负载特性几乎完全相反，硬塞同进程同集群必然互相拖累：

1. **负载性质相反**：PT 重**算力 + 显存**（forward+backward+optimizer state，FSDP/DeepSpeed/Megatron 优化方向），PD 重**延迟 + 并发**（只前向、KV cache、continuous batching、量化，vLLM/TensorRT-LLM 优化方向）；
2. **框架异构**：训练用 FSDP/DeepSpeed（[[Fully Sharded Data Parallel|FSDP]]/[[Megatron-LM]]），部署用 vLLM/Triton，API 与优化点不同，同进程难兼容；
3. **$\theta$ 更新节奏不同**：PT 每 N step 更新一次 $\theta$（频繁），PD 部署期间 $\theta$ 冻结（只在版本升级时换）；
4. **资源弹性不同**：PT 是固定大集群长时间跑训练，PD 需随用户流量弹性扩缩（Kubernetes autoscaling）；
5. **故障域隔离**：PT 训练崩了不该影响线上服务，PD 服务挂了不该打断训练；
6. **版本管理**：PT 持续训出新版本，PD 需灰度 / 蓝绿 / canary 升级，分离才能做版本管理。

故训推分离是大模型从"训完即用"到"持续训练 + 稳定服务"的工程必然——把训与用解耦到专责集群，各自最优配置，经 weight sync 单向传递 $\theta$。

## 3. 核心概念详解

### 3.1 PT 与 PD 的角色分工

| 角色                                 | 职责                         | 框架/优化点                                  | 典型实现                         |
| ---------------------------------- | -------------------------- | --------------------------------------- | ---------------------------- |
| **PT（[[policy training (PT)]]）**   | 预训练/SFT/RM/PPO，产出 $\theta$ | FSDP/DeepSpeed/Megatron + vLLM（rollout） | learner + [[rollout worker]] |
| **PD（[[policy deployment (PD)]]）** | 加载 $\theta$ 服务用户请求         | vLLM/TensorRT-LLM/Triton                | [[inference worker]]         |
| **[[weight sync mechanism]]**      | PT→PD 单向推 $\theta$         | AllGather / 参数服务器 / RPC + 压缩            | weight broadcast             |
| **版本管理**                           | PD 灰度/蓝绿/canary 升级         | 路由 + 多副本                                | blue-green deployment        |

```
PT(训练)  ──weight sync(新 θ)──>  PD(部署)  ──response──>  用户
  ↑                                  │
  │  learner + rollout worker        │  inference worker(多副本+负载均衡)
  └──── 持续训练(版本 t, t+1, ...) ───┘  θ 冻结(版本升级才换)
```

### 3.2 训推分离 vs learner/worker split（最易混点）

| 维度 | [[learner worker split]] | 训推分离（本条） |
|---|---|---|
| **分离的两端** | rollout worker（采样）↔ learner（训练） | PT（训练）↔ PD（部署） |
| **两端在哪** | **都在 PT 内部** | PT 与 PD，**跨阶段** |
| **产物** | 更新后的 $\theta$ | 用户回答 |
| **数据流** | worker 采 trajectory → buffer → learner | 用户请求 → PD → 用户 |
| **weight sync 方向** | learner → rollout worker（训练内频繁） | PT → PD（版本级，低频） |
| **$\theta$ 更新频率** | 每 N step（频繁） | 版本升级（低频） |
| **代表框架** | verl/OpenRLHF/IMPALA | vLLM server + Triton + 训练集群 |

**层次叠加**：训推分离的 PT 内部又用 learner/worker split——即"训推分离 ⊃ split"。split 是 PT 内部的子架构，训推分离是 PT↔PD 的外层架构。两者不是替代，是嵌套。

### 3.3 weight sync 在训推分离的形态

PT 训出新 $\theta_{\text{new}}$ → checkpoint 保存 → PD 加载（或流式 weight sync）→ 热切换：

| 方式 | 机制 | 适用 |
|---|---|---|
| **checkpoint 加载** | 存 safetensors → PD `load_state_dict` | 版本升级，低频 |
| **流式 weight sync** | PT 端 broadcast $\theta$ → PD 端注入 | 频繁同步（如 RLHF 在线服务） |
| **增量同步** | 只传变化的 $\theta$（差分/压缩） | $\theta$ 大、带宽紧 |
| **热切换** | blue-green/canary，零停机 | 生产 PD 必备 |

详见 [[weight sync mechanism]]。

### 3.4 训推分离的版本管理

- **蓝绿部署**：新版本 PD 副本起来 → 流量切过去 → 旧副本下线，零停机；
- **金丝雀（canary）**：新版本先接小比例流量，观察指标稳了再全量；
- **回滚**：新版本异常 → 流量切回旧版本；
- **A/B 测试**：多版本并行，按用户分流比效果。
这是 PD 侧的 SRE 工程，PT 持续训、PD 滚动升。

## 4. 数学原理 / 公式

### 4.1 stale policy 在训推分离的投影

PD 部署 $\theta_{\text{deploy}}$，PT 已训到 $\theta_{\text{new}}$，**版本差 $k$**：
$$k = \text{step}(\theta_{\text{new}}) - \text{step}(\theta_{\text{deploy}})$$

$k$ 大 → 用户接触旧版本（服务落后）。PD 侧的 stale 不像 PT 侧会训崩，但会**服务落后 + KL 漂移**（$\pi_{\theta_{\text{deploy}}}$ 与 $\pi_{\theta_{\text{new}}}$ 行为差大）。防法：定期 weight sync + KL 监控。详见 [[stale policy problem]]。

### 4.2 部署侧的延迟-吞吐 tradeoff

PD 单副本吞吐 $\text{throughput}=\frac{B_{\text{dynamic}}\cdot T}{T_{\text{infer}}}$（tokens/sec），延迟 $\text{latency}\approx\text{TTFT}+T\cdot\text{TPOT}$。多副本并发 $N$ → 总吞吐 $N\times$，但单副本延迟不变。扩副本提吞吐不提延迟，是 PD 扩展主手段。

## 5. 代码示例（可选）

```python
# 伪代码:训推分离的最小骨架(PT 端训 + weight sync + PD 端服务)
import torch, threading, queue

theta_store = {"theta": None}          # 共享权重(实际用 Ray object store / NFS)
buffer = queue.Queue(maxsize=10)

# ===== PT: learner + rollout worker =====
def pt_train_loop():
    while True:
        # rollout worker 采数据(用当前 θ)
        traj = rollout_worker.generate(theta_store["theta"])
        buffer.put(traj)
        # learner 训练
        batch = buffer.get()
        theta_store["theta"] = learner.step(theta_store["theta"], batch)
        # 定期 weight sync 给 PD
        if step % SYNC_EVERY == 0:
            weight_sync_to_pd(theta_store["theta"])   # PT→PD 推 θ

# ===== PD: inference worker 服务 =====
def pd_serve_loop():
    pd_theta = load_checkpoint()        # 初始加载
    while True:
        req = recv_user_request()       # 用户请求
        resp = vllm_generate(pd_theta, req)   # θ 冻结,服务用户
        send_to_user(resp)
        # 版本升级时换 θ(低频)
        if new_version_available():
            pd_theta = weight_sync_from_pt()   # 热切换

# 启动 PT 与 PD(分离的进程/集群)
threading.Thread(target=pt_train_loop, daemon=True).start()
threading.Thread(target=pd_serve_loop, daemon=True).start()
# 实际:PT=FSDP 集群,PD=vLLM server 集群,weight sync=NCCL/RPC
```

> [!tip] 实际工程
> 生产训推分离：PT 用 verl/OpenRLHF（FSDP + vLLM rollout），PD 用 vLLM server / Triton + TensorRT-LLM，weight sync 用 Ray actor 参数注入或 checkpoint 加载，版本管理用 Kubernetes + Argo Rollouts。RLHF 在线服务（边训边用）需频繁 weight sync，离线服务（训完再部署）用 checkpoint 加载。

## 6. 与其他知识点的关系

- **上游（依赖）**: [[learner worker split]]（PT 内部架构）、[[policy training (PT)]]、[[policy deployment (PD)]]、训练好的 LLM。
- **下游（应用）**: [[weight sync mechanism]]、[[stale policy problem]]、[[inference worker]]、[[rollout worker]]、版本管理（蓝绿/canary）、RLHF 在线服务。
- **对比 / 易混**:
  - **训推分离 vs [[learner worker split]]**：见 §3.2。训与用分离 vs 采与训分离，层次不同（训推分离 ⊃ split）。
  - **训推分离 vs [[PD分离]]**：本条是训练-部署分离（PD=Policy Deployment），[[PD分离]]是推理内部 prefill-decode 分离（PD=Prefill-Decode）。**撞名不同物**。
  - **训推分离 weight sync vs split weight sync**：前者 PT→PD（版本级、低频、可 checkpoint），后者 learner→rollout worker（step 级、高频、流式）。机制类似、节奏不同。

## 7. 常见误区与易错点

> [!warning] 误区清单
> 1. **"训推分离 = learner/worker split"**：错。split 是 PT 内部采训分离，训推分离是 PT↔PD 训用分离。层次不同，训推分离 ⊃ split。
> 2. **"训推分离 = [[PD分离]]"**：错（撞名）。本条 PD=Policy Deployment，[[PD分离]] PD=Prefill-Decode。一个训用分离，一个推理内部分离。
> 3. **"PD 的 $\theta$ 也频繁更新"**：错。PD 部署期间 $\theta$ 冻结，只在版本升级时换；频繁更新是 PT 内 rollout worker 的事。
> 4. **"训推分离必须在线"**：不必。离线训完存 checkpoint → PD 加载，也是训推分离（低频 sync）；在线 RLHF 服务才是高频 sync。
> 5. **"PT 和 PD 用同框架"**：低效。PT 用 FSDP/DeepSpeed，PD 用 vLLM/Triton，异构是分离的优势。
> 6. **"weight sync 越频繁越好"**：错。PD 频繁换 $\theta$ → KV cache 失效 + 服务抖动 + 版本管理复杂。需 tradeoff。
> 7. **"训推分离无代价"**：有。引入 [[stale policy problem]]（PD 落后 PT）+ weight sync 带宽 + 版本管理复杂度。
> 8. **"PD 和 PT 同集群"**：可但不优。同集群省 weight sync 延迟（colocate），但故障域不隔离、资源争抢；生产常分集群。

## 8. 延伸细节

### 8.1 RLHF 在线服务（边训边用）

RLHF 训练中 PT 持续更新 $\theta$，若需同时让用户体验新策略（如 ChatGPT 训练期灰度），PD 需高频 weight sync。这是训推分离最难的场景：$\theta$ 频繁变 + KV cache 失效 + stale policy。verl/OpenRLHF 支持，用 Ray actor 流式注入。

### 8.2 colocate vs 分离部署

- **colocate**：PT 与 PD 同节点，weight sync 走 NVLink/同机内存，快几个量级（[[placement group]] PACK 策略）；
- **分离部署**：PT 与 PD 不同集群，故障域隔离、资源独立弹性，但 weight sync 走网络慢。
tradeoff：在线高频 sync 选 colocate，离线版本升级选分离。

### 8.3 训推分离的现代实现

- **verl/OpenRLHF**：RLHF 训练框架，PT 内 split + 可 weight sync 给 PD；
- **vLLM server / Triton**：PD 侧服务，支持热加载 $\theta$；
- **Ray**：PT 与 PD 都是 Ray actor，weight sync 用 `actor.update.remote(new_theta_ref)`；
- **Kubernetes + Argo Rollouts**：PD 版本管理（蓝绿/canary）。

### 8.4 训推分离与持续学习

理想形态：PT 持续从用户数据/反馈学习 → 不断训新 $\theta$ → PD 滚动升级 → 用户始终用最新。这是"自演进 Agent"（如 AReaL 2.0）的架构基础，但需解决 stale policy + 版本一致 + 数据隐私。
